01

What are Route Guards?

Definition

Route Guards are interfaces / functions that tell Angular's Router whether to allow or deny navigation to a route. They run before the component is loaded.


  • Can return boolean, UrlTree, or Observable<boolean>
  • true → allow navigation
  • false → block navigation
  • UrlTree → redirect to another route

5 Types of Guards

GuardPurpose
canActivateBlock access to a route
canActivateChildBlock access to child routes
canDeactivatePrevent leaving a route (unsaved changes)
canMatchBlock route from matching at all (Angular 14+)
canLoadPrevent lazy module from loading (deprecated)

Guards run in order. If any returns false or UrlTree, navigation stops immediately.

✅ Angular 14+ — Functional Guards

New recommended approach — plain functions instead of classes. No need to inject services into a class; just use inject() inside the function.

⚠️ Deprecated in Angular 15+

CanActivate, CanLoad, CanDeactivate class interfaces are deprecated. Use functional guards with canActivateFn, canMatchFn instead.

02

canActivate — Protect a Route

Old Way — Class-based (Angular < 14)

Implement CanActivate interface. Still works but deprecated.

New Way — Functional Guard (Angular 14+) ✅

A plain function using inject(). Cleaner, tree-shakeable, testable.

// ── NEW WAY (Angular 14+) — Recommended ─────────────────────────────
import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { AuthService } from './auth.service';

export const authGuard: CanActivateFn = (route, state) => {
  const auth   = inject(AuthService);
  const router = inject(Router);

  if (auth.isLoggedIn()) {
    return true;                          // ✅ allow navigation
  }
  return router.createUrlTree(['/login']); // 🔀 redirect to login
};

// ── Usage in Routes ──────────────────────────────────────────────────
const routes: Routes = [
  {
    path: 'dashboard',
    component: DashboardComponent,
    canActivate: [authGuard]  // attach guard
  },
  { path: 'login', component: LoginComponent }
];

// ── OLD WAY (Class-based — deprecated) ──────────────────────────────
@Injectable({ providedIn: 'root' })
export class AuthGuard implements CanActivate {
  constructor(private auth: AuthService, private router: Router) {}

  canActivate(): boolean | UrlTree {
    return this.auth.isLoggedIn() || this.router.createUrlTree(['/login']);
  }
}
💡 Return types canActivate can return

boolean · Observable<boolean> · Promise<boolean> · UrlTree · Observable<UrlTree>

03

canActivateChild — Protect Child Routes

When to use?

When you have a parent route with multiple child routes and want to protect all children without repeating the guard on each child route. The guard runs each time a child is navigated to.

vs canActivate on parent

canActivate on parent runs only once when the parent is activated. canActivateChild runs every time a child route is activated — even if parent is already loaded.

// auth-child.guard.ts
import { CanActivateChildFn, Router } from '@angular/router';
import { inject } from '@angular/core';

export const authChildGuard: CanActivateChildFn = (childRoute, state) => {
  const auth = inject(AuthService);
  if (auth.isLoggedIn()) return true;
  return inject(Router).createUrlTree(['/login']);
};

// app.routes.ts — protect ALL children in one place
const routes: Routes = [
  {
    path: 'admin',
    component: AdminLayoutComponent,
    canActivateChild: [authChildGuard],  // guard runs for every child
    children: [
      { path: 'users',    component: UsersComponent },   // protected ✅
      { path: 'settings', component: SettingsComponent }, // protected ✅
      { path: 'reports',  component: ReportsComponent },  // protected ✅
    ]
  }
];

Key Interview Point

canActivate guards the route entry once. canActivateChild guards each child navigation separately — useful when the parent shell stays loaded but each sub-page needs fresh permission checks (e.g., roles changing mid-session).

04

canDeactivate — Prevent Leaving (Unsaved Changes)

When to use?

Prevent users from accidentally navigating away from a form with unsaved changes. Common in edit pages, long forms, or checkout flows.

// can-deactivate.guard.ts — Generic, reusable guard
import { CanDeactivateFn } from '@angular/router';

// Define interface that components must implement
export interface CanComponentDeactivate {
  canDeactivate(): boolean | Observable<boolean>;
}

export const canDeactivateGuard: CanDeactivateFn<CanComponentDeactivate> =
  (component) => {
    return component.canDeactivate
      ? component.canDeactivate()
      : true;
  };

// edit-user.component.ts — implement the interface
export class EditUserComponent implements CanComponentDeactivate {
  isDirty = false;   // set to true when form changes

  canDeactivate(): boolean {
    if (this.isDirty) {
      return window.confirm('You have unsaved changes. Leave anyway?');
    }
    return true;
  }
}

// app.routes.ts
{ path: 'edit/:id', component: EditUserComponent, canDeactivate: [canDeactivateGuard] }
✅ Best Practice

Make the guard generic using an interface (CanComponentDeactivate). This way one guard works for ALL components — each component decides its own logic.

⚠️ Senior Tip

Instead of window.confirm() use a Dialog service (Angular Material Dialog) returning an Observable<boolean> for better UX and testability.

05

canMatch — Block Route Matching (Angular 14+)

What is canMatch?

Controls whether a route definition should even be considered for matching. If it returns false, the router skips this route and tries the next matching route.


This is different from canActivate which blocks navigation — canMatch skips the route entirely as if it doesn't exist.

canMatch vs canActivate vs canLoad

GuardBehaviour when false
canActivateBlocks navigation, shows error or redirects
canLoadPrevents lazy bundle download (deprecated)
canMatchSkips route — tries next route in config ✅
 
// Role-based routing with canMatch — show different component per role
export const adminMatchGuard: CanMatchFn = () => {
  const auth = inject(AuthService);
  return auth.hasRole('admin');   // true → match this route; false → try next
};

export const userMatchGuard: CanMatchFn = () => {
  const auth = inject(AuthService);
  return auth.hasRole('user');
};

// Same path '/dashboard' → loads DIFFERENT component based on role!
const routes: Routes = [
  {
    path: 'dashboard',
    canMatch: [adminMatchGuard],
    loadComponent: () => import('./admin-dashboard.component')
  },
  {
    path: 'dashboard',               // same path — fallback for regular users
    canMatch: [userMatchGuard],
    loadComponent: () => import('./user-dashboard.component')
  }
];
🎯 Key Interview Point — canMatch replaces canLoad

canLoad is deprecated in Angular 15+. Use canMatch instead — it also prevents lazy bundle loading AND allows fallback routing. It is more powerful and flexible.

06

Role-Based Access Control (RBAC) with Guards

Real-World Scenario — Admin / Manager / User roles

Pass the required role as route data and read it inside a single reusable guard. One guard handles all roles.

// role.guard.ts — single guard, handles any role
export const roleGuard: CanActivateFn = (route, state) => {
  const auth   = inject(AuthService);
  const router = inject(Router);

  // Read required roles from route data
  const requiredRoles: string[] = route.data['roles'] ?? [];
  const userRoles: string[]     = auth.getUserRoles();

  // Check if user has at least one required role
  const hasAccess = requiredRoles.some(role => userRoles.includes(role));

  if (hasAccess) return true;

  return router.createUrlTree(['/unauthorized']); // redirect
};

// app.routes.ts
const routes: Routes = [
  {
    path: 'admin',
    component: AdminComponent,
    canActivate: [roleGuard],
    data: { roles: ['admin'] }          // only admin
  },
  {
    path: 'reports',
    component: ReportsComponent,
    canActivate: [roleGuard],
    data: { roles: ['admin', 'manager'] } // admin OR manager
  }
];

// auth.service.ts
getUserRoles(): string[] {
  const token = localStorage.getItem('token');
  return this.decodeJwt(token)?.roles ?? [];  // decode JWT for roles
}

JWT Auth Guard + Async (Real-World Pattern)

Real scenario — Check JWT expiry + refresh token silently

Guards can return Observable<boolean> — perfect for async checks like verifying token validity with the server, refreshing expired tokens, or loading user profile before granting access.

// jwt-auth.guard.ts — async guard with token refresh
import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { catchError, map, of } from 'rxjs';

export const jwtAuthGuard: CanActivateFn = () => {
  const auth   = inject(AuthService);
  const router = inject(Router);

  // Token valid? Allow immediately
  if (auth.isTokenValid()) return true;

  // Token expired? Try to refresh silently
  return auth.refreshToken().pipe(
    map(newToken => {
      auth.setToken(newToken);
      return true;                          // ✅ refreshed successfully
    }),
    catchError(() => {
      auth.logout();
      return of(router.createUrlTree(['/login'])); // ❌ refresh failed → login
    })
  );
};

// auth.service.ts helpers
isTokenValid(): boolean {
  const token = localStorage.getItem('access_token');
  if (!token) return false;
  const { exp } = this.decodeJwt(token);
  return Date.now() < exp * 1000;             // check expiry
}
✅ Senior Pattern — HTTP Interceptor + Guard

In production, handle token refresh in an HTTP Interceptor (catches 401 errors globally) and keep the guard simple — just check isLoggedIn(). Don't duplicate refresh logic in every guard.

Multiple Guards + Lazy Loading

You can apply multiple guards to a single route — they run in order

All guards must return true for navigation to proceed. First false stops everything.

// Multiple guards on lazy-loaded routes
const routes: Routes = [
  {
    path: 'admin',
    canActivate:      [authGuard],         // 1st: is logged in?
    canActivateChild: [roleGuard],         // 2nd: has role for child?
    canMatch:         [featureFlagGuard],  // 3rd: feature enabled?
    loadChildren: () => import('./admin/admin.routes')
      .then(m => m.adminRoutes)
  },
  {
    path: 'profile',
    canActivate: [authGuard],
    canDeactivate: [canDeactivateGuard],  // unsaved changes check on leave
    loadComponent: () => import('./profile.component')
  }
];

// Standalone component routing (Angular 17+ with no NgModule)
export const adminRoutes: Routes = [
  { path: '',        component: AdminLayoutComponent },
  { path: 'users',   component: UsersComponent,   data: { roles: ['admin'] } },
  { path: 'reports', component: ReportsComponent, data: { roles: ['admin', 'manager'] } },
];

Guard Execution Order

  • canMatch runs first — if false, route is skipped entirely
  • canActivate — then checks access
  • canActivateChild — then checks child access
  • resolve — data fetching
  • Component renders last

Functional Guard Composition

Combine guards elegantly with mapToCanActivate() or just pass an array:

canActivate: [authGuard, roleGuard, featureGuard]

All must pass. Short-circuits on first failure.

Senior-Level Interview Q&A

Q1. What is the difference between canActivate and canMatch?
canActivate blocks navigation to a matched route — the route was found but access is denied. canMatch prevents the route from being matched at all — if it returns false, the router tries the next route in the config. Use canMatch for role-based routing where the same URL loads different components per role.
Q2. Why are class-based guards deprecated? What replaced them?
Angular 15+ deprecated class-based guards (CanActivate, CanLoad interfaces) because they required @Injectable() and DI boilerplate. Functional guards (plain functions using inject()) are the replacement — they are simpler, tree-shakeable, and easier to test. The function types are CanActivateFn, CanActivateChildFn, CanDeactivateFn, CanMatchFn.
Q3. How do you implement Role-Based Access Control with a single guard?
Define the required roles in route.data = { roles: ['admin'] }. In the guard, read route.data['roles'] and compare against the user's roles from the JWT (via AuthService). Return true if any role matches, or redirect to /unauthorized via router.createUrlTree(). One guard handles all roles.
Q4. Can a guard return an Observable? When would you use it?
Yes — canActivate can return Observable<boolean | UrlTree>. Use it for async checks: verifying a JWT with the server, silently refreshing an expired token, or loading user permissions from an API before allowing access.
Q5. What is the best way to implement canDeactivate for a form?
Create a generic interface CanComponentDeactivate with a canDeactivate() method. Make the guard call component.canDeactivate(). Each form component implements the interface with its own logic (dirty check). For production, return an Observable<boolean> from a Material Dialog instead of window.confirm().
Q6. What is the execution order when multiple guards are applied?
canMatch → canActivate → canActivateChild → resolve → component loads. When multiple guards are in an array, they run in order and short-circuit on the first false. canDeactivate runs when leaving.
