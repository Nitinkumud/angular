Specificity & the Cascade

What is the Cascade?

When multiple CSS rules target the same element, the browser resolves conflicts using three factors in order:


  • 1. Origin & Importance — !important > author > browser default
  • 2. Specificity — which selector is more specific
  • 3. Source Order — last rule wins when specificity is equal

/* Specificity examples — higher wins */

p                    { color: blue; }      /* (0,0,1)   ← loses */
.text               { color: red; }       /* (0,1,0)   ← wins over element */
#title              { color: green; }     /* (1,0,0)   ← wins over class */
p.text              { color: purple; }    /* (0,1,1)   ← element + class */
div p.text          { color: orange; }    /* (0,1,2)   ← more elements */
.nav .menu .item a  { color: pink; }      /* (0,3,1)   ← 3 classes + 1 element */

/* !important overrides everything (avoid using it!) */
p { color: blue !important; }  /* wins over #id — but bad practice */

/* Source order — when specificity is EQUAL, last one wins */
.text { color: red; }
.text { color: blue; }   /* ← wins — same specificity, comes last */

/* :is() / :where() — modern selectors */
:is(h1, h2, h3)       { color: red; }   /* takes specificity of MOST specific arg */
:where(h1, h2, h3)    { color: red; }   /* always specificity (0,0,0) — easy to override */
✅ Best Practice — Keep specificity low

Use classes only (avoid IDs in CSS, avoid !important). Low specificity = easier to override, maintain, and scale. Use :where() for reusable utility styles that should always be overridable.

Box Model & Positioning

Box Model

Every element is a rectangular box with 4 layers:

margin
border
padding
content (width × height)

box-sizing

/* content-box (default) */
/* width = content only */
/* total = width + padding + border */
.box {
  width: 200px;
  padding: 20px;
  border: 2px solid;
  /* TOTAL = 244px ← surprise! */
}

/* border-box (recommended) */
/* width INCLUDES padding + border */
*, *::before, *::after {
  box-sizing: border-box;
  /* total = exactly 200px ✅ */
}

Flexbox — Complete Guide

Container Properties (parent)

.container {
  display: flex;
  flex-direction: row;       /* row|column|row-reverse|column-reverse */
  flex-wrap: nowrap;         /* nowrap|wrap|wrap-reverse */
  justify-content: flex-start; /* main axis alignment */
  /*  flex-start | flex-end | center
        space-between | space-around | space-evenly */
  align-items: stretch;      /* cross axis — single line */
  /*  flex-start | flex-end | center | baseline */
  align-content: flex-start;  /* cross axis — multi-line (wrap) */
  gap: 16px;                   /* space between items ✅ modern */
}

Item Properties (children)

.item {
  flex-grow: 0;    /* grow to fill remaining space */
  flex-shrink: 1; /* shrink if needed */
  flex-basis: auto; /* initial size before grow/shrink */

  /* Shorthand: flex: grow shrink basis */
  flex: 1;          /* flex: 1 1 0% — grow equally ✅ */
  flex: none;      /* flex: 0 0 auto — fixed size */
  flex: 0 1 200px; /* can shrink, won't grow */

  align-self: center;   /* override align-items for this item */
  order: 2;              /* visual order (not DOM order) */
}

Common Patterns

/* ✅ Perfect centering — one of the most asked interview questions */
.center {
  display: flex;
  justify-content: center;
  align-items: center;
}

/* ✅ Space between nav items, logo left, links right */
.navbar {
  display: flex;
  justify-content: space-between;
  align-items: center;
}

/* ✅ Sticky footer layout */
body { display: flex; flex-direction: column; min-height: 100vh; }
main { flex: 1; }   /* grows to fill — footer pushed down ✅ */

/* ✅ Equal-width columns */
.col { flex: 1; }  /* all columns share space equally */
✅ Main Axis vs Cross Axis
  • flex-direction: row → main axis = horizontal, cross = vertical
  • flex-direction: column → main axis = vertical, cross = horizontal
  • justify-content always controls the main axis
  • align-items always controls the cross axis
✅ Use Flexbox for
  • 1-dimensional layouts (row OR column)
  • Navigation bars, toolbars
  • Centering content
  • Equal-height sidebar + main
  • Card groups with consistent height

CSS Grid — Complete Guide

Container Properties

.grid {
  display: grid;

  /* Define columns */
  grid-template-columns: 200px 1fr 1fr;    /* 3 cols */
  grid-template-columns: repeat(3, 1fr);   /* 3 equal */
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr)); /* responsive ✅ */

  /* Define rows */
  grid-template-rows: auto 1fr auto;   /* header, main, footer */

  gap: 16px;                 /* row + column gap */
  column-gap: 24px;         /* column gap only */
  row-gap: 12px;            /* row gap only */

  justify-items: stretch;  /* horizontal align of items in cell */
  align-items: stretch;    /* vertical align of items in cell */
}

Item Placement

.item {
  /* Span across columns / rows */
  grid-column: 1 / 3;      /* col line 1 to 3 (2 cols) */
  grid-column: 1 / -1;     /* full width ✅ */
  grid-column: span 2;     /* span 2 columns */

  grid-row: 1 / 3;         /* row line 1 to 3 */

  /* Named placement */
  grid-area: header;        /* must match template-areas name */

  justify-self: center;    /* override justify-items for this item */
  align-self: end;          /* override align-items for this item */
}

grid-template-areas — visual layout

.layout {
  display: grid;
  grid-template-columns: 220px 1fr;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header  header"
    "sidebar main  "
    "footer  footer";
  min-height: 100vh;
}
header  { grid-area: header; }
aside   { grid-area: sidebar; }
main    { grid-area: main; }
footer  { grid-area: footer; }

/* ✅ Responsive card grid — auto layout without media queries */
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: 20px;
  /* ← creates as many columns as fit, each min 280px, max 1fr */
}

Flexbox vs CSS Grid — When to Use Which

Flexbox — 1D layout

  • Works in one direction at a time — row or column
  • Content-driven — items determine their own size
  • Ideal for components (nav bar, button group, card header)

Use when:

  • ✅ Navigation / toolbar
  • ✅ Centering one element
  • ✅ Row of buttons or icons
  • ✅ Sidebar + main content side by side
  • ✅ Wrapping tag chips

CSS Grid — 2D layout

  • Works in two directions — rows AND columns together
  • Layout-driven — parent defines the grid, children placed in it
  • Ideal for page layouts and complex grids

Use when:

  • ✅ Full page layout (header/sidebar/main/footer)
  • ✅ Card grid with consistent row height
  • ✅ Dashboard with tiles
  • ✅ Magazine / article layout
  • ✅ Any 2D positioning needed

Side-by-side comparison

FlexboxCSS Grid
Dimensions1D — row OR column2D — rows AND columns
ControlChildren control their sizeParent controls the grid
AlignmentAlong one axis primarilyBoth axes independently
Overlapping items❌ Not supported✅ Items can overlap using grid-area
Browser supportAll browsers ✅All modern browsers ✅
Best forComponents, nav, button groupsPage layouts, dashboards, galleries
✅ They work great TOGETHER

Use Grid for the overall page layout, then Flexbox inside each grid area to arrange the content within it. They are complementary, not competing.

CSS Custom Properties (Variables)

Defining & Using Variables

/* Define on :root — globally available */
:root {
  --primary: #3b82f6;
  --accent:  #f97316;
  --text:    #1f2937;
  --radius:  8px;
  --spacing-md: 16px;
}

/* Use with var() */
.btn {
  background: var(--primary);
  border-radius: var(--radius);
  padding: var(--spacing-md);
  /* fallback value */
  color: var(--text, #000);
}

Dark / Light Theme Toggle

:root {
  --bg:   #ffffff;
  --text: #111111;
}
[data-theme="dark"] {
  --bg:   #0d1117;
  --text: #e6edf3;
}
/* OS-level preference */
@media (prefers-color-scheme: dark) {
  :root { --bg: #0d1117; }
}

/* Change theme via JS */
document.documentElement
  .setAttribute('data-theme', 'dark');

CSS Variables in Angular — :host & ::ng-deep

/* component.scss — Angular ViewEncapsulation */
:host {
  /* Styles the component's HOST element itself */
  display: block;
  --card-radius: 8px;   /* scoped CSS variable */
}
:host(.active) { border: 2px solid blue; }   /* conditional host class */
:host-context(.dark-theme) { background: #000; } /* parent context */

/* ::ng-deep — pierce through ViewEncapsulation to style child components */
/* DEPRECATED but still used — prefer CSS variables instead */
::ng-deep .mat-button { border-radius: 4px; }

/* ✅ Better — expose CSS variable from parent, consume in child */
.parent     { --btn-radius: 4px; }
.child .btn { border-radius: var(--btn-radius, 8px); }
✅ CSS Variables vs SCSS Variables

SCSS variables ($primary) are compiled away at build time — they don't exist in the browser. CSS custom properties (--primary) are live in the browser and can be changed at runtime via JavaScript. Always prefer CSS custom properties for theming.

Responsive Design — Media Queries & Modern Techniques

Media Queries

/* Mobile-first approach (recommended) */
/* Start with mobile styles, scale up */

.grid {
  grid-template-columns: 1fr;    /* mobile: 1 col */
}
@media (min-width: 768px) {
  .grid { grid-template-columns: repeat(2, 1fr); }
}
@media (min-width: 1024px) {
  .grid { grid-template-columns: repeat(3, 1fr); }
}

/* Other media features */
@media (prefers-color-scheme: dark) { }
@media (prefers-reduced-motion: reduce) { }
@media (orientation: landscape) { }
@media print { }

🆕 Container Queries (CSS 2023)

/* Respond to PARENT container size, not viewport */
/* Solves: same component in sidebar (narrow) */
/* vs main content (wide) — different layouts */
.card-wrapper {
  container-type: inline-size;
  container-name: card;
}
@container card (min-width: 400px) {
  .card {
    display: flex;         /* horizontal when wide */
  }
}
@container card (max-width: 399px) {
  .card {
    flex-direction: column; /* vertical when narrow */
  }
}

CSS Transitions & Animations

Transitions — between two states

.btn {
  background: blue;
  /* property duration timing-function delay */
  transition: background .3s ease;
  transition: all .3s ease;          /* ❌ avoid 'all' — perf */
  transition: transform .3s, opacity .2s; /* ✅ specific */
}
.btn:hover { background: darkblue; }

/* ✅ High performance: only animate these two — no reflow */
/* transform and opacity run on the GPU compositor thread */
.card       { transform: translateY(0); opacity: 1; transition: transform .3s; }
.card:hover { transform: translateY(-4px); }

Keyframe Animations

@keyframes fadeInUp {
  from { opacity: 0; transform: translateY(20px); }
  to   { opacity: 1; transform: translateY(0); }
}
@keyframes spin {
  to { transform: rotate(360deg); }
}

.element {
  /* name duration timing fill-mode delay iterations */
  animation: fadeInUp .5s ease forwards;
  animation: spin 1s linear infinite;
}

/* Respect user's motion preference */
@media (prefers-reduced-motion: reduce) {
  * { animation-duration: .01ms !important; }
}

Performance — What can be animated cheaply?

PropertyTriggersPerformance
transform (translate, scale, rotate)Composite only🟢 GPU — cheapest
opacityComposite only🟢 GPU — cheapest
background-color, colorPaint🟡 OK — no layout shift
width, height, top, leftLayout + Paint + Composite🔴 Expensive — causes reflow
✅ Golden Rule — Only animate transform & opacity

Use transform: translateX() instead of left. Use transform: scale() instead of width/height. Add will-change: transform to hint the browser to promote the element to its own GPU layer before animation starts.

Senior-Level Interview Q&A

Q1. What is CSS specificity? How is it calculated? What wins — class or ID?
Specificity determines which rule applies when multiple rules target the same element. It's calculated as (A, B, C): A = inline/ID selectors, B = classes/attributes/pseudo-classes, C = elements/pseudo-elements. A higher A always wins regardless of B and C. So ID (0,1,0,0) always beats class (0,0,1,0). !important overrides all specificity. Best practice: keep specificity low by using classes only, avoid IDs in CSS.
Q2. What is the difference between display:none, visibility:hidden, and opacity:0?
display:none — removes from layout entirely, no space, not accessible, no events. visibility:hidden — invisible but still occupies space in layout, children can override with visibility:visible, no events. opacity:0 — invisible, occupies space, still receives pointer events (clicks!), can be animated via GPU cheaply. Use opacity:0 + pointer-events:none for fade-out animations.
Q3. When would you use Flexbox vs CSS Grid?
Flexbox is for 1D layouts — items in a row or column. Content drives the sizing. Best for: nav bars, button groups, centering. CSS Grid is for 2D layouts — rows and columns simultaneously. Layout drives the structure. Best for: full page layout, dashboards, card grids. They work together: use Grid for the page, Flexbox inside each grid area for the content.
Q4. How does position:absolute work? What is its containing block?
An absolute element is removed from document flow and positioned relative to its nearest positioned ancestor — any ancestor with position other than static (i.e. relative, absolute, fixed, sticky). If no positioned ancestor exists, it's positioned relative to the initial containing block (viewport). Common pattern: add position:relative to a parent to contain an absolute child (tooltip, badge, dropdown).
Q5. Why should you only animate transform and opacity for performance?
Animating properties like width, top, or margin triggers the full rendering pipeline: Layout → Paint → Composite. This is expensive. transform and opacity skip Layout and Paint entirely — they only trigger the Composite step, which runs on the GPU compositor thread separate from the main thread. This results in buttery-smooth 60fps animations even when the main thread is busy. Use will-change: transform to promote the element to its own GPU layer before animation begins.
Q6. What is ::ng-deep and why is it deprecated in Angular?
::ng-deep (previously /deep/ or >>>) pierces Angular's ViewEncapsulation to style child component internals. It's deprecated because it disables encapsulation globally for the affected rule, making styles bleed unpredictably. The recommended alternatives are: (1) expose CSS custom properties (variables) as theming API of the child component, (2) use ViewEncapsulation.None if the child is intentionally unencapsulated, or (3) pass style inputs to the child component.
