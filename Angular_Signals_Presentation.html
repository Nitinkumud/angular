Foundation

What are Signals?

A signal is a reactive wrapper around a value. It notifies all interested consumers whenever that value changes. Angular tracks where each signal is read and automatically re-evaluates consumers on update.

✍️ Writable Signal

const count = signal(0);

// read — also registers dependency
console.log(count());      // 0

// set a new value directly
count.set(3);

// derive from the current value
count.update(v => v + 1);

🔒 Readonly Signal

class CounterState {
  private _count = signal(0);

  // expose as read-only to consumers
  count = this._count.asReadonly();

  increment() {
    this._count.update(v => v + 1);
  }
}
💡 Signals are getter functions — calling count() reads the value AND registers the caller as a consumer inside any active reactive context.
⚠️ Readonly signals do not prevent deep-mutation of their value — they only prevent calling .set() or .update().

Derivation

Computed Signals

Read-only signals that automatically derive their value from one or more other signals. Lazy and memoized by design.

const count = signal(0);
const doubleCount = computed(() => count() * 2);

// Dynamic dependencies — count is only tracked when showCount is true
const showCount   = signal(false);
const conditional = computed(() => {
  if (showCount()) return `Count: ${count()}`;
  return 'Nothing to see here!';  // count NOT tracked here
});

🦥 Lazy

Value not calculated until first read — no wasted work on startup.

📦 Memoized

Result is cached and reused until a dependency changes. Safe for expensive operations like array filtering.

🔗 Dynamic deps

Only signals actually read during the derivation are tracked. Deps can be added or removed per-run.

⚠️ You cannot .set() a computed signal — it produces a compile error since computed() returns a read-only Signal<T>, not a WritableSignal<T>.

Core Concept

Reactive Contexts

A reactive context is a runtime scope where Angular intercepts signal reads to build a live dependency graph. Angular enters one automatically in these scenarios:

  • effect() / afterRenderEffect() callback
  • computed() signal evaluation
  • linkedSignal() evaluation
  • resource() params or loader function
  • Component template rendering (incl. host bindings)
🚨 The reactive context is synchronous only. Signal reads that occur after an await are outside the context and go untracked.

❌ Avoid — reads after await

effect(async () => {
  const data = await fetchUser();
  // theme() read AFTER await — not tracked!
  console.log(theme());
});

✅ Prefer — read before await

effect(async () => {
  const t = theme(); // tracked ✓
  const data = await fetchUser();
  console.log(t, data);
});

Fine-Grained Control

Reading Without Tracking — untracked()

Sometimes you need to read a signal inside a reactive context without creating a dependency on it. Use untracked() to opt out selectively.

// Effect runs when currentUser changes only — counter is read but not tracked
effect(() => {
  console.log(
    `User: ${currentUser()} — counter: ${untracked(counter)}`
  );
});

// Also useful when calling external code that itself reads signals
effect(() => {
  const user = currentUser();       // ← tracked dependency
  untracked(() => {
    // Any signals loggingService reads internally are NOT dependencies
    this.loggingService.log(`User: ${user}`);
  });
});
💡 Use assertNotInReactiveContext(fn) to guard utility functions that must never run inside a reactive context — Angular throws a clear, actionable error pointing to the right API if violated.

Side Effects

effect() & afterRenderEffect()

Use effect() only when you need to bridge the reactive signal world with non-reactive APIs (DOM, localStorage, third-party libs). Prefer computed derivations for everything else.

When to use effect()

  • Logging / analytics on state change
  • Syncing to localStorage or IndexedDB
  • Calling third-party libraries (charts, maps)
  • DOM manipulation outside Angular templates
⚠️ Never use effect() to synchronise signals with each other. Use computed() or linkedSignal() instead — they stay in the reactive graph. 
// effect() — runs on every change to count()
effect(() => {
  document.title = `Count: ${count()}`;
});

// afterRenderEffect — runs after each
// render cycle finishes (good for DOM reads)
afterRenderEffect(() => {
  const el = this.ref.nativeElement;
  el.scrollTop = el.scrollHeight;
});
ℹ️ Every tracked signal read inside effect() automatically becomes a dependency — the effect re-runs when any of them change.

Advanced Derivation

linkedSignal() — Basics

A writable signal whose default value is computed from other signals. When the dependency changes, the signal resets to the new computed value — but users can still override it manually.

The Problem

// ❌ selectedOption may become stale
// when shippingOptions signal changes
selectedOption = signal(
  this.shippingOptions()[0]
);

The Fix — linkedSignal()

// ✅ resets to first option whenever
// shippingOptions signal changes
selectedOption = linkedSignal(
  () => this.shippingOptions()[0]
);

// user can still override it
selectedOption.set('Sea');

How it behaves

const options  = signal(['Ground','Air','Sea']);
const selected = linkedSignal(() => options()[0]);

console.log(selected()); // 'Ground'

selected.set(options()[2]);
console.log(selected()); // 'Sea'

// upstream change → resets to new first item
options.set(['Email','Will Call','Post']);
console.log(selected()); // 'Email' ← reset!
💡 linkedSignal() = computed() + writable. Unlike computed(), consumers can call .set() or .update() on it.

Advanced Derivation NEW

linkedSignal() — source + computation + previous

For cases where the reset logic needs to inspect the previous value, pass a { source, computation } object. The computation receives both the new source value and a previous snapshot.

interface ShippingMethod { id: number; name: string; }

selectedOption = linkedSignal<ShippingMethod[], ShippingMethod>({
  // source: the signal that triggers a reset
  source: this.shippingOptions,

  // computation: receives (newSourceValue, previous)
  computation: (newOptions, previous) => {
    // If the user's previous pick is still in the new list, keep it
    return newOptions.find(opt => opt.id === previous?.value.id)
      ?? newOptions[0];  // otherwise fall back to first option
  },
});

previous object properties

  • previous.source — the prior value of the source signal
  • previous.value — the prior value of the linkedSignal itself

Custom equality on linkedSignal

linkedSignal(() => activeUser(), {
  equal: (a, b) => a.id === b.id
});

Custom equality works the same as with signal() and computed() — downstream consumers only re-run when the equality function returns false.

⚠️ When using the previous parameter, you must provide both generic type arguments explicitly: linkedSignal<SourceType, OutputType>().

resource() — Full Status Lifecycle

Every resource() exposes a status() signal describing its current state. Understanding each status is essential for building correct loading and error UI.

'idle'
value() → undefined
params returned undefined — loader will not run.
'loading'
value() → undefined
Loader running because params signal changed.
'reloading'
value() → previous value
Loader triggered manually via .reload().
'resolved'
value() → resolved data
Loader completed successfully.
'error'
value() → undefined
Loader threw. Read details via .error().
'local'
value() → local value
Value was set locally via .set() or .update().
const userResource = resource({
  params: () => ({ id: userId() }),
  loader: ({ params, abortSignal }) =>
    fetch(`/api/users/${params.id}`, { signal: abortSignal }).then(r => r.json())
});

userResource.status()   // 'idle' | 'loading' | 'reloading' | 'resolved' | 'error' | 'local'
userResource.isLoading() // true when loading OR reloading
userResource.hasValue()  // type-guard: strips undefined AND guards against error state
userResource.reload()    // programmatic re-fetch → 'reloading' status

debounced() — Delay Reacting to a Signal

Wraps a signal so downstream consumers only see the value after it stops changing for a given duration. Returns a Resource whose value() is the debounced output.

import { debounced, resource, signal } from '@angular/core';

export class Search {
  query = signal('');

  // Only fires when query hasn't changed for 300 ms
  debouncedQuery = debounced(this.query, 300);

  results = resource({
    params: () => this.debouncedQuery.value(),
    loader: ({ params }) => fetchResults(params),
  });
}

Custom wait function

debouncedQuery = debounced(query, (val, last) => {
  if (last.status === 'error') return; // retry fast
  const ms = val.length < 3 ? 500 : 200;
  return new Promise(res => setTimeout(res, ms));
});

Status during debounce

  • Timer counting down → status is 'loading', value is last resolved value
  • Timer expires → status becomes 'resolved'
  • Source throws → immediately 'error', no timer
⚠️ debounced() must be called inside an injection context. To use it elsewhere, pass { injector } as the third argument. Angular auto-destroys the resource when the injector is destroyed.

OnPush Components & Equality Functions

Signals + OnPush

Reading a signal in an OnPush component's template registers it as a dependency. When that signal changes, Angular automatically marks the component dirty — no manual markForCheck() required.

Signal value changes
↓
Angular marks component dirty
↓
Next CD cycle re-renders

Custom Equality

Provide an equality function so signals skip updates when the value hasn't meaningfully changed — perfect for objects and arrays.

import isEqual from 'lodash/isEqual';

const data = signal(['test'], {
  equal: isEqual
});

// Same content, different array reference:
// isEqual → true → no update fired 🎉
data.set(['test']);
💡 Default equality is Object.is() — referential. Use isEqual from lodash for deep comparison of objects and arrays.

// Observable → Signal
const data$ = http.get('/api/data');

// Signal → Observable
const count  = signal(0);
const count$ = toObservable(count);

Interview Prep

Memory Tips & Senior Q&A

🏭 Factory Analogy

  • signal() = Raw material — changeable reactive state
  • computed() = Assembly line — derived, auto-updated, read-only
  • linkedSignal() = Resettable shelf — writable but resets on upstream change
  • effect() = Shipping dock — reacts to change, talks to outside world
  • resource() = Async delivery truck — brings external data into the signal world
  • debounced() = Slow mail filter — waits for a lull before delivering
⚡ Rule: State → signal() · Derived → computed() · Dependent writable → linkedSignal() · Async → resource() · Side-effects → effect()
Q1. computed() vs linkedSignal() — when to use which?
Use computed() when you only need read-only derived state. Use linkedSignal() when users also need to override the derived value manually — e.g. a selected option that resets when options change but can still be chosen by the user.
Q2. What is the difference between 'loading' and 'reloading' status in resource()?
'loading' fires when the params signal changed, so value() is undefined. 'reloading' fires when .reload() is called manually — value() still returns the previous resolved value, so you can show stale data while refreshing.
Q3. What does debounced() return and how is it different from a plain signal?
It returns a Resource, not a plain signal. Its value() only updates after the source has been stable for the given delay. During the wait, status() is 'loading' and value() holds the last resolved result — enabling stale-while-revalidate patterns with zero extra code.
