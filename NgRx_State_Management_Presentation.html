01

Why NgRx? The Problem It Solves

❌ Problem — Without NgRx

  • Each component fetches its own data — duplicate HTTP calls
  • Parent-child prop drilling — hard to maintain
  • State scattered across services, components
  • Race conditions — async state hard to debug
  • No history / time-travel debugging
  • Side effects mixed with component logic

✅ Solution — With NgRx

  • Single source of truth — one Store for all state
  • Predictable state changes — only via Actions + Reducers
  • Immutable state — no accidental mutations
  • Side effects isolated in Effects
  • Redux DevTools — time-travel debugging
  • Scales well for large enterprise apps

NgRx is based on the Redux Pattern

3 Principles of Redux:

  • Single source of truth — one global Store
  • State is read-only — change only via Actions
  • Pure functions — Reducers must be pure (no side effects)

When to use NgRx?

  • State shared across many components
  • Complex async data flows
  • Need for undo/redo or time-travel debug
  • Team of 3+ developers on same app
📦 NgRx packages

@ngrx/store · @ngrx/effects · @ngrx/entity · @ngrx/router-store · @ngrx/store-devtools · @ngrx/component-store

⚠️ Don't over-engineer

NgRx adds boilerplate. For simple apps, use Services + BehaviorSubject. Use NgRx when state is complex, shared, or async at scale.

Step 1 — Component dispatches

Component calls store.dispatch(loadProducts()). It does NOT call the service directly.

Step 2–3 — Reducer OR Effect

Sync changes go to Reducer. Async (HTTP) go to Effect → API → dispatch success/failure action.

Step 4–5 — Selector reads state

Component reads state via store.select(selector). Selectors are memoized — recalculate only when input changes.

Store & State — The Single Source of Truth

What is the Store?

A single immutable JavaScript object that holds the entire application state. Think of it as a client-side database.


  • State is read-only — never mutate directly
  • Only Reducers can produce a new state
  • Observable — components subscribe to slices
  • Registered at app level via provideStore()

AppState Shape

The global state is a tree of feature states:

// app.state.ts
export interface AppState {
  products: ProductState;
  cart:     CartState;
  auth:     AuthState;
  router:   RouterReducerState;
}
// product.reducer.ts — State interface (from your diagram)
import { Product } from './product.model';

export interface ProductState {
  products: Product[];   // list of products
  loading:  boolean;     // true while API is loading
  error:    string | null; // error message if API fails
}

export const initialState: ProductState = {
  products: [],
  loading:  false,
  error:    null
};

// app.config.ts — register store (standalone Angular 17+)
import { provideStore } from '@ngrx/store';
import { provideEffects } from '@ngrx/effects';
import { provideStoreDevtools } from '@ngrx/store-devtools';

export const appConfig: ApplicationConfig = {
  providers: [
    provideStore({ products: productReducer }),  // register reducers
    provideEffects([ProductEffects]),             // register effects
    provideStoreDevtools({ maxAge: 25 })          // Redux DevTools
  ]
};
✅ Feature State (Lazy Loading)

Use provideState('products', productReducer) inside lazy-loaded routes to register feature state only when needed. Keeps the initial bundle small.

Actions — Describe What Happened

What is an Action?

  • A plain object with a type string and optional payload
  • The only way to communicate with the NgRx Store
  • Describes an event: "what happened", not how
  • Naming convention: [Source] Event

Action Naming Convention

Format: [Feature] EventName

  • [Product] Load Products → trigger load
  • [Product] Load Products Success → API succeeded
  • [Product] Load Products Failure → API failed
  • [Product Page] Add To Cart → UI event
  • [Cart API] Checkout Success → API event
// product.actions.ts — exactly as in your diagram
import { createAction, props } from '@ngrx/store';
import { Product } from './product.model';

// Trigger action — no payload needed
export const loadProducts = createAction('[Product] Load Products');

// Success — carries the products array as payload
export const loadProductsSuccess = createAction(
  '[Product] Load Products Success',
  props<{ products: Product[] }>()
);

// Failure — carries the error message
export const loadProductsFailure = createAction(
  '[Product] Load Products Failure',
  props<{ error: string }>()
);

// CRUD actions with props
export const addProduct    = createAction('[Product] Add Product',    props<{ product: Product }>());
export const deleteProduct = createAction('[Product] Delete Product', props<{ id: number }>());
export const updateProduct = createAction('[Product] Update Product', props<{ product: Product }>());
✅ Best Practice — Action Groups (NgRx 15+)

Use createActionGroup to group related actions. Reduces boilerplate and avoids repeating the source string.

export const ProductActions = createActionGroup({
  source: 'Product',
  events: {
    'Load Products':         emptyProps(),
    'Load Products Success': props<{ products: Product[] }>(),
    'Load Products Failure': props<{ error: string }>(),
  }
});
⚠️ Actions should be events, not commands
  • ✅ [Cart] Item Added — event (what happened)
  • ❌ [Cart] AddItem — command (what to do)
  • Actions describe facts, not instructions

Reducers — Decide How State Changes

What is a Reducer?

A pure function that takes the current state + an action, and returns a new state object. It never mutates the existing state — it creates a new reference. (state, action) => newState

// product.reducer.ts — from your diagram, enhanced
import { createReducer, on } from '@ngrx/store';
import * as ProductActions from './product.actions';
import { ProductState, initialState } from './product.state';

export const productReducer = createReducer(
  initialState,

  // When loadProducts fires → set loading: true
  on(ProductActions.loadProducts, (state) => ({
    ...state,
    loading: true,
    error: null
  })),

  // When API success → store products, stop loading
  on(ProductActions.loadProductsSuccess, (state, { products }) => ({
    ...state,
    loading: false,
    products          // shorthand: products: products
  })),

  // When API fails → store error, stop loading
  on(ProductActions.loadProductsFailure, (state, { error }) => ({
    ...state,
    loading: false,
    error
  })),

  // Add product — immutably append
  on(ProductActions.addProduct, (state, { product }) => ({
    ...state,
    products: [...state.products, product]
  })),

  // Delete product — filter out by id
  on(ProductActions.deleteProduct, (state, { id }) => ({
    ...state,
    products: state.products.filter(p => p.id !== id)
  }))
);
✅ Pure Function Rules
  • No HTTP calls, no side effects
  • Same input always produces same output
  • Always spread state: { ...state, changes }
  • Never mutate: ❌ state.products.push()

Effects — Handle Async Side Effects

What is an Effect?

An Effect listens for specific Actions, performs async work (HTTP, localStorage, WebSocket), then dispatches a new Action with the result. Keeps Reducers pure and components clean.

// product.effects.ts — from your diagram, enhanced
import { Injectable } from '@angular/core';
import { Actions, createEffect, ofType } from '@ngrx/effects';
import { catchError, map, switchMap, of } from 'rxjs';
import { ProductService } from './product.service';
import * as ProductActions from './product.actions';

@Injectable()
export class ProductEffects {

  // Load products when 'loadProducts' action dispatched
  loadProducts$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.loadProducts),    // listen for this action
      switchMap(() =>                            // cancel prev, start new
        this.productService.getAll().pipe(
          map(products =>
            ProductActions.loadProductsSuccess({ products })  // ✅ success
          ),
          catchError(err =>
            of(ProductActions.loadProductsFailure({ error: err.message })) // ❌ failure
          )
        )
      )
    )
  );

  // Add product to API
  addProduct$ = createEffect(() =>
    this.actions$.pipe(
      ofType(ProductActions.addProduct),
      switchMap(({ product }) =>
        this.productService.create(product).pipe(
          map(saved => ProductActions.loadProducts()),  // reload list
          catchError(err => of(ProductActions.loadProductsFailure({ error: err.message })))
        )
      )
    )
  );

  constructor(
    private actions$: Actions,
    private productService: ProductService
  ) {}
}

Which RxJS operator to use in Effects?

OperatorUse case
switchMapCancels previous — search, load (most common)
mergeMapAll run in parallel — delete, analytics
concatMapQueue in order — save form, cart checkout
exhaustMapIgnore new while busy — login button
✅ Non-dispatching Effects

Set { dispatch: false } for effects that don't dispatch new actions (e.g., navigate, localStorage, logging):

navigate$ = createEffect(() =>
  this.actions$.pipe(
    ofType(ProductActions.loadProductsSuccess),
    tap(() => this.router.navigate(['/products']))
  ), { dispatch: false }
);

Selectors — Read State Efficiently

What is a Selector?

  • A pure function to read / compute data from the Store
  • Memoized — recalculates only when input state changes
  • Composable — build complex selectors from simple ones
  • Prevents unnecessary re-renders

3 Types

  • Feature selector — gets the feature slice
  • Simple selector — reads one property
  • Composed selector — combines multiple, computes derived data
// product.selectors.ts
import { createFeatureSelector, createSelector } from '@ngrx/store';
import { ProductState } from './product.state';

// Step 1 — Feature selector (entry point to product state)
export const selectProductState = createFeatureSelector<ProductState>('products');

// Step 2 — Simple selectors (read individual properties)
export const selectAllProducts = createSelector(selectProductState, state => state.products);
export const selectLoading     = createSelector(selectProductState, state => state.loading);
export const selectError       = createSelector(selectProductState, state => state.error);

// Step 3 — Composed selectors (computed / derived data)
export const selectProductCount = createSelector(
  selectAllProducts,
  products => products.length
);

// Selector with props — find a product by id
export const selectProductById = (id: number) =>
  createSelector(selectAllProducts, products => products.find(p => p.id === id));

// Combine multiple feature states
export const selectProductsInCart = createSelector(
  selectAllProducts,
  selectCartItems,  // from cart selector
  (products, cartItems) => products.filter(p => cartItems.includes(p.id))
);
🎯 Memoization — Why it matters

If the products array hasn't changed, selectProductCount returns the cached result without recalculating. This prevents expensive re-computation on every Store update — critical for performance.

Component — Dispatch & Select

Component's role — it should be thin (Smart/Dumb pattern)

Component only dispatches actions and reads state via selectors. No direct API calls, no business logic.

// product-list.component.ts
import { Component, OnInit } from '@angular/core';
import { Store } from '@ngrx/store';
import * as ProductActions from './product.actions';
import { selectAllProducts, selectLoading, selectError } from './product.selectors';

@Component({
  selector: 'app-product-list',
  template: `
    <div *ngIf="loading$ | async">Loading...</div>
    <div *ngIf="error$ | async as err" class="error">{{ err }}</div>
    <div *ngFor="let p of products$ | async">
      {{ p.name }} — {{ p.price }}
      <button (click)="delete(p.id)">Delete</button>
    </div>
    <button (click)="load()">Reload</button>
  `
})
export class ProductListComponent implements OnInit {

  // Select data from store using selectors (Observables)
  products$ = this.store.select(selectAllProducts);
  loading$  = this.store.select(selectLoading);
  error$    = this.store.select(selectError);

  constructor(private store: Store) {}

  ngOnInit() {
    // Dispatch action — component does NOT call service
    this.store.dispatch(ProductActions.loadProducts());
  }

  delete(id: number) {
    this.store.dispatch(ProductActions.deleteProduct({ id }));
  }

  load() {
    this.store.dispatch(ProductActions.loadProducts());
  }
}
✅ Always use async pipe — no manual subscribe/unsubscribe needed

products$ | async automatically subscribes and unsubscribes. Never .subscribe() manually in a component unless absolutely necessary.

NgRx Entity — Manage Collections Efficiently

What is NgRx Entity?

@ngrx/entity provides an adapter for managing normalized collections (arrays of objects with IDs). Removes boilerplate for CRUD operations.

  • Stores items as { ids: [], entities: {} }
  • Fast lookup by ID — O(1) vs O(n) filter
  • Built-in CRUD adapter methods

EntityState Structure

// Instead of: Product[]
// NgRx Entity stores:
{
  ids: [1, 2, 3],
  entities: {
    1: { id:1, name:'Phone'   },
    2: { id:2, name:'Laptop'  },
    3: { id:3, name:'Tablet'  }
  },
  loading: false,  // your custom props
  error: null
}
// product.reducer.ts — with NgRx Entity
import { createEntityAdapter, EntityAdapter, EntityState } from '@ngrx/entity';

export interface ProductState extends EntityState<Product> {
  loading: boolean;
  error: string | null;
}

export const adapter: EntityAdapter<Product> = createEntityAdapter<Product>();
export const initialState: ProductState = adapter.getInitialState({ loading: false, error: null });

export const productReducer = createReducer(
  initialState,
  on(ProductActions.loadProductsSuccess, (state, { products }) =>
    adapter.setAll(products, { ...state, loading: false })  // replaces all
  ),
  on(ProductActions.addProduct, (state, { product }) =>
    adapter.addOne(product, state)       // adds one
  ),
  on(ProductActions.deleteProduct, (state, { id }) =>
    adapter.removeOne(id, state)         // removes by id
  ),
  on(ProductActions.updateProduct, (state, { product }) =>
    adapter.upsertOne(product, state)    // add or update
  )
);

// Built-in selectors from adapter
export const { selectAll, selectEntities, selectIds, selectTotal } =
  adapter.getSelectors(selectProductState);

Feature State, Router Store & DevTools

Feature State — Lazy Loading

Register state only when the feature module/route loads. Keeps the initial bundle small.

// products.routes.ts
export const productRoutes: Routes = [{
  path: 'products',
  providers: [
    provideState('products', productReducer),
    provideEffects([ProductEffects])
  ],
  loadComponent: () => import('./product-list.component')
}];

Router Store

Syncs Angular Router state into the NgRx Store — URL, params, query params become part of state.

// app.config.ts
provideRouterStore()  // in providers

// Use in selectors
import { selectRouteParam } from '@ngrx/router-store';

productId$ = this.store.select(
  selectRouteParam('id')  // reads :id from URL
);

🧠 Memory Tips & Mnemonics

🏦 Bank Analogy
  • Store = Bank vault — all money (state) in one safe place
  • Action = Withdrawal/deposit slip — describes what you want
  • Reducer = Bank teller — processes the slip, updates balance
  • Effect = Bank's back office — verifies with external system (API)
  • Selector = ATM balance screen — shows you only what you need
🔑 One-Line Rules
  • Action → "What happened?" (event, not command)
  • Reducer → "How does state change?" (pure, sync only)
  • Effect → "What side effects run?" (async, HTTP)
  • Selector → "What data does UI need?" (memoized)
  • Store → "Where is all state?" (single source of truth)
📝 A.R.E.S.S

Action — describes events
Reducer — pure state fn
Effect — async work
Selector — read state
Store — holds all state

🎯 Action triplet pattern

Every async operation has 3 actions:
trigger → load
success → loadSuccess
failure → loadFailure

⚡ Effect operator choice

switchMap → load/search (cancel prev)
concatMap → save/order (queue)
mergeMap → parallel ops
exhaustMap → login (ignore new)

Senior-Level Interview Q&A

Q1. What is NgRx and why would you use it over a Service + BehaviorSubject?
NgRx is a reactive state management library based on the Redux pattern. I use it over Service+BehaviorSubject when the app has complex shared state, async side effects at scale, and needs Redux DevTools for debugging. For simpler apps, Service+BehaviorSubject is less boilerplate. NgRx enforces one-way data flow, immutable state, and separates side effects into Effects.
Q2. What is the difference between a Reducer and an Effect?
A Reducer is a pure synchronous function — takes state + action, returns new state. No async, no side effects. An Effect is impure and async — it listens for actions, performs API calls or other side effects, and dispatches new actions with the results. Effects keep Reducers pure.
Q3. Why are Selectors memoized and why does it matter?
Selectors cache their last result. If the input state slice hasn't changed, they return the same reference without recalculating. This prevents unnecessary Angular change detection cycles and expensive derived data recomputation on every Store update — critical for performance in large apps.
Q4. When would you use switchMap vs concatMap in Effects?
Use switchMap for search/load operations where you want to cancel previous requests when a new one starts. Use concatMap when order matters and all requests should complete in sequence (e.g., save form, cart checkout). Use exhaustMap for login — ignore new requests while one is in progress.
Q5. What is NgRx Entity and when would you use it?
NgRx Entity provides an adapter to manage normalized collections as { ids: [], entities: {} } instead of plain arrays. Use it when you have a collection of objects with unique IDs and need fast CRUD operations. The adapter provides addOne, removeOne, upsertOne, setAll — no boilerplate needed. Lookup by ID is O(1) vs O(n) with array filter.
Q6. What is Feature State and why is it important?
Feature State registers a slice of the Store only when a lazy-loaded route/module is activated — using provideState(). This keeps the initial bundle small and loads state only when needed. The feature state is merged into the global store at runtime and removed when the feature is unloaded (unless configured otherwise).
