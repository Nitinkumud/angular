Fundamentals

What is Hydration?

Official Definition — angular.dev/guide/hydration

Hydration is the process that restores a server-side rendered application on the client. It includes: reusing server-rendered DOM structures, persisting application state, transferring data already retrieved by the server, and attaching event listeners to existing DOM nodes.

❌ Without Hydration (CSR Takeover)

  • Server sends fully rendered HTML string
  • Angular bootstraps on the client
  • Angular destroys all server-rendered DOM
  • Angular re-creates the entire component tree from scratch
  • Visible UI flicker as content is wiped and redrawn
  • All data fetched again on client (double fetch)
  • Hurts LCP, CLS — bad for Core Web Vitals

✅ With Hydration

  • Server sends fully rendered HTML string
  • Angular bootstraps on the client
  • Angular walks the existing DOM and matches component nodes
  • Reuses DOM nodes — no destroy & recreate
  • No visible flicker — content is stable from first paint
  • Server-fetched data transferred via TransferState
  • Dramatically improves LCP, FID, CLS

Key Insight

Hydration bridges the gap between the fast initial paint (SSR) and full interactivity (client bootstrap). Without it, you pay the SSR cost with none of the benefit — users see a flash of empty content as Angular destroys and re-renders.

Core Web Vitals

LCP — Largest Contentful Paint

What is LCP?

LCP measures how fast the largest visible content element (image, heading, or banner) appears on screen after the user navigates to the page. It is the single most important indicator of perceived load speed.

What Elements Count for LCP?

  • <img> tags — product images, hero banners
  • <image> inside SVG
  • CSS background-image elements (largest visible)
  • <video> poster image
  • Block-level text — large headings (<h1>, <p>)

LCP Score Thresholds

ScoreRatingGoogle's Verdict
0 – 2.5sGoodGoogle rewards this with better ranking
2.5 – 4sNeeds WorkAcceptable, but room to improve
4s+PoorRanking penalty, users bounce

Real-World Example

You open Flipkart. The first thing that appears is a large product image or sale banner.


⏱️ LCP = the moment that big image becomes fully visible


If the image takes 5 seconds → LCP is Poor → user may leave → Google notices → ranking drops.

How Angular Hydration Improves LCP

❌ Without Hydration

Angular destroys server HTML → hero image disappears → re-renders → LCP fires twice, second time much later

✅ With Hydration

Server HTML stays intact → hero image never disappears → LCP fires once, at initial server paint

Pro Tip — How to Improve LCP

Use SSR + Hydration so critical content is in the HTML from the start. Add loading="eager" and fetchpriority="high" on your hero image. Use NgOptimizedImage directive in Angular — it adds these attributes automatically and serves modern formats (WebP/AVIF).

Core Web Vitals

FID — First Input Delay

What is FID?

FID measures the time from when a user first interacts with the page (click, tap, key press) to when the browser actually begins processing that event. It captures the responsiveness of the page during load.

Why Does FID Happen?

The browser's main thread is busy parsing and executing JavaScript. During this time it cannot respond to user input. The delay between the click and the response is FID.


User clicks button
↓
Main thread busy with JS
↓ (FID = this delay)
Browser processes click

FID Score Thresholds

ScoreRatingUser Experience
0 – 100msGoodFeels instant — user is happy
100 – 300msNeeds WorkNoticeable lag
300ms+PoorUser thinks site is broken

Real-World Example

You open a news website and immediately click "Subscribe". The button looks clickable. But nothing happens for 2 seconds.


The browser was busy downloading and running JavaScript. Your click was registered but processing was delayed by 2 seconds — that is a 2000ms FID.

How Angular Hydration Improves FID

❌ Without Event Replay

User clicks "Add to Cart" before Angular is ready → click is silently lost → user frustrated, clicks again

✅ With withEventReplay()

Click is captured and queued by JSAction → Angular ready → event replayed → user's action honoured

Note: FID is being replaced by INP

Google is transitioning from FID to INP (Interaction to Next Paint) as a Core Web Vital. INP measures all interactions (not just the first), making it a more complete responsiveness metric. Angular's Event Replay and smaller bundles improve both.

Core Web Vitals

CLS — Cumulative Layout Shift

What is CLS?

CLS measures the total amount of unexpected layout shifts that occur during the entire life of a page. It quantifies how much visible content jumps around — a score of 0 means nothing moved, higher means more instability.

How CLS is Calculated

For each layout shift: Impact Fraction × Distance Fraction = Shift Score


  • Impact Fraction — how much of the viewport was affected
  • Distance Fraction — how far the element moved (% of viewport)
  • CLS = sum of all individual shift scores

CLS Score Thresholds

ScoreRatingUser Experience
0 – 0.1GoodStable — content doesn't jump
0.1 – 0.25Needs WorkNoticeable shifts
0.25+PoorUser clicks wrong things

Common Causes of CLS

  • Images without width & height attributes
  • Ads / embeds injected above existing content
  • Web fonts causing text to reflow (FOUT)
  • Angular destroying and re-rendering the DOM (the big one!)
  • Dynamic content inserted above the fold

Real-World Example

You're reading an article. Just as you're about to click "Read More", an ad banner loads above it — everything shifts down by 200px — you accidentally click the ad instead.


This is a layout shift. CLS captures these frustrating moments.

Setup

Enabling Hydration

Prerequisite

Hydration works only with SSR. You must enable server-side rendering first via ng add @angular/ssr or configure it manually. Hydration is a client-side process that needs server-rendered HTML to reuse.

Method 1 — Angular CLI (Recommended)

Generate new project with SSR
ng new my-app --ssr

# Or add SSR to existing project
ng add @angular/ssr
Auto-configured

The CLI automatically adds provideClientHydration() to your app config. No manual step needed.

Method 2 — Manual (Standalone)

app.config.ts
import {
  bootstrapApplication,
  provideClientHydration
} from '@angular/platform-browser';
import { AppComponent } from './app/app.component';

bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration()
  ]
});

Method 3 — NgModule-based Apps

app.module.ts
import { provideClientHydration } from '@angular/platform-browser';
import { NgModule } from '@angular/core';

@NgModule({
  declarations: [AppComponent],
  bootstrap:    [AppComponent],
  providers:    [provideClientHydration()],  // ← Add this
})
export class AppModule {}

// ⚠ Also ensure provideClientHydration() is in your SERVER bootstrap providers

Internals

How Hydration Works Internally

The DOM Reconciliation Algorithm

Angular's hydration algorithm walks the server-rendered DOM tree in parallel with the component tree it expects from the component templates. For each component, it tries to claim the matching DOM nodes rather than creating new ones.

Phase 1: Serialization (Server)

  • Angular renders the app on the server
  • Serializes component tree structure as hydration annotation comments (<!--ngh=N-->)
  • Serializes TransferState as a JSON blob in a <script> tag
  • Sends fully rendered HTML to browser

Phase 2: Node Claiming (Client)

  • Angular bootstraps on the client
  • Instead of creating DOM nodes, it reads hydration markers
  • Claims existing server-rendered nodes
  • Tracks unclaimed nodes for cleanup
  • Verifies DOM structure matches expected template

Phase 3: Cleanup & Attach

  • Attaches event listeners to claimed nodes
  • Rehydrates TransferState into HTTP cache
  • Removes any unused server-only DOM nodes
  • Fires lifecycle hooks: AfterViewInit, etc.
  • App is now fully interactive
TransferState — Preventing Double HTTP Requests
// Server fetches data and serializes to HTML
// The data is embedded as JSON in a script tag:
<script id="ng-state" type="application/json">
  {"products": [{"id": 1, "name": "..."}]}
</script>

// On client, Angular's HttpClient reads from TransferState FIRST
// — no actual HTTP call is made if data is already there
@Injectable()
class ProductService {
  constructor(private http: HttpClient) {}

  getProducts(): Observable<Product[]> {
    // Angular's HttpClient + TransferState does this automatically
    return this.http.get<Product[]>('/api/products');
  }
}
v18+ Feature

Event Replay — withEventReplay()

Introduced in Angular v18 · Powered by JSAction

The server HTML is visible immediately, but Angular JS isn't ready yet. Users who click buttons during this window expect their actions to work. Event Replay captures those early interactions and replays them once Angular has fully bootstrapped.

Phase 1 — Capture

Before Angular boots, a lightweight inline script (JSAction) attaches to the document root and captures all native browser events: click, mouseover, focusin, keydown, etc.

Phase 2 — Store

The Event Contract keeps all captured interactions in memory. Nothing is lost. Events are queued in order, preserving sequence.

Phase 3 — Replay

Once Angular hydration completes and event listeners are attached, Angular re-dispatches all queued events in the correct order. Users' actions are honoured.

Enabling Event Replay — app.config.ts
import {
  provideClientHydration,
  withEventReplay
} from '@angular/platform-browser';

bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration(
      withEventReplay()  // ← v18+: Captures events before hydration
    )
  ]
});
Auto-enabled with Incremental Hydration

If you use incremental hydration (provideClientHydration() default in v19+), event replay is automatically enabled. You don't need to add withEventReplay() separately.

What events are supported

All native browser events are supported: click, mouseover, focusin, etc. Angular custom events (EventEmitter) triggered before hydration are not replayed, as they have no DOM counterpart.

v19+ Feature

Incremental Hydration — The Next Level

Angular v19+ · Default-on in v22 via provideClientHydration()

Incremental hydration is an advanced form of hydration that leaves sections of your app dehydrated and lazily triggers hydration of those sections as needed. It builds on top of full app hydration + deferrable views (@defer) + event replay.

The Problem It Solves

Full app hydration loads and hydrates everything at once. For large pages, this means downloading and executing JavaScript for components that may be:

  • Below the fold (never seen)
  • Rarely interacted with
  • Heavy third-party components

This wastes time, bandwidth, and CPU on mobile.

The Solution

Incremental hydration lets you mark sections with hydrate triggers on @defer blocks. Those sections:

  • Are rendered by the server (no placeholder flash)
  • Remain dehydrated on the client initially
  • Download their JS bundles on demand (viewport entry, interaction, hover, idle, timer)
  • Hydrate instantly when triggered

v19+

Incremental Hydration — Setup & Mechanics

Enabling (v19–v21)

// v19 – v21: explicit opt-in
import {
  provideClientHydration,
  withIncrementalHydration
} from '@angular/platform-browser';

bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration(
      withIncrementalHydration()
    )
  ]
});
// v22+: enabled BY DEFAULT with provideClientHydration()
provideClientHydration()  // incremental hydration is ON

// Opt OUT if you need to:
provideClientHydration(withNoIncrementalHydration())

How It Works Mechanically

  • SSR phase: Angular renders the @defer block's main template (not the placeholder) because it has a hydrate trigger
  • Client phase: Dependencies are not loaded. The block is "dehydrated" — rendered HTML is present but no JS attached
  • Trigger fires: Browser triggers the condition (scroll, hover, etc.)
  • Lazy load: Angular fetches the chunk for that block's dependencies
  • Hydrate: Angular hydrates that section of the DOM
  • Event replay: Any interactions before hydration are replayed
Important: Parent-First Hydration

Angular's component tree is hierarchical. Hydrating a child requires the parent to be hydrated first. If a user triggers a deeply nested @defer block, Angular walks up the tree and hydrates parent blocks first, in top-down order.

Incremental Hydration

Hydrate Triggers — Complete Reference

Trigger When it fires Best Use Case
hydrate on idle Browser enters idle state via requestIdleCallback. Optional timeout: hydrate on idle(500) Non-critical widgets (chat, analytics, footers) that should hydrate when browser isn't busy
hydrate on viewport Content enters the viewport (Intersection Observer API) Below-the-fold sections: product grids, comment sections, related articles
hydrate on interaction User clicks or presses a key on the element Modals, dropdowns, accordions — hydrate on first interaction
hydrate on hover mouseover or focusin on the element Tooltips, mega-menus — hydrate slightly before click
hydrate on immediate Immediately after all non-deferred content has rendered Important but slightly secondary content — hydrate ASAP after critical path
hydrate on timer(Xms) After a fixed duration (ms or s) Ads, banners — delay hydration by a set time
hydrate when condition When a custom boolean expression becomes true User-state-driven (logged in, feature flag enabled, route param set)
hydrate never Never hydrates on initial load — stays as static HTML forever Pure display components: static banners, marketing copy, server-only rendered blocks
Trigger Examples in Templates
<!-- Hydrate when user scrolls this into view -->
@defer (hydrate on viewport) {
  <app-product-grid />
} @placeholder { <div>Loading products...</div> }

<!-- Hydrate after 2 seconds -->
@defer (hydrate on timer(2s)) {
  <app-chat-widget />
} @placeholder { <div>Chat loading...</div> }

<!-- Hydrate when user is authenticated -->
@defer (hydrate when isLoggedIn()) {
  <app-user-dashboard />
} @placeholder { <div>Please log in</div> }

<!-- Static marketing block — never hydrates -->
@defer (on viewport; hydrate never) {
  <app-hero-banner />
} @placeholder { <div>...</div> }

@defer Triggers vs Hydrate Triggers

Two Separate Trigger Systems Working Together

A @defer block can have both a regular trigger (for client-side render navigation) and a hydrate trigger (for initial SSR load). They serve different rendering paths and don't conflict.

Dual Triggers Example
@defer (on idle; hydrate on interaction) {
  <app-comments-section />
} @placeholder {
  <div>Comments loading...</div>
}


┌─────────────────────────────────────────────────────────────────┐
│  INITIAL LOAD (SSR + Hydration path):                           │
│    "hydrate on interaction" applies                             │
│    Server renders the main template (no placeholder)           │
│    Client dehydrates the block, waits for user to click         │
│    On click → lazy load JS chunk → hydrate the section         │
├─────────────────────────────────────────────────────────────────┤
│  SUBSEQUENT CLIENT-SIDE NAVIGATION (CSR path):                 │
│    "on idle" applies                                            │
│    No SSR, so regular @defer logic kicks in                    │
│    Shows placeholder → loads chunk on browser idle             │
│    → renders main template                                      │
└─────────────────────────────────────────────────────────────────┘

Nested @defer Hydration Order

@defer (hydrate on interaction) {
  <parent-cmp />
  @defer (hydrate on hover) {
    <child-cmp />   ← hover here
  }
}

Hover on child triggers:
1. Parent hydrates first
2. Then child hydrates
(Always top-down)

@placeholder Still Required

Why placeholder is still needed

Even with incremental hydration, the @placeholder block is needed for client-side navigation paths. If the user navigates to a route fresh on the client (not SSR initial load), the @defer block behaves normally and shows the placeholder while loading.

Escape Hatch

ngSkipHydration — Opting Out Per Component

Use as a Last Resort Only

ngSkipHydration is an escape hatch for components that can't be made hydration-compatible (e.g., using heavy direct DOM manipulation or 3rd-party libraries). Using it means those components lose ALL hydration benefits and behave as if hydration isn't enabled.

Usage — Template Attribute

<!-- On the component's tag in parent template -->
<app-d3-chart ngSkipHydration />

<!-- Works with property binding syntax too -->
<app-d3-chart [ngSkipHydration]="true" />

Usage — Host Binding

@Component({
  selector: 'app-d3-chart',
  template: `<div #chartHost></div>`,
  host: { ngSkipHydration: 'true' }  // ← host binding
})
export class D3ChartComponent {
  @ViewChild('chartHost') chartEl!: ElementRef;

  ngAfterViewInit() {
    // D3 DOM manipulation — safe here because of ngSkipHydration
    d3.select(this.chartEl.nativeElement).append('svg')...;
  }
}

What ngSkipHydration Does

  • Angular destroys the server-rendered DOM for that component
  • Re-renders the component from scratch on the client
  • All children are also skipped — it's an entire subtree skip
  • The component will cause a layout shift (content wipes and redraws)
  • Data is fetched again (no TransferState benefit for that component)
Critical Warning

Adding ngSkipHydration to your root AppComponent effectively disables hydration for the entire application. Always apply it as narrowly as possible to the specific component that needs it.

When to use it
  • 3rd-party D3 / Chart.js / Leaflet components
  • Legacy Angular components that do direct DOM manipulation
  • Components using innerHTML / outerHTML heavily
  • While you're iterating on fixing the root cause
Constraints

Hydration Constraints — What You Must Know

The Core Rule

The HTML produced by server-side rendering must NOT be altered between the server and when Angular runs on the client. The DOM tree must have the same structure — including whitespace and comment nodes — in both environments.

❌ Direct DOM Manipulation

  • Using document.querySelector
  • Using appendChild / removeChild
  • Using innerHTML / outerHTML
  • Detaching and moving DOM nodes
  • Using ElementRef.nativeElement to modify structure

Fix: Use Angular APIs (@if, @for, Renderer2, structural directives)

⚠️ Invalid HTML Structure

  • <table> without explicit <tbody> — browser auto-inserts <tbody>, Angular expects it too
  • <div> directly inside <p> — browsers auto-close the <p>
  • Nested <a> tags — invalid, causes parse differences
  • <li> outside of <ul>/<ol>

Fix: Validate templates at validator.w3.org

⚠️ Other Constraints

  • preserveWhitespaces: true in tsconfig — risky; set consistently in both client and server tsconfigs
  • Custom/Noop Zone.js — may cause hydration to trigger too early/late. Not fully supported
  • Conditional renders based on isPlatformBrowser — causes client/server DOM mismatch
  • Third-party scripts that modify DOM before hydration (ads, analytics)

❌ Bad — Platform-conditional render

@if (isPlatformBrowser(platformId)) {
  <app-chart />
}
// Server renders nothing, client renders chart
// ← DOM MISMATCH ERROR

✅ Good — AfterNextRender for browser-only

// Use AfterNextRender to run browser-only code
// AFTER hydration, not during rendering
afterNextRender(() => {
  // safely run browser-only code here
});
Errors

Common Hydration Errors & Fixes

NG0500: Hydration Node Mismatch
Cause: Angular expected a specific DOM node (e.g. a <div>) but found something different (e.g. a <p>) because the DOM was mutated between server and client.
Fix: Check for direct DOM manipulation, invalid HTML, or platform-conditional rendering. Use ngSkipHydration as a temporary fix.
NG0501: Hydration Missing Siblings
Cause: Angular expected sibling nodes after a component but found none. Usually caused by DOM mutation that removed elements.
Fix: Stop removing nodes before hydration. Use @if/@for instead of manually removing DOM.
NG0502: Hydration Missing Node
Cause: A DOM node present on the server is missing on the client — possibly stripped by a CDN HTML minifier or a 3rd party script.
Fix: Ensure your CDN/proxy doesn't modify Angular's HTML. Check for aggressive HTML minification that removes comment nodes.
NG0506: Application Remains Unstable
Cause: App hasn't reached a stable state after 10 seconds. Often caused by unresolved timers (setInterval), pending promises, infinite observables, or uncleared macrotasks in Zone.js.
Fix: Use provideStabilityDebugging() to identify what's keeping the app unstable.
NG0507: Skip Hydration on Invalid Host
Cause: ngSkipHydration was applied to a non-component node (e.g. an <ng-container> or a plain DOM element).
Fix: ngSkipHydration can only be placed on a component's host element, not on directives or plain DOM.
Using provideStabilityDebugging()
import { provideStabilityDebugging } from '@angular/core';
import 'zone.js/plugins/task-tracking';

bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration(),
    provideStabilityDebugging()  // dev only!
  ]
});

// Console will log pending tasks:
// [Zone.js] Macro task: setInterval (src/app.ts:42)
// This tells you EXACTLY what is preventing stability
Remove from production

provideStabilityDebugging() is NOT tree-shaken. Do not leave it in production builds — it adds overhead and exposes internals.

Stability

Application Stability & Hydration Timing

Critical Concept

Hydration and all post-hydration processes only start once the application reports stability. Zone.js tracks pending async work (HTTP, timers, promises). When the Zone becomes idle, Angular considers the app "stable" and begins or completes hydration processes.

What Prevents Stability

  • setInterval / setInterval running indefinitely
  • Unresolved Promises that never resolve or reject
  • Pending microtasks in the Zone.js queue
  • Infinite RxJS observables not properly unsubscribed
  • Router guards that never complete
  • Pending HTTP requests that timeout silently
Common Mistake — Uncleared Interval
export class AppComponent implements OnInit {
  ngOnInit() {
    // ❌ This prevents app stability FOREVER
    setInterval(() => {}, 1000);
  }
}

// ✅ Fix: Run outside Zone.js
constructor(private ngZone: NgZone) {}
ngOnInit() {
  this.ngZone.runOutsideAngular(() => {
    setInterval(() => {}, 1000);
  });
}

The 10-Second Rule

NG0506 Threshold

If your app hasn't stabilized within 10 seconds of bootstrapping, Angular throws NG0506: Application remains unstable error. This is a hard timeout meant to surface bugs early.

Stability Debugging Workflow

1 Add provideStabilityDebugging() to providers
2 Add import 'zone.js/plugins/task-tracking'
3 Check console — see pending task names + stack traces
4 Fix each source: clear timers, complete observables
5 Remove debug provider before deploying
i18n & Libraries

i18n Support & Third-Party Libraries

i18n — International Support

Default Behaviour

By default, Angular skips hydration for components that use i18n blocks — they re-render from scratch on the client. This avoids complex locale-based DOM mismatches.

Enabling i18n Hydration Support
import {
  provideClientHydration,
  withI18nSupport
} from '@angular/platform-browser';

bootstrapApplication(AppComponent, {
  providers: [
    provideClientHydration(
      withI18nSupport()  // ← Opt-in for i18n hydration
    )
  ]
});
Consistency is key

Ensure your server and client use the same locale. Locale mismatch = DOM mismatch = hydration error.

Third-Party Libraries

Libraries with DOM Manipulation

Many libraries (D3.js, Chart.js, Leaflet, ag-Grid) rely on direct DOM manipulation. This causes hydration mismatches.


Solution: Apply ngSkipHydration to the wrapper component until the library is updated or you can isolate its DOM work.

Third-Party Scripts (Analytics, Ads)

Scripts like Google Analytics, ad trackers, or A/B testing scripts modify DOM before Angular hydrates. This causes hydration errors.


Solution: Use afterNextRender() to defer script injection until after hydration completes.

Defer 3rd Party Script Until Post-Hydration
import { afterNextRender, Component } from '@angular/core';

@Component({...})
export class AppComponent {
  constructor() {
    afterNextRender(() => {
      // Load analytics ONLY after hydration
      loadGoogleAnalytics();
    });
  }
}
Best Practices

Hydration Best Practices for Production

✅ Always Use Angular APIs for DOM

  • Use @if, @for, @switch for conditional/loop rendering
  • Use Renderer2 for programmatic DOM changes
  • Use @ViewChild + ElementRef for reading, not for structural changes
  • Use afterNextRender() or afterRender() for browser-only DOM work

✅ Consistent Server + Client Rendering

  • Avoid isPlatformBrowser() in templates to conditionally render different content
  • Use the same locale for i18n on server and client
  • Use the same preserveWhitespaces setting in both tsconfig.app.json and tsconfig.server.json
  • Avoid random data (dates, UUIDs) in templates without seeding

✅ Use TransferState for Data

  • Let Angular's HttpClient + TransferState handle data transfer automatically
  • Services using HttpClient get transfer state for free in Angular 15+
  • Avoid re-fetching data inside ngOnInit if the data was already loaded on the server
  • Use makeStateKey for custom non-HTTP data

✅ Incremental Hydration for Performance

  • Use hydrate on viewport for below-the-fold heavy components
  • Use hydrate on idle for non-critical secondary widgets
  • Use hydrate never for purely static server-rendered blocks
  • Measure bundle impact with Angular CLI --stats-json

✅ Validate HTML Templates

  • Always declare <tbody> in tables explicitly
  • Run templates through validator.w3.org
  • Avoid nesting block-level elements inside inline elements
  • Never nest <a> inside <a>

✅ Maintain App Stability

  • Run timers/intervals outside Zone.js with ngZone.runOutsideAngular()
  • Complete all RxJS subscriptions properly
  • Use takeUntilDestroyed() or DestroyRef for cleanup
  • Test stability with provideStabilityDebugging() before release
The Golden Rule

Your application's server-rendered HTML is a contract. Angular hydration trusts that contract completely. Any deviation — whether from DOM manipulation, scripts, CDN transforms, or conditional rendering — breaks that contract and causes errors. Treat the server-rendered DOM as immutable until Angular has fully bootstrapped.

Interview Q&A

Interview Questions — Part 1: Fundamentals

Q1. What is Angular Hydration and why was it introduced? Senior
Hydration is the process of restoring an SSR-rendered Angular application on the client by reusing existing server-rendered DOM nodes instead of destroying and recreating them. It was introduced because without hydration, SSR apps suffered from a double-render problem: the server rendered HTML for a fast first paint, but Angular's client bootstrap destroyed that DOM and re-rendered it from scratch, causing a visible flicker, layout shift, and a second round of data fetching. Hydration eliminates this waste by making Angular "claim" existing nodes, dramatically improving LCP, FID, and CLS Core Web Vitals.
Q2. How does Angular hydration differ from React's hydration? Expert
Both React and Angular's hydration walk the server-rendered DOM to attach client-side logic without rebuilding the DOM. The key difference is Angular's Incremental Hydration (v19+) which uses @defer block annotations to leave portions of the tree dehydrated and load them lazily on demand — React achieves something similar with Selective Hydration / Suspense. Angular's implementation is unique because the hydration boundaries are defined via template syntax (hydrate on viewport, hydrate on interaction) rather than code boundaries, making it more declarative. Angular also automatically handles TransferState for HTTP cache transfer, while React requires manual implementation.
Q3. What is a DOM mismatch error in Angular Hydration and what are the top causes? Senior
A DOM mismatch (NG0500) occurs when Angular's expected component DOM tree doesn't match the actual DOM it finds in the browser. Top causes: (1) Direct DOM manipulation via native APIs before hydration (appendChild, innerHTML); (2) Invalid HTML structure where browsers auto-correct (e.g., missing <tbody>); (3) Platform-conditional rendering using isPlatformBrowser() causing server/client to render different templates; (4) Third-party scripts (ads, analytics) that modify DOM before Angular hydrates; (5) CDN HTML minifiers that strip Angular's hydration comment markers.
Q4. What is the minimum setup required to enable hydration in a new Angular project? Mid-Senior
SSR must be enabled first (ng add @angular/ssr). Then add provideClientHydration() to the app's bootstrap providers (in app.config.ts or AppModule). The CLI does both steps automatically when you use ng new my-app --ssr. Crucially, provideClientHydration() must be present in both the client bootstrap config AND the server bootstrap config — missing it from the server config is a common mistake.
Interview Q&A

Interview Questions — Part 2: Advanced

Q5. Explain Event Replay. Why is it needed and how does it work? Senior
Event Replay solves the "zombie click" problem: the server HTML is visible and looks interactive, but Angular's JS hasn't bootstrapped yet. Without event replay, any user clicks during this window are silently discarded. With withEventReplay() (v18+), a lightweight JSAction library attaches to the document root via an inline script (runs before Angular) and captures all native browser events (click, keydown, focusin, etc.) into a queue. Once Angular finishes hydration and attaches real event listeners, it replays those queued events in order. This ensures not a single user interaction is lost during the bootstrap window.
Q6. What is Incremental Hydration and how does it integrate with @defer? Expert
Incremental Hydration extends @defer blocks with additional hydrate triggers. During SSR, if a @defer block has a hydrate trigger, Angular renders its main template (not the placeholder), so the server HTML is complete. On the client, the block's JS dependencies are not loaded initially — the block remains "dehydrated" (HTML present, no JS attached). When the specified trigger fires (e.g., hydrate on viewport), Angular lazily downloads that block's chunk, then hydrates that section. This produces smaller initial JS bundles while avoiding layout shifts — a combination previously impossible because using @defer above-the-fold always showed a placeholder flash.
Q7. When would you use ngSkipHydration and what are its trade-offs? Senior
ngSkipHydration is used when a component fundamentally can't be made hydration-compatible — typically wrapper components around 3rd-party libraries (D3, Leaflet, ag-Grid) that imperatively build their own DOM structures. Trade-offs: the component and its entire child subtree lose all hydration benefits — Angular destroys the server-rendered DOM for that subtree and re-renders from scratch, causing a layout shift and potentially a second data fetch. It should be a temporary workaround, not a permanent solution. The root component must never have ngSkipHydration as it disables hydration for the entire app.
Q8. What is Application Stability in the context of hydration and how do you debug it? Expert
Angular's hydration and post-hydration cleanup only execute after Zone.js reports the application as "stable" — meaning no pending macrotasks, microtasks, or promises. Common stability blockers: setInterval running without being cleared or moved outside the Zone, pending HTTP requests, infinite RxJS streams not unsubscribed. If unstable for 10 seconds, Angular throws NG0506. Debug using provideStabilityDebugging() + import 'zone.js/plugins/task-tracking' which logs each pending task with its creation stack trace. Fix by: moving timers outside Zone with ngZone.runOutsideAngular(), using takeUntilDestroyed() for subscriptions, and completing all async operations properly.
Interview Q&A

Interview Questions — Part 3: Expert Level

Q9. How does TransferState work with hydration to prevent double data fetching? Expert
During SSR, Angular's HttpClient intercepts HTTP responses and stores them in a TransferState store keyed by URL. This store is serialized as a JSON blob in a <script id="ng-state"> tag embedded in the server HTML. On the client, before making any HTTP request, Angular's built-in TransferStateInterceptor checks this store first. If the data is present, it returns it synchronously as a completed observable — no actual HTTP request is made. This means the app displays data instantly on client bootstrap without waiting for another round-trip. The state is cleared from memory after the first read. For custom non-HTTP data, use makeStateKey<T>() + TransferState service directly.
Q10. How do hydrate triggers differ between an initial SSR load and subsequent client-side navigation? Expert
A @defer block can have both a regular trigger and a hydrate trigger: @defer (on idle; hydrate on interaction). These apply to different rendering scenarios. The hydrate on interaction trigger only applies to the initial SSR page load — it controls when the already-server-rendered dehydrated block gets hydrated. The on idle trigger applies to subsequent client-side navigation (e.g., via RouterLink) where the page is rendered freshly on the client with no SSR. In that case, the regular @defer behavior applies: show placeholder → load on idle → render main content. This dual-trigger design lets you optimize both the initial load experience and the SPA navigation experience independently.
Q11. What are the implications of preserveWhitespaces on hydration and how do you safely configure it? Senior
preserveWhitespaces: true in tsconfig makes Angular preserve whitespace text nodes between HTML elements. The problem: if this setting is inconsistent between server and client tsconfigs (tsconfig.server.json vs tsconfig.app.json), the server renders a different DOM structure (with extra text nodes) than the client expects. Angular hydration then sees DOM mismatches everywhere. The safe approach: set it only in tsconfig.app.json (which tsconfig.server.json inherits from), use the default value of false, and never override it in tsconfig.server.json independently.
Q12. How would you architect a large e-commerce page using incremental hydration for maximum performance? Expert
Above the fold (critical path): Hero section, navigation — no @defer, hydrates as part of full app hydration immediately. Primary product grid (below fold): @defer (on viewport; hydrate on viewport) — hydrates when user scrolls to it. Product detail tooltips: @defer (hydrate on hover) — loads only when user hovers, very cheap. Reviews section: @defer (hydrate on idle) — loads in background when browser is free. Ad/recommendation widgets: @defer (hydrate on timer(3s)) — delay to not compete with critical hydration. Footer/SEO content: @defer (hydrate never) — purely static, never needs to hydrate. This architecture produces a tiny initial JS bundle covering only above-the-fold code, with progressive loading of everything else.
Interview Q&A

Interview Questions — Part 4: Scenario-Based

Q13. A team reports hydration errors after adding a D3 chart component. Walk through your debugging and resolution approach. Expert
Step 1 — Confirm the error: Open Angular DevTools, enable the hydration overlay. The D3 component will be highlighted as the error source. Step 2 — Quick fix: Add ngSkipHydration to the D3 component's host binding to unblock the team while a proper fix is developed. Step 3 — Proper fix options: (a) Wrap the D3 logic in afterNextRender() so it runs after hydration, keeping the server-rendered SVG intact; (b) If D3 must control the SVG from scratch, use @defer (on immediate; hydrate never) to make it truly server-static — D3 takes over as static display, interaction added client-side; (c) Refactor to use Angular-native SVG bindings with [attr.d] etc. so Angular, not D3, controls the DOM structure. Step 4 — Remove ngSkipHydration once the fix is validated.
Q14. Your SSR Angular app has great LCP but users report that clicks in the first 2 seconds do nothing. What is the issue and how do you fix it? Senior
This is the classic hydration gap problem. The server HTML is visible (fast LCP), but Angular's JS bootstrap takes ~1-2 seconds. During this window, the DOM has no event listeners attached, so clicks are silently lost. The fix is withEventReplay() (Angular v18+): provideClientHydration(withEventReplay()). This injects a lightweight JSAction script that runs before Angular, captures all user events into a queue, and replays them once hydration completes. In v19+ with incremental hydration enabled, event replay is already on automatically. Also worth checking if the JS bundle can be split further to reduce bootstrap time itself.
Q15. Explain hydrate never — what is it really good for and what are its limitations? Senior
hydrate never marks a @defer block to remain permanently dehydrated — it renders its server HTML and stays as static DOM, never downloading its JS or attaching any event listeners. It's ideal for purely presentational, non-interactive blocks: marketing copy, static banners, footer links, SEO metadata sections. The key limitation: it only applies to the initial SSR load. If the user navigates to this route via client-side routing (a RouterLink click), Angular renders the page fresh on the client without SSR — in that case, the regular @defer triggers apply normally (on viewport, etc.). Also, hydrate never on a block prevents any nested @defer blocks inside it from ever hydrating on the initial load too — it's a subtree-wide freeze.

Complete Example — E-Commerce Page with Incremental Hydration

app.config.ts
import {
  provideClientHydration   // v22: incremental + event replay ON by default
} from '@angular/platform-browser';
import { provideHttpClient, withFetch } from '@angular/common/http';
import { provideRouter } from '@angular/router';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(withFetch()),  // fetch works well with SSR
    provideClientHydration(),         // full + incremental + event replay
  ]
};
product-page.component.ts
@Component({
  template: `
    <!-- ✅ Critical: no @defer, hydrates immediately -->
    <app-hero-banner />
    <app-nav />
    <app-product-detail />

    <!-- ✅ Viewport: hydrates when user scrolls down -->
    @defer(hydrate on viewport) {
      <app-similar-products />
    } @placeholder { <div class="skeleton"></div> }

    <!-- ✅ Idle: low priority, hydrates when browser free -->
    @defer(hydrate on idle) {
      <app-customer-reviews />
    } @placeholder { <div class="skeleton"></div> }

    <!-- ✅ Never: purely static footer -->
    @defer(on viewport; hydrate never) {
      <app-footer />
    } @placeholder { <div></div> }
  `
})
export class ProductPageComponent {}
Result of This Architecture

Initial JS bundle: Contains only hero, nav, and product detail code. On scroll: Similar products chunk loads lazily. While idle: Reviews chunk loads silently. Footer: Never loads JS — pure static HTML forever. User clicks: Never lost thanks to event replay. This is the ideal production setup for Angular v22+.
