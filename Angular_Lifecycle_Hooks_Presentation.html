01

What Are Lifecycle Hooks?

Definition

Lifecycle hooks are interfaces Angular provides that allow you to tap into key moments of a component or directive's life — from creation, through every change detection cycle, to final destruction.

How to Use

  • Implement the interface: implements OnInit
  • Define the method: ngOnInit()
  • Interfaces are optional at runtime (TypeScript only) but highly recommended for type safety
  • Multiple hooks can be implemented on one class

Works On

  • Components — all 8 hooks available
  • Directives — all 8 hooks available
  • Pipes — only ngOnDestroy
  • Services — only ngOnDestroy (when provided in component)

Key Principle

Never do heavy work in the constructor — use ngOnInit for initialization. The constructor should only wire up dependency injection. Angular calls hooks after the constructor completes.

import { Component, OnInit, OnDestroy } from '@angular/core';

@Component({ selector: 'app-demo', template: '...' })
export class DemoComponent implements OnInit, OnDestroy {

  constructor(private service: DataService) {}  // ✅ DI only

  ngOnInit(): void {
    this.service.load();   // ✅ initialization logic here
  }

  ngOnDestroy(): void {
    // ✅ cleanup — unsubscribe, clear timers
  }
}

03
ngOnChanges implements OnChanges Multiple times

When It Runs

  • Before ngOnInit (on first run)
  • Every time a bound @Input() property changes
  • Only for primitive changes or reference changes — not for mutated objects/arrays
  • Not called if component has NO @Input() properties

Real-World Use Cases

  • React to parent passing new data (e.g., reload chart on new config)
  • Transform incoming @Input before use
  • Detect first load vs subsequent change using firstChange
  • Log or audit input changes
  • Avoid: do NOT trigger heavy API calls here on every change — debounce instead
import { Component, Input, OnChanges, SimpleChanges } from '@angular/core';

@Component({ selector: 'app-chart', template: '...' })
export class ChartComponent implements OnChanges {
  @Input() config: ChartConfig;
  @Input() theme: string;

  ngOnChanges(changes: SimpleChanges): void {
    // SimpleChanges: { [inputName]: SimpleChange }
    if (changes['config']) {
      const c = changes['config'];
      console.log('Previous:', c.previousValue);
      console.log('Current:',  c.currentValue);
      console.log('First change?', c.firstChange);  // true on ngOnInit

      if (!c.firstChange) {
        this.reloadChart();  // skip on initial load
      }
    }

    if (changes['theme']?.currentValue) {
      this.applyTheme(changes['theme'].currentValue);
    }
  }

  // ⚠️ GOTCHA: mutating an object won't trigger ngOnChanges
  // Parent must do: this.config = { ...this.config, key: val }  (new reference)
}
04
ngOnInit implements OnInit Once only

When It Runs

  • Called once, after the first ngOnChanges
  • All @Input() values are available
  • Child views are NOT yet initialized
  • Use for all initialization logic

Constructor vs ngOnInit

  • Constructor: Dependency injection only, no @Input values, no DOM
  • ngOnInit: @Input values ready, HTTP calls, subscriptions, service logic
  • Angular can call ngOnInit without calling the constructor again (e.g., test beds)
  • Always prefer ngOnInit for initialization — it is testable and predictable
@Component({ selector: 'app-users', template: '...' })
export class UsersComponent implements OnInit {
  @Input() roleFilter: string;
  users$: Observable<User[]>;

  constructor(private userService: UserService) {
    // ❌ Don't do this — @Input roleFilter not set yet
    // this.users$ = this.userService.getByRole(this.roleFilter);
  }

  ngOnInit(): void {
    // ✅ @Input() is ready here
    this.users$ = this.userService.getByRole(this.roleFilter);

    // ✅ Setup subscriptions, start polling, initialize form, etc.
    this.userService.trackActivity().pipe(
      takeUntilDestroyed(this.destroyRef)  // Angular 16+ pattern
    ).subscribe();
  }
}

ngDoCheck implements DoCheck Every CD cycle ⚠️

When It Runs

  • Called on every change detection cycle — even if nothing changed
  • Runs immediately after ngOnChanges and ngOnInit
  • Use to detect changes Angular cannot detect (mutated objects, arrays)
  • ⚠️ Never put heavy logic here — it can destroy performance

Real Use Cases

  • Detect deep mutations in arrays or objects (Angular misses these)
  • Custom dirty-checking with KeyValueDiffer or IterableDiffer
  • Rarely needed — prefer immutable data patterns instead
  • Used internally by Angular's NgClass, NgStyle directives
import { DoCheck, KeyValueDiffer, KeyValueDiffers } from '@angular/core';

@Component({ selector: 'app-tracker', template: '...' })
export class TrackerComponent implements DoCheck {
  @Input() user: { name: string; age: number };  // mutable object
  private differ: KeyValueDiffer<string, any>;

  constructor(private differs: KeyValueDiffers) {}

  ngOnInit() {
    this.differ = this.differs.find(this.user).create();
  }

  ngDoCheck(): void {
    const changes = this.differ.diff(this.user);
    if (changes) {
      changes.forEachChangedItem(item =>
        console.log('Changed:', item.key, item.previousValue, '→', item.currentValue)
      );
    }
  }

  // ✅ Better approach: use immutable updates + ngOnChanges instead
  // this.user = { ...this.user, age: 31 }  ← triggers ngOnChanges
}
06
ngAfterContentInit implements AfterContentInit Once only

When It Runs

  • Called once after Angular projects external content via <ng-content>
  • @ContentChild / @ContentChildren are populated here
  • Child component's ngOnInit runs BEFORE parent's ngAfterContentInit

Content Projection vs View

  • Content = what parent passes inside the component's tag (<ng-content>)
  • View = what the component defines in its own template
  • Use @ContentChild to query projected content
import { AfterContentInit, ContentChild, ContentChildren, QueryList } from '@angular/core';

// Parent template:
// <app-tab-group>
//   <app-tab label="Home">...</app-tab>    ← projected content
// </app-tab-group>

@Component({ selector: 'app-tab-group', template: '<ng-content></ng-content>' })
export class TabGroupComponent implements AfterContentInit {
  @ContentChildren(TabComponent) tabs: QueryList<TabComponent>;

  ngAfterContentInit(): void {
    // ✅ tabs QueryList is populated here
    console.log('Tabs count:', this.tabs.length);

    // Activate first tab by default
    const firstTab = this.tabs.first();
    if (firstTab) firstTab.active = true;

    // Listen for dynamic tab additions
    this.tabs.changes.subscribe(() => this.updateTabs());
  }
}

ngAfterContentChecked implements AfterContentChecked Every CD cycle ⚠️

When It Runs

  • Called after every change detection that checks projected content
  • Runs after ngAfterContentInit on first run
  • Be careful — this is called very frequently
  • Avoid triggering change detection here (ExpressionChangedAfterItHasBeenChecked risk)

Practical Use

  • Re-evaluate state after projected content changes (e.g., count visible tabs)
  • Sync internal state based on what was projected
  • Rarely needed — use QueryList.changes observable instead when possible
import { AfterContentChecked, ContentChildren, QueryList } from '@angular/core';

@Component({ selector: 'app-tab-group', template: '<ng-content></ng-content>' })
export class TabGroupComponent implements AfterContentChecked {
  @ContentChildren(TabComponent) tabs: QueryList<TabComponent>;
  activeTabCount = 0;

  ngAfterContentChecked(): void {
    const newCount = this.tabs.filter(t => t.active).length;

    // ✅ Only act if value actually changed — avoid infinite loops
    if (newCount !== this.activeTabCount) {
      this.activeTabCount = newCount;
    }
  }

  // ❌ Anti-pattern: updating a bound property here triggers ExpressionChangedError
  // ngAfterContentChecked() { this.title = 'new'; }  ← causes error in dev mode
}
08
ngAfterViewInit implements AfterViewInit Once only

When It Runs

  • Called once after the component's own view and all child views are initialized
  • @ViewChild and @ViewChildren are populated here
  • Safe to interact with the DOM or child components here
  • Do NOT update component properties bound in the template here (in dev mode) — use setTimeout or ChangeDetectorRef.detectChanges()

Real-World Use Cases

  • Initialize third-party libraries (D3.js, Chart.js, AG Grid)
  • Focus an input element programmatically
  • Read native element dimensions for animations
  • Interact with a child component's public methods
import { AfterViewInit, ViewChild, ElementRef } from '@angular/core';
import * as Chart from 'chart.js';

@Component({
  selector: 'app-dashboard',
  template: `<canvas #myChart></canvas>`
})
export class DashboardComponent implements AfterViewInit {
  @ViewChild('myChart') chartRef: ElementRef<HTMLCanvasElement>;
  @ViewChild(ChildGridComponent) grid: ChildGridComponent;

  ngAfterViewInit(): void {
    // ✅ DOM element available — initialize Chart.js
    new Chart(this.chartRef.nativeElement, {
      type: 'bar',
      data: this.chartData
    });

    // ✅ Access child component's public API
    this.grid.selectFirstRow();

    // ✅ Use ChangeDetectorRef if you must update a template-bound property
    this.cdr.detectChanges();
  }
}

ngAfterViewChecked implements AfterViewChecked Every CD cycle ⚠️

When It Runs

  • Called after every change detection that checks the component's view
  • Runs after ngAfterViewInit on first run
  • Most frequently called hook — runs after every user interaction, HTTP response, timer, etc.
  • Almost any async operation will trigger this

⚠️ Common Pitfalls

  • Updating template-bound properties here throws ExpressionChangedAfterItHasBeenChecked in dev mode
  • Avoid calling detectChanges() here — causes infinite loop
  • Keep logic absolutely minimal (flag checks only)
  • Use setTimeout() or Promise.resolve() to defer updates safely
import { AfterViewChecked, ChangeDetectorRef } from '@angular/core';

@Component({ selector: 'app-scroll', template: '...' })
export class ScrollComponent implements AfterViewChecked {
  private shouldScrollBottom = false;

  addMessage(msg: string) {
    this.messages.push(msg);
    this.shouldScrollBottom = true;  // set flag — don't scroll here
  }

  ngAfterViewChecked(): void {
    // ✅ Only act when flag is set — prevents running every cycle
    if (this.shouldScrollBottom) {
      this.scrollToBottom();
      this.shouldScrollBottom = false;
    }
  }

  private scrollToBottom() {
    this.scrollContainer.nativeElement.scrollTop =
      this.scrollContainer.nativeElement.scrollHeight;
  }
}

ngOnDestroy implements OnDestroy Once only

When It Runs

  • Called once just before Angular destroys the component
  • Triggered by: route navigation away, *ngIf becoming false, component removed dynamically
  • Last chance to clean up resources
  • Not called if the browser tab is closed — use window.beforeunload for that

What to Clean Up

  • RxJS subscriptions (prevent memory leaks)
  • setInterval / setTimeout handles
  • Third-party library instances (charts, maps)
  • Event listeners added manually
  • WebSocket connections
import { OnDestroy, DestroyRef, inject } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { Subject } from 'rxjs';

// ── Pattern 1: Classic Subject destroy (Angular < 16) ──
export class OldComponent implements OnDestroy {
  private destroy$ = new Subject<void>();

  ngOnInit() {
    this.data$.pipe(takeUntil(this.destroy$)).subscribe(...);
  }

  ngOnDestroy() {
    this.destroy$.next();
    this.destroy$.complete();
    this.chartInstance?.destroy();  // cleanup Chart.js
    clearInterval(this.pollInterval);
  }
}

// ── Pattern 2: Modern takeUntilDestroyed (Angular 16+) ──
export class ModernComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    this.data$.pipe(
      takeUntilDestroyed(this.destroyRef)  // auto-unsubscribes on destroy
    ).subscribe(...);
  }
  // No ngOnDestroy needed for subscriptions! 🎉
}

Parent–Child Lifecycle Order

Understanding the interleaved order is critical for senior-level interviews.

// Execution order when Parent contains Child:

1.  Parent constructor
2.  Parent ngOnChanges
3.  Parent ngOnInit
4.  Parent ngDoCheck
5.  Parent ngAfterContentInit
6.  Parent ngAfterContentChecked
7.    Child constructor
8.    Child ngOnChanges
9.    Child ngOnInit
10.   Child ngDoCheck
11.   Child ngAfterContentInit
12.   Child ngAfterContentChecked
13.   Child ngAfterViewInit
14.   Child ngAfterViewChecked
15. Parent ngAfterViewInit     ← after ALL children init
16. Parent ngAfterViewChecked

Key Interview Insight

Parent's ngAfterViewInit fires after all child views are initialized. This is why you can safely access @ViewChild references in ngAfterViewInit.

On Destroy Order

Destroy order is Child first, then Parent:

Child ngOnDestroy → Parent ngOnDestroy

Change Detection Re-run

After init, on every CD cycle: Parent DoCheck → Parent AfterContentChecked → Child DoCheck → Child AfterContentChecked → Child AfterViewChecked → Parent AfterViewChecked

Complete Component — All Hooks

import {
  Component, Input, OnChanges, OnInit, DoCheck,
  AfterContentInit, AfterContentChecked,
  AfterViewInit, AfterViewChecked, OnDestroy,
  SimpleChanges, ChangeDetectorRef, inject, DestroyRef
} from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ selector: 'app-complete', template: `
  <ng-content></ng-content>
  <child-comp #child></child-comp>
` })
export class CompleteComponent implements
  OnChanges, OnInit, DoCheck,
  AfterContentInit, AfterContentChecked,
  AfterViewInit, AfterViewChecked, OnDestroy {

  @Input() title: string;
  private destroyRef = inject(DestroyRef);

  constructor(private cdr: ChangeDetectorRef) { console.log('1. constructor'); }

  ngOnChanges(c: SimpleChanges) { console.log('2. ngOnChanges', c); }
  ngOnInit()            { console.log('3. ngOnInit'); }
  ngDoCheck()           { console.log('4. ngDoCheck'); }
  ngAfterContentInit()  { console.log('5. ngAfterContentInit'); }
  ngAfterContentChecked() { console.log('6. ngAfterContentChecked'); }
  ngAfterViewInit()     { console.log('7. ngAfterViewInit'); }
  ngAfterViewChecked()  { console.log('8. ngAfterViewChecked'); }
  ngOnDestroy()         { console.log('9. ngOnDestroy'); }
}

Things Commonly Missed 🎯

1. ngOnChanges not called without @Input

If a component has no @Input() properties, ngOnChanges is never called. Many devs expect it to fire regardless.

2. Object mutation ≠ ngOnChanges

Mutating an object (this.obj.name = 'x') does NOT trigger ngOnChanges. Must assign a new reference: this.obj = {...this.obj}.

3. ExpressionChangedAfterChecked

Updating a template-bound variable in ngAfterViewInit or ngAfterViewChecked throws this error in dev mode. Use cdr.detectChanges() or setTimeout().

4. DestroyRef / takeUntilDestroyed

Angular 16+ introduced takeUntilDestroyed(destroyRef) — eliminates need for manual Subject-based cleanup. Works in constructors too (no lifecycle hook needed).

5. Hooks work on Directives too

All 8 hooks work on directives, not just components. A structural directive like a custom *appLazy can use ngOnInit and ngOnDestroy.

6. OnPush + ngDoCheck

With ChangeDetectionStrategy.OnPush, ngDoCheck still runs every cycle but the view is only updated when Angular decides to re-render. ngDoCheck ≠ view update.

7. ngAfterViewInit in SSR

In Angular Universal (SSR), ngAfterViewInit runs on the server where DOM APIs don't exist. Guard DOM access with isPlatformBrowser().

8. Services with ngOnDestroy

Services can implement ngOnDestroy when provided at component level (providers: [...]). Angular calls it when the component is destroyed. Use for cleanup in component-scoped services.

9. Signals & Lifecycle (Angular 17+)

With Angular Signals, effect() replaces many ngOnChanges use-cases. Effects auto-track signal reads and re-run reactively — no need for SimpleChanges boilerplate.

Interview Q&A — Fundamentals

Q1. What is the difference between constructor and ngOnInit?
Constructor is a TypeScript feature used purely for dependency injection — Angular has not yet set @Input() values or initialized child views. ngOnInit is an Angular lifecycle hook called after all @Input() values are set; use it for all initialization logic, HTTP calls, and subscriptions.
Q2. In what order do lifecycle hooks execute?
constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → ngOnDestroy. The "Checked" hooks repeat on every change detection cycle.
Q3. When does ngOnChanges NOT fire?
When (1) the component has no @Input() properties, (2) the parent mutates an object/array instead of replacing the reference — Angular only compares references for @Input(), and (3) the same reference is re-assigned with identical content.
Q4. What is SimpleChanges and what does firstChange mean?
SimpleChanges is an object keyed by @Input property name, each value being a SimpleChange with previousValue, currentValue, and firstChange. firstChange is true only on the first call (same run as ngOnInit), allowing you to distinguish initial load from subsequent updates.
Q5. What is the difference between @ViewChild and @ContentChild?
@ViewChild queries elements or components defined in the component's own template. Available in ngAfterViewInit. @ContentChild queries elements projected via <ng-content> from the parent. Available in ngAfterContentInit.
16

Interview Q&A — Intermediate

Q6. How do you prevent memory leaks in Angular?
Use takeUntilDestroyed(destroyRef) (Angular 16+) or a Subject-based pattern with takeUntil(destroy$) in ngOnDestroy. Also clean up: setInterval/setTimeout handles, third-party library instances (charts, maps), and manually added event listeners. For component-scoped services, implement ngOnDestroy on the service.
Q7. Why do you get ExpressionChangedAfterItHasBeenCheckedError?
Angular performs a second check pass in dev mode to verify nothing changed after the first pass. If you update a template-bound property in ngAfterViewInit or ngAfterViewChecked, the values differ between the two passes. Fix: use ChangeDetectorRef.detectChanges() immediately after the change, or Promise.resolve().then() / setTimeout() to defer to the next cycle.
Q8. What is ngDoCheck and when should you use it?
ngDoCheck runs on every change detection cycle and allows custom dirty checking. Use it with KeyValueDiffer or IterableDiffer to detect mutations in objects or arrays that Angular's default comparison misses. In practice, it is rarely needed — prefer immutable data patterns that use reference changes to trigger ngOnChanges instead. Avoid heavy logic here as it degrades performance.
Q9. What happens to lifecycle hooks with ChangeDetectionStrategy.OnPush?
With OnPush, Angular skips change detection for the component unless: an @Input() reference changes, an event originates from the component, an Observable used with async pipe emits, or markForCheck() / detectChanges() is called. However, ngDoCheck still runs every global CD cycle even with OnPush — it does NOT mean the view is being updated.

Interview Q&A — Senior Level (10 YOE)

Q10. How do lifecycle hooks behave differently in Angular Universal (SSR)?
ngOnInit, ngAfterContentInit, and ngAfterViewInit all run on the server in SSR. However, DOM APIs (document, window, localStorage) are not available server-side. Guard DOM access with isPlatformBrowser(PLATFORM_ID). ngOnDestroy runs during server-side rendering cleanup to prevent memory leaks between requests.
Q11. How do Angular Signals (v17+) change lifecycle hook usage?
effect() replaces most ngOnChanges use cases — it auto-tracks signal reads and re-runs reactively. computed() replaces derived state logic. input() signal replaces @Input() + ngOnChanges combo. takeUntilDestroyed in constructor eliminates ngOnDestroy for subscriptions. The trend is toward fewer explicit lifecycle hooks.
Q12. In a micro-frontend (Single SPA / Module Federation) context, what lifecycle considerations apply?
In micro-frontends, Angular lifecycle hooks behave normally within each app. Key considerations: (1) ngOnDestroy must be robust — the shell unmounts the app aggressively; (2) global event listeners and singleton services from multiple Angular instances can leak if not cleaned up in ngOnDestroy; (3) Zone.js conflicts between multiple Angular instances can cause unexpected change detection — use zoneless rendering (provideExperimentalZonelessChangeDetection) or careful zone forking.
Q13. How do you use lifecycle hooks in dynamic components (ComponentRef)?
Dynamically created components via ViewContainerRef.createComponent() go through all lifecycle hooks normally. You get a ComponentRef — call componentRef.destroy() to trigger ngOnDestroy. You must manage the lifecycle manually: Angular does NOT automatically destroy components created via createComponent when the parent navigates away unless you explicitly destroy them in the parent's ngOnDestroy.
Q14. What is the lifecycle of a standalone component vs NgModule component?
Lifecycle hooks are identical for both standalone and NgModule components. The difference is purely in bootstrapping and DI scope, not in how hooks fire. Standalone components introduced in Angular 14 use bootstrapApplication with provideRouter etc., but the component class lifecycle remains the same.

Patterns & Best Practices

✅ DO

  • Use ngOnInit for all initialization logic
  • Always implement the interface (e.g. implements OnInit)
  • Use takeUntilDestroyed (Angular 16+) for subscriptions
  • Use firstChange in ngOnChanges to skip initial run
  • Use cdr.detectChanges() after DOM mutations in ngAfterViewInit
  • Clean up all timers, third-party libs in ngOnDestroy
  • Use immutable patterns to leverage ngOnChanges over ngDoCheck
  • Guard DOM access in SSR with isPlatformBrowser()

❌ DON'T

  • Put HTTP calls or business logic in the constructor
  • Mutate objects and expect ngOnChanges to fire
  • Put heavy logic in ngDoCheck, ngAfterViewChecked, or ngAfterContentChecked
  • Update template-bound properties in ngAfterViewChecked without cdr
  • Forget to destroy dynamic components created via createComponent
  • Call detectChanges() inside ngAfterViewChecked (infinite loop)
  • Subscribe without unsubscribing (memory leak)
// ✅ Best practice: Modern Angular 16+ component
@Component({
  selector: 'app-best-practice',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: '...'
})
export class BestPracticeComponent implements OnInit, AfterViewInit {
  @Input({ required: true }) userId!: number;

  private destroyRef = inject(DestroyRef);
  private cdr       = inject(ChangeDetectorRef);
  user$!: Observable<User>;

  ngOnInit(): void {
    this.user$ = this.userService.getUser(this.userId);

    this.userService.updates$
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(() => this.cdr.markForCheck());
  }

  ngAfterViewInit(): void {
    // DOM is ready — init 3rd-party lib
    this.initChart();
    this.cdr.detectChanges();  // safe here if needed
  }
}
