Eager vs Lazy Loading

🐢 Eager Loading (Default)

  • Every component/module is statically imported — bundler includes everything in one main.js
  • Router navigates instantly (code already downloaded) but initial page load is slow
  • Large initial bundle → slower FCP & TTI
  • ng build output: only main.js + polyfills.js

🚀 Lazy Loading

  • Feature routes use dynamic import() inside loadComponent / loadChildren
  • Bundler sees each import() as a split point → emits a separate chunk file
  • That chunk is only downloaded on first visit to that route
  • ng build output: main.js + one chunk.js per lazy route

📦 Actual ng build output — same app, two configurations

// ❌ WITHOUT lazy loading (all eager)
// Every route uses: component: HomeComponent

dist/my-app/
  main-a1b2c3.js     // 850 KB — everything
  polyfills-xyz.js   // ~35 KB
  styles-abc.css      // CSS

// Browser downloads 850 KB before app starts
// ✅ WITH lazy loading (loadComponent)
// Routes use: loadComponent: () => import(...)

dist/my-app/
  main-a1b2c3.js       // 120 KB — bootstrap only
  chunk-admin-x9z.js   // 350 KB — on demand
  chunk-products-abc.js // 280 KB — on demand
  chunk-reports-def.js  // 200 KB — on demand
  polyfills-xyz.js      // ~35 KB
  styles-abc.css        // CSS

// Browser downloads only 120 KB at startup ✅
⚠️ Key Point — Separate chunks only appear WHEN you use dynamic import()

If your route uses component: AdminComponent (static import), there is no extra chunk — everything stays in main.js. The separate chunk-*.js files are created by the bundler only when it encounters loadComponent: () => import('./admin/admin.component'). The import() call is what triggers code splitting.

✅ How to verify in your own project

Run ng build and check the dist/ folder. If you see only main.js, your routes are all eager. Add loadComponent and rebuild — you will see new chunk-*.js files appear.

🔑 Key Metrics Impacted by Lazy Loading
  • FCP – First Contentful Paint
    Time until the browser renders the first piece of visible content (text, image). A smaller main.js (due to lazy loading) means the browser parses less JS → FCP improves. Target: < 1.8s
  • TTI – Time to Interactive
    Time until the page is fully interactive — all event handlers attached, no long tasks blocking the main thread. Less JS to parse at startup = faster TTI. Target: < 3.8s
  • TBT – Total Blocking Time
    Sum of time the main thread was blocked (by JS parsing/execution) between FCP and TTI. A 850 KB eager bundle blocks the thread far longer than a 120 KB lazy bundle. Target: < 200ms
  • Lighthouse Performance Score
    Weighted score (0–100) calculated from FCP, TBT, LCP, CLS & Speed Index. Lazy loading directly improves FCP & TBT which together account for ~50% of the total score.

loadComponent — Lazy Standalone Components

What is loadComponent?

Introduced in Angular 14, default pattern in Angular 17+. Lazy-loads a single standalone component directly in the route config — no feature module needed.

When to use?

  • Standalone component architecture (Angular 17+)
  • Single-component routes (profile, settings, about)
  • Any route that doesn't need a feature module grouping
// app.routes.ts — Angular v22 (Standalone Architecture)
import { Routes } from '@angular/router';

export const routes: Routes = [
  {
    path: '',
    loadComponent: () => import('./home/home.component')
                            .then(m => m.HomeComponent)  // named export
  },
  {
    path: 'dashboard',
    loadComponent: () => import('./dashboard/dashboard.component')
                            .then(m => m.DashboardComponent),
    canActivate: [authGuard]
  },
  {
    path: 'admin',
    // default export shortcut — no .then() needed
    loadComponent: () => import('./admin/admin.component')
  }
];

// main.ts — bootstrapApplication
bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(routes, withPreloadingStrategy(PreloadAllModules))
  ]
});
✅ Default Export Shortcut — Explained with Code

When a file has a named export, you must tell Angular which class to use via .then(m => m.XComponent). When a file uses default export, Angular picks it up automatically — no .then() needed.

// ❌ NAMED export — .then() IS required

// admin.component.ts
@Component({ standalone: true, ... })
export class AdminComponent {}   // named export

// routes — must pick the class by name:
{
  path: 'admin',
  loadComponent: () =>
    import('./admin/admin.component')
      .then(m => m.AdminComponent) // required ✅
}
// ✅ DEFAULT export — .then() NOT needed

// admin.component.ts
@Component({ standalone: true, ... })
export default class AdminComponent {} // default

// routes — Angular resolves it automatically:
{
  path: 'admin',
  loadComponent: () =>
    import('./admin/admin.component')
    // no .then() needed ✅
}

Why does it work? A dynamic import() returns a module object. A default export is stored under the key default on that object (m.default). Angular's router automatically checks m.default when no .then() is provided — so it finds and uses your component class without you writing the selector manually.

⚠️ Must be Standalone

The component must have standalone: true (default in Angular 17+). Class-based non-standalone components need loadChildren with a module.

loadChildren — Lazy Feature Routes (Deep Dive)

loadChildren tells Angular's router to defer loading an entire group of child routes until the user navigates to that path. Two approaches exist — pick based on your Angular version.

How it works

  • loadChildren returns a dynamic import of a .routes.ts file
  • That file exports a Routes array with child routes
  • Each child can also use loadComponent → max code splitting
  • No @NgModule, no RouterModule.forChild() needed

Folder Structure

src/app/
  app.routes.ts         ← root routes
  admin/
    admin.routes.ts     ← feature routes
    admin-shell.component.ts
    users/
      users.component.ts
    reports/
      reports.component.ts
// ── STEP 1: app.routes.ts (root) ────────────────────────────────────
import { Routes } from '@angular/router';

export const routes: Routes = [
  { path: '',      loadComponent: () => import('./home/home.component') },

  {
    path: 'admin',
    // 👇 dynamically imports admin.routes.ts only when user visits /admin
    loadChildren: () => import('./admin/admin.routes')
                            .then(m => m.adminRoutes)
  }
];

// ── STEP 2: admin/admin.routes.ts (feature routes) ──────────────────
import { Routes } from '@angular/router';

export const adminRoutes: Routes = [
  // '' means /admin itself → loads AdminShellComponent
  { path: '',        loadComponent: () => import('./admin-shell.component') },

  // /admin/users → loads UsersComponent
  { path: 'users',   loadComponent: () => import('./users/users.component') },

  // /admin/reports → loads ReportsComponent
  { path: 'reports', loadComponent: () => import('./reports/reports.component') }
];

// ── STEP 3: admin-shell.component.ts ────────────────────────────────
import { Component } from '@angular/core';
import { RouterOutlet, RouterLink } from '@angular/router';

@Component({
  standalone: true,
  imports: [RouterOutlet, RouterLink],
  template: `
    <nav>
      <a routerLink="users">Users</a>
      <a routerLink="reports">Reports</a>
    </nav>
    <router-outlet />  <!-- child components render here -->
  `
})
export default class AdminShellComponent {}

How it works

  • loadChildren returns a dynamic import of an NgModule class
  • That module uses RouterModule.forChild() to define child routes
  • Module's declarations array lists all feature components
  • Angular 16 and below — this was the only way

Folder Structure

src/app/
  app-routing.module.ts   ← root routes
  admin/
    admin.module.ts       ← NgModule
    admin-routing.module.ts ← child routes
    admin.component.ts
    users.component.ts
    reports.component.ts
// ── STEP 1: app-routing.module.ts ───────────────────────────────────
const routes: Routes = [
  { path: '', component: HomeComponent },       // eager
  {
    path: 'admin',
    // 👇 dynamically imports AdminModule when user visits /admin
    loadChildren: () => import('./admin/admin.module')
                            .then(m => m.AdminModule)
  }
];

@NgModule({ imports: [RouterModule.forRoot(routes)], exports: [RouterModule] })
export class AppRoutingModule {}

// ── STEP 2: admin/admin-routing.module.ts ───────────────────────────
const adminRoutes: Routes = [
  { path: '',        component: AdminComponent   },
  { path: 'users',   component: UsersComponent   },
  { path: 'reports', component: ReportsComponent }
];

// 👇 forChild() — NOT forRoot() — registers routes for this module only
@NgModule({ imports: [RouterModule.forChild(adminRoutes)], exports: [RouterModule] })
export class AdminRoutingModule {}

// ── STEP 3: admin/admin.module.ts ───────────────────────────────────
@NgModule({
  declarations: [            // register ALL components for this module
    AdminComponent,
    UsersComponent,
    ReportsComponent
  ],
  imports: [
    CommonModule,
    AdminRoutingModule     // import routing module
  ]
})
export class AdminModule {}
⚠️ Problems with Old Way

Every component must be declared in NgModule — boilerplate heavy. RouterModule.forChild() is required in every feature. Cannot use loadComponent for individual child routes — all 3 components load together in one chunk even if user visits only one.

Aspect 🗂 Old Way — NgModule (pre v17) ✅ New Way — Routes Array (v17+ / v22)
loadChildren returns An NgModule class A Routes array
Child routes defined in RouterModule.forChild() inside NgModule A plain .routes.ts file — just an array
Components registered via declarations: [] in NgModule loadComponent inside each route
Individual chunk per component? No — all components in one chunk Yes — each loadComponent = its own chunk
Boilerplate High — 3 files minimum (module, routing module, component) Low — 1 routes file + components
standalone: true required? No — class components Yes — standalone components (default in v17+)
Tree-shakeable? Partial Fully tree-shakeable
Angular v22 recommended? No — being phased out Yes — official best practice
✅ Key Takeaway

Old way puts ALL feature components in one chunk (admin.module.js). New way gives each component its own chunk via loadComponent — so if a user visits /admin/users, only the users chunk downloads, not the reports chunk. This is the real performance advantage of the new approach.

How Angular Splits Bundles (Internals)

Angular doesn't split bundles itself — it writes dynamic import() calls and hands the code to a bundler (Webpack or esbuild) which physically creates the separate files.

Step 1 — You write a dynamic import()

When you use loadComponent: () => import('./admin.component'), you are writing a dynamic import — a JavaScript standard (ES2020) that loads a module at runtime rather than at parse time.


This is different from a static import at the top of a file. Static imports are resolved at build time. Dynamic imports are resolved at runtime.

Step 2 — Bundler sees it as a split boundary

During ng build, the bundler scans all files. Every time it finds a import() call, it treats that as a code-split point and emits the imported file (and all its dependencies) into a separate output chunk file.


The main bundle gets a tiny loader stub that knows the chunk's filename and fetches it over HTTP when needed.

// ── What YOU write in routes ──────────────────────────────────────────────────
{
  path: 'admin',
  loadComponent: () => import('./admin/admin.component')  // dynamic import()
}

// ── What the BUNDLER emits ────────────────────────────────────────────────────
// main.js gets a tiny stub (pseudo-code):
const chunkMap = {
  './admin/admin.component': 'chunk-admin-x9z.js'   // filename decided at build time
};
// When router calls loadComponent(), browser runs:
fetch('/chunk-admin-x9z.js')
  .then(script => eval(script))    // executes the chunk
  .then(module => module.default)  // returns the component class

// ── Actual ng build output ────────────────────────────────────────────────────
dist/my-app/
  main-a1b2c3.js          // bootstrap + eagerly imported code
  chunk-admin-x9z.js      // AdminComponent + its imports
  chunk-products-abc.js   // ProductComponent + its imports
  common-xyz.js           // shared code (e.g. @angular/forms used by 2+ chunks)
✅ Static vs Dynamic Import
TypeSyntaxResolved atCreates chunk?
Staticimport { X } from './x'Build timeNo — goes into main.js
Dynamicimport('./x')RuntimeYes — separate chunk

Shared Chunk (Common Chunk)

If @angular/forms is used by both admin.chunk.js and products.chunk.js, the bundler moves it into a shared common chunk downloaded once and cached — preventing duplication across multiple lazy chunks.

What is Webpack?

Webpack is a JavaScript module bundler. It takes your source files (TypeScript, CSS, assets) and packages them into optimised output files for the browser. Angular CLI used Webpack as its default bundler from Angular 2 up to Angular 16.

How Webpack handles dynamic import()

  • Webpack scans the entire dependency graph at build time
  • Every import('./x') becomes a split point
  • Webpack assigns each split point a chunk ID (a number or name)
  • Emits a separate chunk-[id].js file for each split point
  • Injects a runtime manifest into main.js that maps chunk IDs to filenames
// ── Webpack internals (simplified) ───────────────────────────────────────────

// 1. You write:
loadComponent: () => import('./admin/admin.component')

// 2. Webpack transforms it into:
loadComponent: () => __webpack_require__.e(/* chunk ID */ 42)
  .then(__webpack_require__.bind(__webpack_require__, 1234))

// 3. __webpack_require__.e() does this at runtime:
//    - checks if chunk 42 is already loaded (cache)
//    - if not: creates a <script> tag pointing to chunk-42.js
//    - resolves the promise when the script loads

// ── Magic comments — control chunk names ─────────────────────────────────────
loadComponent: () =>
  import(
    /* webpackChunkName: "admin" */     // names the output file
    /* webpackPrefetch: true */          // browser prefetches when idle
    /* webpackPreload: true */           // browser loads in parallel
    './admin/admin.component'
  )
// Output file: chunk-admin.js  (instead of chunk-42.js)

// ── Angular CLI Webpack config (angular.json) ────────────────────────────────
// Angular 2–16 used Webpack under the hood via @angular-devkit/build-angular
// You rarely need webpack.config.js — Angular CLI handled it automatically
⚠️ Webpack in Angular v17+

Angular CLI switched the default builder to esbuild in v17. Webpack is still available via @angular-devkit/build-angular:browser builder but esbuild (application builder) is now the recommended default.

✅ webpackPrefetch vs webpackPreload
  • prefetch — low priority, downloaded during browser idle. Use for routes the user might visit next.
  • preload — high priority, downloaded in parallel with parent chunk. Use for resources needed immediately after parent loads.

What is esbuild?

esbuild is an extremely fast JavaScript/TypeScript bundler and minifier written in Go (not JavaScript). It is 10–100× faster than Webpack because it uses Go's concurrency model and avoids the overhead of Node.js.


Angular CLI adopted esbuild as the default builder starting Angular v17 via the application builder.

✅ Practical Impact of esbuild for Lazy Loading

With esbuild, ng build is dramatically faster and ng serve hot-reload is near-instant. Code splitting behaviour is identical to Webpack — every import() still creates a separate chunk. The difference is purely build speed, not the runtime behaviour in the browser.

🔄 Complete Step-by-Step — User navigates to /admin

1
User clicks a link or types /admin in browser
Browser fires a popstate / click event. Angular's router intercepts it — no full page reload.
2
Router scans the routes array for a matching path
It finds { path: 'admin', loadChildren: () => import(...) } and sees it's a lazy route.
3
canMatch guards run (if any)
If a guard returns false, the route is skipped entirely — chunk is never downloaded. If true, proceed.
4
Router calls the loadChildren / loadComponent factory function
This executes import('./admin/admin.routes') — the browser sends an HTTP GET request for chunk-admin-x9z.js.
5
Browser downloads and executes the chunk
The chunk JS runs, registering the module. The promise resolves with the Routes array or component class.
6
Router registers child routes from the loaded Routes array
Angular merges the lazy routes into its internal route tree for this session.
7
canActivate guards run (if any)
Final auth/permission check before the component is created.
8
Resolver runs (if any) — pre-fetches data
Route resolver fetches API data before the component renders, so no loading spinner inside the component.
9
Component is instantiated and rendered into <router-outlet>
Angular creates the component, runs lifecycle hooks, and inserts the DOM.
✓
Chunk is now cached in the browser
Second visit to /admin skips steps 4–5 entirely — the cached chunk is used immediately. HTTP 304 or memory cache.

Preloading Strategies

Preloading downloads lazy chunks in the background after the app is interactive — combining fast initial load with near-instant subsequent navigation.

NoPreloading (default)

  • Chunks downloaded only on demand
  • First visit to a lazy route has a small delay
  • Best for slow networks / rarely visited routes

PreloadAllModules

  • All lazy chunks preloaded after initial render
  • Uses browser idle time
  • Best for fast networks & feature-rich apps

Custom Strategy

  • Selective preloading via route data
  • Preload only high-priority routes
  • Best approach for large enterprise apps
// ── Built-in strategies ─────────────────────────────────────────────
import { provideRouter, withPreloadingStrategy,
         PreloadAllModules, NoPreloading } from '@angular/router';

bootstrapApplication(AppComponent, {
  providers: [
    provideRouter(
      routes,
      withPreloadingStrategy(PreloadAllModules)   // ← change here
    )
  ]
});

// ── Custom Selective Preloading Strategy ────────────────────────────
import { PreloadingStrategy, Route } from '@angular/router';
import { Observable, of } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class SelectivePreloadStrategy extends PreloadingStrategy {
  preload(route: Route, load: () => Observable<any>): Observable<any> {
    // Only preload routes flagged with data: { preload: true }
    return route.data?.['preload'] === true ? load() : of(null);
  }
}

// Routes — mark high-priority routes:
{ path: 'dashboard', loadComponent: () => import('...'), data: { preload: true } },
{ path: 'admin',     loadComponent: () => import('...'), data: { preload: false } },

// Register custom strategy:
provideRouter(routes, withPreloadingStrategy(SelectivePreloadStrategy))

canMatch — Lazy Load + Guard Fusion

canMatch vs canActivate — Key Difference

GuardRunsDownloads Chunk?Use Case
canActivateAfter route matched & chunk downloadedYes — alwaysAuth check after load
canMatchBefore route matching & before chunk downloadNo — blocked earlyRole-based lazy routes
canLoadBefore chunk download (deprecated)Replaced by canMatchUse canMatch instead

canMatch Real-World Pattern

Use canMatch to serve different feature routes to different user roles — the wrong route's chunk is never downloaded at all.


Example: /dashboard serves AdminDashboard for admins and UserDashboard for regular users.

// guards/role.guard.ts
import { inject } from '@angular/core';
import { CanMatchFn } from '@angular/router';
import { AuthService } from './auth.service';

export const adminCanMatch: CanMatchFn = () =>
  inject(AuthService).hasRole('admin');

export const userCanMatch: CanMatchFn = () =>
  inject(AuthService).hasRole('user');

// app.routes.ts — same path, different component per role
export const routes: Routes = [
  {
    path: 'dashboard',
    canMatch: [adminCanMatch],                       // checked first
    loadComponent: () => import('./admin/admin-dashboard.component')
  },
  {
    path: 'dashboard',
    canMatch: [userCanMatch],                        // fallback
    loadComponent: () => import('./user/user-dashboard.component')
  }
];
// 🔑 Admin chunk is NEVER downloaded for regular users!
✅ Senior Insight

canMatch prevents unnecessary chunk downloads for unauthorized users. This is both a security improvement (code not sent to client) and a performance improvement (no wasted bandwidth). Always prefer canMatch over canLoad (deprecated in v15).

@defer — View-Level Lazy Loading

What is @defer?

Introduced in Angular 17. A template-level construct that defers loading of any component, directive, or pipe inside a template block. No routing required — works anywhere in the template.

Route Lazy vs @defer

Route Lazy@defer
ScopeRoute levelTemplate / component level
TriggerNavigationViewport, idle, timer, interaction, custom
Best forFeature pagesHeavy widgets, charts, below-fold content
 
<!-- Angular v22 — @defer with all trigger types ──────────────────────────────>

<!-- 1. Viewport trigger — load when element scrolls into view -->
@defer (on viewport) {
  <app-heavy-chart />
} @loading {
  <app-skeleton-loader />
} @placeholder {
  <div>Chart will appear here</div>
} @error {
  <p>Failed to load chart</p>
}

<!-- 2. Idle trigger — load during browser idle time -->
@defer (on idle) {
  <app-footer-recommendations />
}

<!-- 3. Interaction trigger — load on user hover/click -->
@defer (on interaction(triggerBtn)) {
  <app-comments-section />
}
<button #triggerBtn>Show Comments</button>

<!-- 4. Timer trigger — load after delay -->
@defer (on timer(2000)) {
  <app-promo-banner />
}

<!-- 5. Prefetch while showing placeholder -->
@defer (on interaction; prefetch on idle) {
  <app-rich-editor />
}
✅ Senior Use Case — @defer + minimum

Use @defer (on viewport; minimum 500ms) to prevent flash of loading state. The minimum option ensures @loading is shown for at least N ms even if the chunk loads instantly.

Lazy Loading NgRx State with Feature Routes

Why Lazy-load NgRx State?

  • Register reducers/effects only when the feature route is visited
  • Keeps global store lean at startup
  • Avoids memory waste for rarely-visited features
  • State is automatically unregistered on route leave (optional)

provideState() — Standalone API ✅

In NgRx v17+ (pairs with Angular 17+), use provideState() and provideEffects() in the route's providers array for fully standalone lazy state.

// admin/admin.routes.ts — lazy NgRx state registration
import { provideState, provideEffects } from '@ngrx/store';
import { adminReducer }  from './state/admin.reducer';
import { AdminEffects }  from './state/admin.effects';

export const adminRoutes: Routes = [
  {
    path: '',
    loadComponent: () => import('./admin-shell.component'),
    providers: [                                     // ← route-level providers
      provideState('admin', adminReducer),           // lazy reducer
      provideEffects(AdminEffects)                  // lazy effects
    ]
  }
];

// admin.reducer.ts
export interface AdminState { users: User[]; loading: boolean; }

export const adminReducer = createReducer(
  { users: [], loading: false },
  on(AdminActions.loadUsers,  state => ({ ...state, loading: true  })),
  on(AdminActions.loadSuccess, (state, { users }) => ({ users, loading: false }))
);

// Selector — works just like eager state
export const selectAdminUsers = createFeatureSelector<AdminState>('admin');
✅ Route-level providers (Angular 14+)

The providers array on a route is an Angular 14+ feature. It creates a scoped injector — perfect for lazy services, interceptors, or NgRx slices that should only exist while the route is active.

Route-Level Providers & Scoped Injection

What are Route-Level Providers?

Angular 14+ allows each route to declare its own providers. These are available to the component tree of that route only. Think of it as a scoped module without NgModule.

// Scoped services, interceptors, configs per lazy route
export const productRoutes: Routes = [
  {
    path: '',
    loadComponent: () => import('./product-shell.component'),
    providers: [
      ProductApiService,                        // scoped to this route tree
      { provide: HTTP_INTERCEPTORS,
        useClass: ProductAuthInterceptor,
        multi: true },
      { provide: PRODUCT_CONFIG,
        useValue: { pageSize: 20, currency: 'USD' } }
    ],
    children: [
      { path: 'list',   loadComponent: () => import('./list/product-list.component') },
      { path: ':id',    loadComponent: () => import('./detail/product-detail.component') },
      { path: 'new',    loadComponent: () => import('./form/product-form.component') }
    ]
  }
];

// Injecting in a child component:
@Component({ standalone: true, ... })
export class ProductDetailComponent {
  private config = inject(PRODUCT_CONFIG); // comes from route providers
  private api    = inject(ProductApiService);
}
✅ Replaces NgModule forRoot/forChild

Route-level providers are the standalone replacement for RouterModule.forChild() module providers. Each feature module's services are now declared directly on the route.

⚠️ Scope Awareness

A service declared in route providers is not available outside that route tree. Do not inject it in components outside the route. Angular will throw a NullInjectorError.

Common Pitfalls & Gotchas

❌ Pitfall 1 — Importing lazy component in AppModule

If you import a lazy component/module in AppModule or any eagerly loaded file, the bundler includes it in main.js — defeating the purpose.


Fix: Never import lazy components statically. Access them only via loadComponent / loadChildren.

❌ Pitfall 2 — Using canLoad (deprecated)

canLoad was deprecated in Angular 15 and removed in v16+. It also prevented preloading.


Fix: Replace all canLoad guards with canMatch. It does the same job plus supports preloading.

❌ Pitfall 3 — Services provided in lazy root

A service with providedIn: 'root' inside a lazy feature is still singleton and eagerly included if imported elsewhere.


Fix: For truly feature-scoped services, use route-level providers array instead.

❌ Pitfall 4 — Forgetting @loading / @placeholder

Using @defer without a @placeholder causes layout shift (CLS) when the component loads.


Fix: Always provide a @placeholder with the same dimensions as the deferred content.

❌ Pitfall 5 — Not analysing bundle size

Developers assume lazy loading is working without actually checking bundle output. Shared deps may silently inflate chunks.


Fix: Run ng build --stats-json + webpack-bundle-analyzer after every major feature addition.

❌ Pitfall 6 — Lazy loading already-small components

Lazy loading tiny components (under ~5 KB) adds an extra HTTP round-trip with no meaningful size benefit.


Fix: Only lazy-load routes with significant code weight. Small utility components should be eagerly imported.

Senior Interview Q&A — Part 1

Q1. What is the difference between loadComponent and loadChildren? CONCEPT
loadComponent lazy-loads a single standalone component for a route. loadChildren lazy-loads an entire Routes array (or legacy NgModule) that can contain multiple child routes. Use loadComponent for simple single-component routes and loadChildren when you have a feature with multiple nested routes.
Q2. Why is canMatch preferred over canLoad? GUARDS
canLoad was deprecated in Angular v15. Unlike canLoad, canMatch works with preloading strategies — a canLoad guard would block preloading entirely. canMatch also allows the same path to match different routes based on conditions (role-based routing), which canLoad cannot do.
Q3. How does Angular create separate JavaScript bundles for lazy routes? INTERNALS
Angular uses dynamic imports (import()) inside loadComponent/loadChildren. The bundler (esbuild in Angular v17+) treats each unique dynamic import as a code-split boundary and emits a separate chunk file. Shared dependencies are hoisted into a common chunk to avoid duplication.
Q4. What is the difference between PreloadAllModules and a custom preloading strategy? PERFORMANCE
PreloadAllModules preloads every lazy chunk after the initial render. A custom strategy implements PreloadingStrategy and selectively preloads only routes whose data flag is set (e.g. data: { preload: true }). Custom strategy is better for large apps where preloading everything wastes bandwidth on low-priority features.
Q5. How does @defer differ from route-level lazy loading? DEFER
Route lazy loading splits bundles at the routing level — chunks load when the user navigates. @defer is template-level — it defers loading of components/directives within a page based on triggers like viewport, idle, interaction, or timer. Both create separate JS chunks. @defer is ideal for heavy widgets within a page (charts, editors, comment sections).
Q6. If you import a lazy component in AppModule, what happens? PITFALL
The component gets bundled into main.js at build time, completely defeating lazy loading. The loadComponent call will still work but the chunk will be empty (or duplicated). Always ensure lazy components are referenced only through loadComponent / loadChildren — never statically imported in any eagerly-loaded file.

Senior Interview Q&A — Part 2

Q7. How do you lazy-load NgRx state in a standalone Angular app? NGRX
Use provideState(featureName, reducer) and provideEffects(EffectsClass) inside the route's providers array. This registers the reducer/effects only when that route is first visited. The state slice becomes available to the component tree of that route and can be accessed via createFeatureSelector.
Q8. What is route-level providers and why is it important? DI
Route-level providers (Angular 14+) creates a scoped DI injector for that route's component tree. It replaces the need for NgModule's forChild() pattern. Services provided here are only available within that route and its children — not globally. This enables proper isolation, avoids polluting the root injector, and supports tree-shaking of feature-specific services.
Q9. How would you implement role-based lazy loading? ADVANCED
Define the same path twice in the routes array, each with a different canMatch guard and different loadComponent. Angular tries each route definition in order — the first one whose canMatch returns true wins. The chunk for the other role is never downloaded. This is more secure than canActivate because the wrong component's code never reaches the client.
Q10. How do you prevent CLS (Cumulative Layout Shift) when using @defer? PERFORMANCE
Always provide a @placeholder block with the same height/width as the deferred content. Use CSS skeletons or fixed-height containers. Also set minimum duration to prevent a flash of loading state: @defer (on viewport; minimum 300ms). This ensures a stable layout while content loads.
Q11. What does withComponentInputBinding() do and how does it help with lazy routes? ANGULAR V22
It allows Angular to automatically bind route params, query params, and resolve data to component @Input() properties by name. Without it, you'd need to inject ActivatedRoute and subscribe to params manually. With lazy routes, this removes boilerplate and keeps components pure and testable — they just receive data as inputs without knowing they're inside a router.
Q12. How would you debug why a route is not being lazy-loaded as expected? DEBUG
1) Run ng build --stats-json and inspect the output chunks — if no separate chunk exists, the component is eagerly loaded. 2) Check for static imports of the lazy component anywhere in eagerly loaded files. 3) Add withDebugTracing() to provideRouter() to log router events. 4) Use Chrome DevTools Network tab — filter by JS and navigate to the route to see if a new chunk is fetched.
