01

What is RxJS?

Definition

RxJS (Reactive Extensions for JavaScript) is a library for composing asynchronous and event-based programs using Observables. It provides a powerful set of operators to transform, filter, combine and control streams of data over time.

Core Concepts

  • Observable — a stream of values over time (lazy, not started until subscribed)
  • Observer — consumes values: next, error, complete
  • Subscription — the connection between Observable and Observer
  • Operator — pure function that transforms an Observable into another Observable
  • Subject — both Observable and Observer (multicast)

Operator Categories

CategoryOperatorsPurpose
Transformationmap, mergeMap, switchMap, concatMap, exhaustMapTransform emitted values
Filteringfilter, debounceTime, distinctUntilChanged, take, skipControl which values pass through
CombinationcombineLatest, forkJoin, zip, merge, concatCombine multiple Observables
Utilitytap, delay, timeout, finalizeSide effects, debugging
Error HandlingcatchError, retry, retryWhenHandle errors gracefully
MulticastingSubject, BehaviorSubject, ReplaySubjectShare a single subscription
 
import { Observable, of } from 'rxjs';
import { map, filter } from 'rxjs/operators';

// Create an observable
const numbers$ = of(1, 2, 3, 4, 5);

// Apply operators via pipe()
numbers$.pipe(
  filter(n => n % 2 === 0),   // keep even: 2, 4
  map(n => n * 10)             // multiply: 20, 40
).subscribe(val => console.log(val));
// Output: 20, 40
02
map / filter / tap Transformation & Utility Most Basic

map

Transforms each emitted value. Like Array.map() but for streams.

Input: 1 ── 2 ── 3
map(x*10): 10 ─ 20 ─ 30

filter

Only passes values that satisfy the predicate. Like Array.filter() for streams.

Input: 1 ── 2 ── 3 ── 4
filter(even): ──── 2 ──── 4

tap

Side-effect only — does NOT modify the stream. Used for logging or debugging.

Input: 1 ── 2 ── 3
tap(log): 1 ── 2 ── 3 ← unchanged 
import { map, filter, tap } from 'rxjs/operators';

this.userService.getUsers().pipe(

  tap(users => console.log('Raw API response:', users)),
  // ↑ log without modifying the stream

  filter(users => users.length > 0),
  // ↑ only continue if array is not empty

  map(users => users.map(u => ({ ...u, fullName: `${u.firstName} ${u.lastName}` }))),
  // ↑ transform: add fullName property to each user

  tap(users => console.log('Transformed:', users))
  // ↑ log the final result before subscribe

).subscribe(users => this.users = users);

Key Difference — tap vs map

OperatorModifies stream?Use for
map✅ Yes — returns new valueTransform data
tap❌ No — passes value through unchangedLogging, side effects, analytics
03
switchMap Higher-Order Mapping Most Asked ⭐

What it does

  • Maps each value to an inner Observable
  • Cancels the previous inner Observable when a new value arrives
  • Only the latest inner Observable's result is used
  • Perfect for: search-as-you-type, route params, dropdown selections

Marble Diagram

Source: A ──── B ──── C
A inner: a1 ─ a2❌ cancelled when B arrives
B inner: ──── b1 b2❌ cancelled when C arrives
Output: a1 ──── b1 ── c1 
import { switchMap } from 'rxjs/operators';

// ── Search as you type ───────────────────────────────────────
this.searchControl.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.api.search(term))
  // ↑ if user types 'n', 'ni', 'nit' rapidly:
  //   cancels request for 'n', cancels request for 'ni'
  //   only keeps the latest request for 'nit'
).subscribe(results => this.results = results);

// ── Route param change ───────────────────────────────────────
this.route.params.pipe(
  switchMap(params => this.api.getUser(params['id']))
  // ↑ if user navigates from /user/1 → /user/2
  //   cancels request for user 1, fetches user 2
).subscribe(user => this.user = user);

⚠️ When NOT to use switchMap

Do NOT use switchMap for write operations (POST, PUT, DELETE). If the user clicks "Save" twice, the first request gets cancelled — data may not be saved. Use concatMap or exhaustMap instead.

04
mergeMap Higher-Order Mapping Parallel Execution

What it does

  • Maps each value to an inner Observable
  • Does NOT cancel previous inner Observables
  • All inner Observables run concurrently
  • Results arrive in the order they complete (not necessarily the source order)
  • Use for: parallel API calls, file uploads, independent tasks

Marble Diagram

Source: A ──── B
A inner: a1 ── a2 ← still running ✅
B inner: ──── b1 ← runs in parallel ✅
Output: a1 ─ b1 ─ a2 
import { mergeMap } from 'rxjs/operators';

// ── Upload multiple files in parallel ────────────────────────
const files$ = from([file1, file2, file3]);

files$.pipe(
  mergeMap(file => this.uploadService.upload(file))
  // ↑ all 3 uploads start simultaneously
  //   results arrive as each upload completes
).subscribe(result => console.log('Uploaded:', result));

// ── Parallel API calls for each item ─────────────────────────
from(userIds).pipe(
  mergeMap(id => this.api.getUser(id))
  // ↑ all requests fire at once — fastest possible
).subscribe(user => this.users.push(user));

⚠️ Risk — too many concurrent requests

If your source emits 1000 items, mergeMap fires 1000 HTTP requests simultaneously. Use the concurrent parameter to limit: mergeMap(fn, 3) — max 3 at a time.

05
concatMap Higher-Order Mapping Sequential / Ordered

What it does

  • Maps each value to an inner Observable
  • Waits for each inner Observable to complete before subscribing to the next
  • Preserves the order of emissions
  • Queues requests — nothing is dropped or cancelled
  • Use for: sequential saves, ordered operations, step-by-step wizards

Marble Diagram

Source: A ── B ── C
Output: a1 a2 ─ b1 b2 ─ c1

B waits for A to complete, C waits for B to complete

import { concatMap } from 'rxjs/operators';

// ── Save form steps in order ─────────────────────────────────
const steps$ = from([step1Data, step2Data, step3Data]);

steps$.pipe(
  concatMap(step => this.api.saveStep(step))
  // ↑ step2 only saves AFTER step1 completes
  //   step3 only saves AFTER step2 completes
  //   order is guaranteed ✅
).subscribe(res => console.log('Step saved:', res));

// ── Queue button clicks (don't drop any) ─────────────────────
this.saveClick$.pipe(
  concatMap(() => this.api.save(this.formData))
  // ↑ every click is queued and processed in order
  //   no click is lost (unlike exhaustMap)
).subscribe();
06
exhaustMap Higher-Order Mapping Ignore while busy

What it does

  • Maps each value to an inner Observable
  • Ignores new source values while an inner Observable is still running
  • Once the inner Observable completes, it accepts the next value
  • Use for: login button, submit button, prevent double-submit

Marble Diagram

Clicks: C1 ── C2❌ ── C3❌ ── C4
Output: r1 ──────────── r4

C2 and C3 ignored while C1's request is in-flight. C4 accepted after C1 completes.

import { exhaustMap } from 'rxjs/operators';

// ── Login button — prevent double submit ─────────────────────
fromEvent(this.loginBtn.nativeElement, 'click').pipe(
  exhaustMap(() => this.authService.login(this.credentials))
  // ↑ user clicks login button 3 times rapidly:
  //   1st click fires the login request
  //   2nd and 3rd clicks are IGNORED while request is in-flight
  //   no duplicate login requests ✅
).subscribe(user => this.navigateToHome());

// ── Form submit button ───────────────────────────────────────
this.submitClick$.pipe(
  exhaustMap(() => this.api.saveForm(this.form.value))
).subscribe();
07

switchMap vs mergeMap vs concatMap vs exhaustMap

Side-by-side Comparison

Operator Previous request Concurrency Order Best for
switchMap ❌ Cancelled One at a time Only latest Search, route params, dropdowns
mergeMap ✅ Kept running All parallel Completion order Parallel file uploads, parallel GET requests
concatMap ⏳ Queued One at a time Source order Sequential saves, ordered steps, wizard
exhaustMap 🚫 Ignored One at a time First wins Login, submit button, prevent double-click

Decision Guide — which one to pick?

// Ask yourself: "What should happen if a NEW value arrives while a request is in-flight?"
//
// "Cancel the old request, only care about the latest"
//   → switchMap   (search, route params)
//
// "Keep all requests running at the same time, I don't care about order"
//   → mergeMap    (parallel uploads, independent GETs)
//
// "Queue all requests, process them one by one in order"
//   → concatMap   (step-by-step saves, sequential mutations)
//
// "Ignore new requests until the current one finishes"
//   → exhaustMap  (login button, form submit, prevent double-click)

✅ Use switchMap for GET (read)

// Search — cancel old, use latest
search$.pipe(
  switchMap(q => api.search(q))
)

❌ Avoid switchMap for POST (write)

// Submit — use exhaustMap instead!
submit$.pipe(
  exhaustMap(() => api.save(data))
)
08
combineLatest / forkJoin Combination Combine Streams

combineLatest

  • Emits whenever any source Observable emits
  • Requires all sources to have emitted at least once before emitting
  • Always emits the latest value from each source
  • Works with ongoing streams
  • Use for: filters + data, form fields combined, live config
A: 1────2
B: ──X──────Y
Output: ────[1,X]─[2,X]─[2,Y]

forkJoin

  • Waits for all Observables to complete
  • Emits a single array of the last values
  • Works only with finite Observables (HTTP calls)
  • If any Observable errors, the whole forkJoin errors
  • Use for: parallel HTTP calls that must all finish first
A (HTTP): ─A|
B (HTTP): ────B|
Output: ────────[A,B] 
import { combineLatest, forkJoin } from 'rxjs';

// ── combineLatest: filter + paginator + data ─────────────────
combineLatest([
  this.filterControl.valueChanges,
  this.pageControl.valueChanges,
]).pipe(
  switchMap(([filter, page]) => this.api.getUsers({ filter, page }))
  // ↑ re-fetches whenever filter OR page changes
).subscribe(users => this.users = users);

// ── forkJoin: load dashboard data (all required) ─────────────
forkJoin({
  users:    this.api.getUsers(),
  products: this.api.getProducts(),
  stats:    this.api.getStats(),
}).subscribe(({ users, products, stats }) => {
  this.users    = users;
  this.products = products;
  this.stats    = stats;
  // ↑ only runs when ALL three HTTP calls complete
});

combineLatest vs forkJoin

FeaturecombineLatestforkJoin
EmitsEvery time any source emitsOnly once — when all complete
Works withOngoing streams (subjects, controls)Finite streams (HTTP calls)
If one source never emitsNever emitsNever emits
Use caseReactive filters, live form stateDashboard load, parallel API calls
09
debounceTime / distinctUntilChanged Filtering Search Pattern ⭐

debounceTime(ms)

  • Waits ms milliseconds after the last emission before passing the value on
  • Resets the timer on every new emission
  • Use for: search input, resize events, scroll events
Keystrokes: nitin
debounce(300): ──────────────── 300ms ───nitin

5 keystrokes → 1 emission after user stops typing

distinctUntilChanged()

  • Skips emission if the value is the same as the previous value
  • Prevents redundant operations when value hasn't actually changed
  • Use for: prevent re-fetching same search term, state changes
Input: AABBC
distinct: A──B──C

Duplicate consecutive values are dropped

import { debounceTime, distinctUntilChanged, switchMap } from 'rxjs/operators';

// ── Complete search pattern ──────────────────────────────────
this.searchControl.valueChanges.pipe(

  debounceTime(400),
  // ↑ wait 400ms after user stops typing

  distinctUntilChanged(),
  // ↑ skip if same value (e.g. user types 'a', deletes, types 'a' again)

  switchMap(term => this.api.search(term)),
  // ↑ cancel previous request, only keep latest

).subscribe(results => this.results = results);

// ── Resize events — don't fire on every pixel ────────────────
fromEvent(window, 'resize').pipe(
  debounceTime(200),
  map(() => window.innerWidth)
).subscribe(width => this.onResize(width));

throttleTime vs debounceTime

debounceTimethrottleTime
FiresAfter user stops for X msAt most once per X ms interval
Use caseSearch input, API calls on inputScroll events, mouse move, analytics

catchError / retry / retryWhen Error Handling Critical

catchError

  • Catches errors and lets you recover with a fallback Observable
  • The stream completes after the fallback — unless you re-throw
  • Must return an Observable from the handler
  • Use EMPTY to silently complete, of(fallback) for default data

retry / retryWhen

  • retry(n) — resubscribes to the source Observable up to n times on error
  • retryWhen — custom retry logic (e.g. exponential backoff)
  • Place retry before catchError so retries happen first
  • Use for: transient network errors, flaky APIs
import { catchError, retry } from 'rxjs/operators';
import { EMPTY, of, throwError } from 'rxjs';

// ── Retry 3 times, then show fallback data ───────────────────
this.api.getUsers().pipe(
  retry(3),
  // ↑ retry up to 3 times before giving up
  catchError(err => {
    console.error('API failed after 3 retries:', err);
    return of([]);   // ← return empty array as fallback
  })
).subscribe(users => this.users = users);

// ── Rethrow specific errors, handle others ───────────────────
this.api.getUser(id).pipe(
  catchError(err => {
    if (err.status === 404) {
      return of(null);          // not found → return null
    }
    if (err.status === 401) {
      this.router.navigate(['/login']);
      return EMPTY;             // silently complete
    }
    return throwError(() => err); // rethrow for unknown errors
  })
).subscribe(user => this.user = user);

// ── Exponential backoff with retryWhen ───────────────────────
import { retryWhen, delayWhen, scan } from 'rxjs/operators';
import { timer } from 'rxjs';

this.api.getData().pipe(
  retryWhen(errors$ => errors$.pipe(
    scan((count, err) => {
      if (count >= 3) throw err;   // give up after 3
      return count + 1;
    }, 0),
    delayWhen(count => timer(count * 1000))
    // retry after 1s, 2s, 3s
  ))
).subscribe();

takeUntil / takeUntilDestroyed Filtering / Lifecycle Memory Leak Prevention ⭐

takeUntil

  • Completes the source Observable when the notifier Observable emits
  • Automatically unsubscribes — no manual cleanup needed
  • Standard pattern: use a Subject as the notifier, emit in ngOnDestroy
  • Prevents memory leaks from subscriptions outliving the component

takeUntilDestroyed (Angular 16+)

  • Angular's built-in operator tied to the component's destroy lifecycle
  • No need for a Subject — Angular handles it automatically
  • Must be called in an injection context (constructor or field initializer)
  • Recommended modern approach in Angular 16+
// ── Pattern 1: takeUntil with Subject (Angular < 16) ────────
export class MyComponent implements OnInit, OnDestroy {
  private destroy$ = new Subject<void>();

  ngOnInit(): void {
    this.api.getUpdates().pipe(
      takeUntil(this.destroy$)
      // ↑ auto-completes when destroy$ emits
    ).subscribe(data => this.data = data);
  }

  ngOnDestroy(): void {
    this.destroy$.next();    // triggers takeUntil → unsubscribes
    this.destroy$.complete();
  }
}

// ── Pattern 2: takeUntilDestroyed (Angular 16+) ──────────────
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

export class MyComponent {
  constructor() {
    this.api.getUpdates().pipe(
      takeUntilDestroyed()
      // ↑ no Subject needed — Angular handles cleanup automatically
    ).subscribe(data => this.data = data);
  }
}

// ── Without takeUntil — MEMORY LEAK ❌ ───────────────────────
// ngOnInit() {
//   this.api.getUpdates().subscribe(...)  // keeps running after component destroyed!
// }

Subject / BehaviorSubject / ReplaySubject Multicasting Most Asked ⭐

Subject

  • Both an Observable and an Observer
  • New subscribers get values emitted after they subscribe
  • No initial value, no replay
  • Use for: events, user actions, destroy$
const s = new Subject<number>();
s.next(1);      // no subscribers yet
s.subscribe(v => console.log(v));
s.next(2);      // logs: 2
s.next(3);      // logs: 3
// 1 was missed ❌

BehaviorSubject

  • Requires an initial value
  • New subscribers immediately get the current value
  • Always has a current value (access via .value)
  • Use for: state management, user auth status, loading flags
const b = new BehaviorSubject(0);
b.next(1);
b.next(2);
b.subscribe(v => console.log(v));
// logs: 2 immediately ✅
b.next(3);
// logs: 3
console.log(b.value); // 3

ReplaySubject

  • Replays the last N values to new subscribers
  • No initial value required
  • Can also set a time window for replayed values
  • Use for: caching last N events, late subscribers catching up
const r = new ReplaySubject(2);
// replays last 2 values
r.next(1);
r.next(2);
r.next(3);
r.subscribe(v => console.log(v));
// logs: 2, 3 (last 2) ✅

Real-world usage in Angular services

// ── Auth service with BehaviorSubject ────────────────────────
@Injectable({ providedIn: 'root' })
export class AuthService {
  private currentUser$ = new BehaviorSubject<User | null>(null);
  // ↑ null = not logged in (initial value)

  // expose as Observable (components can't call .next())
  user$ = this.currentUser$.asObservable();

  login(user: User): void {
    this.currentUser$.next(user);   // push new user
  }
  logout(): void {
    this.currentUser$.next(null);   // clear user
  }
  getUser(): User | null {
    return this.currentUser$.value;  // sync access
  }
}

Interview Questions & Answers

Q1. What is the difference between Observable and Promise?
Promise: eager (executes immediately), single value, not cancellable, no operators.
Observable: lazy (executes only when subscribed), multiple values over time, cancellable (unsubscribe), composable with operators via pipe(). Observables can represent click streams, WebSocket data, HTTP calls — all with the same API.
Q2. What is the difference between switchMap, mergeMap, concatMap and exhaustMap?
switchMap: cancels the previous inner Observable when a new value arrives — use for search.
mergeMap: runs all inner Observables in parallel — use for parallel uploads.
concatMap: queues inner Observables and processes them one by one in order — use for sequential saves.
exhaustMap: ignores new values while an inner Observable is in-flight — use for login/submit buttons.
Q3. What is the difference between Subject, BehaviorSubject and ReplaySubject?
Subject: no initial value, late subscribers miss past emissions.
BehaviorSubject: has an initial value, late subscribers immediately get the current value — ideal for state management.
ReplaySubject(n): replays the last N emissions to new subscribers — ideal for caching recent events.
Q4. What is the difference between combineLatest and forkJoin?
combineLatest: emits whenever any source emits (after all have emitted at least once) — works with ongoing streams like form controls.
forkJoin: waits for ALL sources to complete, then emits one combined result — works only with finite streams like HTTP calls.
Q5. How do you prevent memory leaks from subscriptions in Angular?
Three approaches:
1. takeUntil pattern: use a Subject destroy$, call destroy$.next() in ngOnDestroy.
2. takeUntilDestroyed() (Angular 16+): modern built-in operator — no manual Subject needed.
3. async pipe: Angular automatically unsubscribes when the component is destroyed. Best for template-only subscriptions.
Q6. What does tap() do and when would you use it?
tap() performs side effects without modifying the stream. The value passes through unchanged. Use it for: logging/debugging during development, triggering analytics events, showing loading spinners, and setting flags — without breaking the data pipeline.
Q7. When would you NOT use switchMap?
Never use switchMap for write operations (POST, PUT, DELETE). If the user clicks "Save" twice, the first HTTP request gets cancelled — the data may never be saved. Use exhaustMap (ignore extra clicks) or concatMap (process all clicks in order) instead.
Q8. What is the difference between debounceTime and throttleTime?
debounceTime(ms): waits until no new values arrive for ms milliseconds, then emits the last value. Ideal for search inputs — waits for the user to stop typing.
throttleTime(ms): emits the first value, then ignores all values for ms milliseconds. Ideal for scroll/resize events — fires at a controlled rate regardless of how often the source emits.
