01

What is Change Detection?

Definition

Change Detection is Angular's mechanism to synchronise the component data model (TypeScript) with the DOM (View). Whenever data changes, Angular checks what needs to update and re-renders only affected parts.


  • Triggered by async events (click, HTTP, timers)
  • Walks the component tree top-down
  • Compares old vs new values (dirty check)
  • Updates the DOM if a difference is found

What triggers Change Detection?

  • 📌 DOM events — click, keyup, input, scroll
  • 📌 HTTP requests — XHR, fetch responses
  • 📌 Timers — setTimeout, setInterval
  • 📌 Promises — resolve / reject
  • 📌 WebSockets — messages received
  • 📌 Manual trigger — markForCheck, detectChanges

All of these are intercepted by Zone.js (NgZone), which tells Angular to run CD after each async task.

🔄 The CD Cycle

Async Eventclick / HTTP / timer
→
Zone.js patches itNgZone.run()
→
CD triggeredApplicationRef.tick()
→
Walk tree top-downcheck each component
→
Update DOMonly changed bindings
✅ Angular checks the ENTIRE tree by default

With Default strategy, every component from root to leaf is checked on every CD cycle — even if only one component changed data.

⚠️ Performance Problem

A large app with hundreds of components runs CD on ALL of them for every mouse click. That's why OnPush strategy and Signals exist — to skip unnecessary checks.

02

Zone.js — The Trigger of Change Detection

What is Zone.js?

Zone.js is a library that monkey-patches async browser APIs (setTimeout, Promise, XHR, addEventListener). It creates an execution context called a "zone" that intercepts async operations and notifies Angular when each completes.


When any async operation inside Angular's zone completes → NgZone calls ApplicationRef.tick() → triggers CD for the whole app.

Patched APIs

  • setTimeout / setInterval
  • Promise.then / .catch
  • XMLHttpRequest (fetch API)
  • addEventListener / removeEventListener
  • requestAnimationFrame
  • MutationObserver, IntersectionObserver

🐒 What does "Monkey-Patching" mean?

Monkey-patching means replacing or wrapping an existing function at runtime — without modifying the original source code. Zone.js does this at app startup, before your code runs.

// Original browser API
window.setTimeout = function(fn, delay) { /* native browser code */ };

// Zone.js silently REPLACES it with its own wrapped version
window.setTimeout = function(fn, delay) {
  return nativeSetTimeout(function() {
    zone.run(fn);          // ← intercepts: "async task completed!"
    notifyAngular();       // ← tells Angular → run Change Detection now
  }, delay);
};

// Your code calls setTimeout() thinking it's the real one…
// …but it's actually Zone.js's version watching in the background 🐒

🔔 Doorbell analogy: Imagine swapping a store's doorbell with your own — every time a customer enters, YOU are notified, but the store owner has no idea you swapped the bell. That's monkey-patching.

// NgZone — run code INSIDE or OUTSIDE Angular's zone
import { NgZone } from '@angular/core';

@Component({ selector: 'app-root', template: '' })
export class AppComponent {
  constructor(private ngZone: NgZone) {}

  // ✅ Run OUTSIDE Angular zone — no CD triggered
  // Good for: heavy computations, polling, third-party libs (Google Maps, Chart.js)
  startPolling() {
    this.ngZone.runOutsideAngular(() => {
      setInterval(() => {
        this.fetchData();  // does NOT trigger CD
      }, 1000);
    });
  }

  fetchData() {
    // When you NEED to update UI, re-enter the zone
    this.ngZone.run(() => {
      this.data = 'updated';  // ✅ CD triggered for this update only
    });
  }
}

// Real-world: third-party library initialisation outside Angular zone
ngAfterViewInit() {
  this.ngZone.runOutsideAngular(() => {
    const map = new google.maps.Map(this.mapEl.nativeElement, opts);
    map.addListener('click', () => {
      this.ngZone.run(() => this.onMapClick());  // re-enter only on map click
    });
  });
}
✅ runOutsideAngular Use Cases
  • Google Maps / Mapbox initialisation
  • Chart.js / D3.js animations
  • WebSocket heartbeat polling
  • Heavy canvas rendering loops
🆕 Angular 18+ — Zoneless

Angular is moving towards Zone-less change detection using Signals. Zone.js becomes optional. Components update only when signals they read change. No more global CD sweeps.

03

Default vs OnPush Strategy

Default Strategy — CheckAlways

Angular checks this component on every CD cycle regardless of whether inputs changed. It's the default for all components.


  • Checks on every event, HTTP, timer
  • Simple to use — just works
  • ❌ Expensive for large trees
  • Cannot skip checks based on input
@Component({
  // Default — no changeDetection property needed
  changeDetection: ChangeDetectionStrategy.Default
})

OnPush Strategy — CheckOnce ✅

Angular skips checking this component unless one of these conditions is met:


  • An @Input() reference changes (not mutation)
  • An event originates from this component or its children
  • markForCheck() is called manually
  • An async pipe emits a new value
  • A Signal read in the template changes (Angular 16+)
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush
})

Component Tree — Which gets checked?

Default — all components checked ✅

✅ AppComponent
  ├─ ✅ NavComponent
  ├─ ✅ ProductListComponent
  │   ├─ ✅ ProductCardComponent
  │   └─ ✅ ProductCardComponent
  └─ ✅ FooterComponent

OnPush — only dirty path checked ✅

✅ AppComponent
  ├─ ⛔ NavComponent (skipped)
  ├─ ✅ ProductListComponent
  │   ├─ ✅ ProductCard (input changed)
  │   └─ ⛔ ProductCard (skipped)
  └─ ⛔ FooterComponent (skipped)
⚠️ OnPush + Mutating Objects — Common Bug

this.products.push(item) → same array reference → OnPush won't detect change! Always create a new reference: this.products = [...this.products, item]

04

OnPush Deep Dive — When it triggers & When it doesn't

@Component({
  selector: 'app-product-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<div>{{ product.name }} — {{ product.price }}</div>`
})
export class ProductCardComponent {
  @Input() product!: Product;
}

// ── In Parent ──────────────────────────────────────────────────────────

// ❌ WRONG — mutating same object — OnPush WON'T detect change
badUpdate() {
  this.product.name = 'New Name';  // same reference! OnPush skips check
}

// ✅ CORRECT — new object reference — OnPush WILL detect change
goodUpdate() {
  this.product = { ...this.product, name: 'New Name' };  // new reference ✅
}

// ❌ WRONG — push mutates existing array reference
badAddToList() {
  this.products.push(newProduct);  // same array reference — OnPush skips
}

// ✅ CORRECT — spread creates new array reference
goodAddToList() {
  this.products = [...this.products, newProduct];  // new reference ✅
}

When OnPush DOES trigger CD

  • ✅ @Input() receives a new reference
  • ✅ Event fires inside this component (click, input)
  • ✅ markForCheck() called manually
  • ✅ async pipe emits new value
  • ✅ Signal read in template changes (Angular 16+)

When OnPush SKIPS CD

  • ❌ Parent's unrelated state changes
  • ❌ @Input() value mutated (same reference)
  • ❌ Service emits but no async pipe or markForCheck
  • ❌ setTimeout inside child with mutable update
✅ Best Practice — Use OnPush everywhere for performance

Apply ChangeDetectionStrategy.OnPush to all components — especially presentational (dumb) components that only receive @Input. Combine with immutable data patterns or NgRx which always creates new references.

ChangeDetectorRef — Manual Control

4 Manual CD methods via ChangeDetectorRef

MethodWhat it doesUse case
markForCheck()Marks component + ancestors as dirty — checked on next CD cycleOnPush + async service data
detectChanges()Runs CD immediately for this subtree — synchronousAfter manual DOM updates
detach()Removes component from CD tree — never checked automaticallyReal-time dashboards, virtual scroll
reattach()Re-adds component to CD treeAfter detach when update needed
 
import { ChangeDetectorRef, Component, OnInit } from '@angular/core';

@Component({ changeDetection: ChangeDetectionStrategy.OnPush, template: `{{ data }}` })
export class DashboardComponent implements OnInit {
  data = '';

  constructor(private cdr: ChangeDetectorRef, private ws: WebSocketService) {}

  ngOnInit() {
    // markForCheck — OnPush component + WebSocket (no async pipe)
    this.ws.messages$.subscribe(msg => {
      this.data = msg;
      this.cdr.markForCheck();  // ✅ queue for next CD cycle
    });

    // detectChanges — immediate sync update
    setTimeout(() => {
      this.data = 'updated';
      this.cdr.detectChanges();   // ✅ runs CD immediately on this subtree
    }, 2000);
  }
}

// detach / reattach — ultra-performance for high-frequency updates
@Component({ changeDetection: ChangeDetectionStrategy.OnPush, template: `{{ price }}` })
export class StockTickerComponent implements OnInit, OnDestroy {
  price = 0;

  constructor(private cdr: ChangeDetectorRef) {
    this.cdr.detach();  // ⚡ detach from CD tree — no automatic checks
  }

  ngOnInit() {
    // Update every 2 seconds instead of every millisecond tick
    setInterval(() => {
      this.price = this.getLatestPrice();
      this.cdr.detectChanges();   // ✅ manually refresh at our cadence
    }, 2000);
  }
}
✅ markForCheck vs detectChanges
  • markForCheck — schedules for the NEXT cycle (async, preferred)
  • detectChanges — runs IMMEDIATELY and synchronously (use sparingly)
⚠️ Avoid detectChanges in loops

Calling detectChanges() inside a loop or high-frequency subscription is a performance anti-pattern. Use markForCheck() or the async pipe instead.

06

async Pipe — The CD-Aware Subscription

Why async pipe is the best way to handle Observables in templates

  • Automatically subscribes to Observable/Promise
  • Automatically unsubscribes on component destroy — no memory leaks
  • Calls markForCheck() internally when value arrives — works perfectly with OnPush
  • Re-renders only when new emission comes
// ❌ BAD — manual subscribe in component
export class ProductsComponent implements OnInit, OnDestroy {
  products: Product[] = [];
  private sub: Subscription;

  ngOnInit() {
    this.sub = this.productService.products$.subscribe(p => {
      this.products = p;
      this.cdr.markForCheck();  // manually trigger CD — easy to forget!
    });
  }
  ngOnDestroy() { this.sub.unsubscribe(); }  // easy to forget!
}

// ✅ BEST — async pipe handles everything automatically
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <div *ngIf="loading$ | async">Loading...</div>
    <div *ngFor="let p of products$ | async">{{ p.name }}</div>

    <!-- Avoid multiple subscriptions for same observable -->
    <ng-container *ngIf="products$ | async as products">
      <span>{{ products.length }} items</span>
      <div *ngFor="let p of products">{{ p.name }}</div>
    </ng-container>
  `
})
export class ProductsComponent {
  products$ = this.store.select(selectAllProducts);
  loading$  = this.store.select(selectLoading);
  // ✅ no subscribe, no unsubscribe, no markForCheck needed!
}
✅ async pipe + OnPush = perfect combo

async pipe internally calls markForCheck() when the Observable emits, which queues the OnPush component for the next CD check. This is the recommended pattern for reactive Angular applications.

🆕 Signals — Angular 16+ Change Detection Revolution

🆕 What are Signals? (Angular 16+)

A Signal is a reactive primitive — a wrapper around a value that notifies Angular precisely which components need updating when the value changes. No Zone.js needed, no full-tree scan.


3 Signal primitives:

  • signal() — writable value
  • computed() — derived value (memoized)
  • effect() — side effects when signal changes

Signals vs Zone.js

Zone.jsSignals
CD triggerAny async event → full tree scanOnly affected components
GranularityComponent levelTemplate binding level
PerformanceO(tree size)O(consumers)
BoilerplateNgZone, runOutsideAngularNone
Angular versionAll versionsAngular 16+
 
import { signal, computed, effect } from '@angular/core';

@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <p>Count: {{ count() }}</p>
    <p>Double: {{ double() }}</p>
    <button (click)="increment()">+1</button>
  `
})
export class CounterComponent {

  // Writable signal
  count = signal(0);

  // Computed — auto-recalculates when count changes
  double = computed(() => this.count() * 2);

  // Effect — runs when count changes (side effects)
  _log = effect(() => console.log('Count changed:', this.count()));

  increment() {
    this.count.update(v => v + 1);  // update based on prev value
    // OR: this.count.set(5)       — set to exact value
  }
}

// Signals with OnPush — Angular updates ONLY the bindings that read changed signals
// No markForCheck() needed — Angular tracks signal reads automatically!
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<p>{{ products().length }} products</p>`
})
export class ProductsComponent {
  products = signal<Product[]>([]);

  load() {
    this.productService.getAll().subscribe(p => {
      this.products.set(p);  // ✅ no markForCheck needed! Signal notifies Angular
    });
  }
}
🆕 input() / output() / model() signals (Angular 17.1+)

input() replaces @Input() and is a read-only Signal. model() is a two-way signal. output() replaces @Output(). These integrate directly with the signal-based CD system.

🆕 Zoneless Change Detection — Angular 18+

🆕 What is Zoneless? (Angular 18 experimental, Angular 19 stable)

Angular can now run without Zone.js entirely. Instead of patching all async APIs and doing full-tree scans, CD is triggered only by Signals, markForCheck, or detectChanges. This is the future of Angular performance.

// app.config.ts — enable Zoneless (Angular 18+)
import { provideExperimentalZonelessChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideExperimentalZonelessChangeDetection(),  // ← enables Zoneless
    provideRouter(routes),
  ]
};

// angular.json — remove zone.js from polyfills
// "polyfills": []  ← remove zone.js!

// ── In Zoneless app — update via Signals ────────────────────────────
@Component({
  template: `<p>{{ name() }}</p><button (click)="update()">Change</button>`
})
export class MyComponent {
  name = signal('Nitin');

  update() {
    this.name.set('Kumar');  // ✅ only this binding updates — no global CD!
  }
}

// Or with markForCheck for non-signal use cases
@Component({ template: `{{ title }}` })
export class HeaderComponent {
  title = '';
  constructor(private cdr: ChangeDetectorRef, private svc: TitleService) {
    this.svc.title$.subscribe(t => {
      this.title = t;
      this.cdr.markForCheck();  // still works in Zoneless
    });
  }
}
🚀 Zoneless Benefits
  • Smaller bundle (no zone.js ~15KB)
  • Faster initial load
  • No patching browser APIs
  • Better SSR / SSG performance
  • Easier integration with micro-frontends

Evolution Timeline

  • Angular 2–15 — Zone.js required
  • Angular 16 — Signals introduced (dev preview)
  • Angular 17 — Signals stable, new control flow
  • Angular 18 — Zoneless experimental
  • Angular 19 — Zoneless stable ✅
✅ Migration Path
  • Start with OnPush everywhere
  • Replace BehaviorSubjects with Signals
  • Replace @Input()/@Output() with input()/output()
  • Enable Zoneless experimentally

Performance Patterns & Anti-Patterns

✅ Best Practices

  • Use OnPush for ALL components
  • Use async pipe over manual subscribe
  • Use trackBy in *ngFor
  • Use pure pipes over methods in templates
  • Use ngZone.runOutsideAngular() for non-UI work
  • Use Signals instead of class properties (Angular 16+)
  • Avoid expensive logic in getters / template expressions

❌ Anti-Patterns

  • Methods in templates: {{ getItems() }} — called on every CD
  • Impure pipes — recalculate on every CD
  • Mutating objects with OnPush
  • Manual subscribe without unsubscribe
  • detectChanges() inside loops
  • Heavy computation in ngDoCheck
  • Third-party events without runOutsideAngular
// ❌ BAD — method called on EVERY change detection cycle
export class BadComponent {
  getFilteredProducts() {  // ← called 100s of times per second!
    return this.products.filter(p => p.price > 100);
  }
}
// template: *ngFor="let p of getFilteredProducts()"  ❌

// ✅ GOOD — use pure pipe (only recalculates when input changes)
@Pipe({ name: 'filterByPrice', pure: true })  // pure: true is default
export class FilterByPricePipe implements PipeTransform {
  transform(products: Product[], minPrice: number): Product[] {
    return products.filter(p => p.price > minPrice);
  }
}
// template: *ngFor="let p of products | filterByPrice:100"  ✅

// ✅ GOOD — trackBy avoids re-rendering unchanged list items
export class ProductListComponent {
  trackByProductId(index: number, product: Product): number {
    return product.id;
  }
}
// template: *ngFor="let p of products; trackBy: trackByProductId"  ✅

// ✅ GOOD — Signal replaces method call
export class GoodComponent {
  products  = signal<Product[]>([]);
  expensive = computed(() =>
    this.products().filter(p => p.price > 100)  // only recalculates when products changes
  );
}
// template: *ngFor="let p of expensive()"  ✅

Senior-Level Interview Q&A

Q1. What is the difference between Default and OnPush change detection?
Default checks every component on every CD cycle triggered by any async event. OnPush skips a component unless: an @Input() reference changes, an event fires from that component, markForCheck() is called, an async pipe emits, or a Signal it reads changes. OnPush significantly reduces unnecessary re-renders in large trees.
Q2. What is Zone.js and what does runOutsideAngular do?
Zone.js monkey-patches async browser APIs (setTimeout, XHR, Promises) and notifies Angular's NgZone after each async task completes, triggering ApplicationRef.tick() which runs CD. runOutsideAngular() executes code without Zone.js interception — no CD is triggered. Used for: Google Maps/Chart.js init, WebSocket heartbeats, or heavy canvas animations that shouldn't trigger Angular CD.
Q3. What is the difference between markForCheck() and detectChanges()?
markForCheck() marks the component and its ancestors as dirty, queuing them for the next CD cycle (async). detectChanges() immediately runs CD synchronously on the component and its children right now. Use markForCheck() for OnPush + async subscriptions. Use detectChanges() only when you need an immediate synchronous update (e.g., after third-party library manipulation).
Q4. Why is using a method in the template a performance issue?
A method like {{ getFilteredItems() }} is called on every single CD cycle — potentially hundreds of times per second. Use a pure pipe instead (only recalculates when input reference changes) or a computed Signal (recalculates only when its dependent signals change). Both are memoized and dramatically reduce computation.
Q5. What are Angular Signals and how do they improve change detection?
Signals (Angular 16+) are reactive primitives — signal(), computed(), effect(). When a Signal's value changes, Angular knows precisely which template bindings read that signal and updates only those — no full component tree scan. This eliminates the need for Zone.js, avoids unnecessary checks, and enables truly granular updates at the binding level rather than component level.
Q6. What is Zoneless change detection in Angular 18+? How does it work?
Zoneless removes Zone.js entirely from the Angular app. CD is triggered only by: Signal value changes, markForCheck() / detectChanges() calls, or async pipe emissions. Benefits: ~15KB smaller bundle, no browser API patching, better SSR performance, easier micro-frontend integration. Enable with provideExperimentalZonelessChangeDetection(). Became stable in Angular 19. The migration path is: OnPush everywhere → adopt Signals → enable Zoneless.
