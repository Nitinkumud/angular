Strictly Typed Forms

🌍 Real World Analogy

Think of a regular form like a plain box — you can put anything in it and TypeScript won't stop you. A Typed Form is like a labelled storage box — every compartment has a name and a type. If you try to put a number where a string goes, TypeScript shows an error before you even run the app.

What are the 4 Typed Form Classes?

1

FormControl<T> — a single field. FormControl<string> means this field holds a string.

2

FormGroup<T> — a group of related fields. Like a "Personal Info" section.

3

FormArray<T> — a list of fields. Like multiple phone numbers.

4

FormRecord<T> — a map where keys are added at runtime. Like user-defined custom fields.

The Most Important Difference: .value vs .getRawValue()

PropertyWhat it returnsDisabled fields?
.valuePartial object — some fields may be missing❌ Excluded
.getRawValue()Complete object — ALL fields present✅ Included

⚠️ Always use getRawValue() when submitting a form!

🧠 How to Remember

NNFB = "Never Null Form Builder" — inject NonNullableFormBuilder and every .control() call gives you a non-null type by default. Only use new FormControl<string|null>(null) when the field genuinely needs to be empty/null.

Rule of thumb: Use NNFB for required fields. Use new FormControl<T|null>(null) for optional fields.

typed-form.ts — Simple Example
// Define what each control holds (the "shape")
interface LoginControls {
  email:    FormControl<string>;       // NEVER null
  password: FormControl<string>;       // NEVER null
  age:      FormControl<number | null>; // CAN be null
}

@Component({ standalone: true, ... })
export class LoginComponent {
  // inject NonNullableFormBuilder (NNFB)
  private fb = inject(NonNullableFormBuilder);

  form: FormGroup<LoginControls> = this.fb.group({

    // fb.control() = NonNullable by default
    email:    this.fb.control('', Validators.required),
    password: this.fb.control('', Validators.required),

    // new FormControl = explicitly nullable
    age: new FormControl<number | null>(null),
  });

  onSubmit() {
    // ✅ CORRECT — always use getRawValue()
    const data = this.form.getRawValue();
    console.log('Submit:', data);

    // ❌ RISKY — .value might have undefined fields!
    const partial = this.form.value;
    console.log('Partial:', partial);
  }
}

Async Validation & Performance

🌍 Real World Analogy

Think of async validation like a hotel receptionist checking if a room is available. You don't ask after every letter you type — you wait until you finish typing the room number (blur event). The receptionist calls the system (API), and while waiting, shows a "Checking..." spinner. Then either says "Available ✅" or "Already taken ❌".

How Async Validation Works — Step by Step

1

User types in the email field

2

User clicks away (blur). Only NOW the validator fires. (updateOn: 'blur')

3

Control status changes to PENDING — "checking in progress"

4

After 400ms wait (timer debounce), the HTTP call is made

5

If user blurs again before 400ms → previous call is cancelled by switchMap

6

Response arrives → status becomes VALID or INVALID

3 RxJS Operators You Must Know

OperatorWhat it does (plain English)
timer(400)Waits 400ms before proceeding. Acts as a debounce.
switchMap()"Cancel previous, start new" — if user blurs again, old HTTP call is discarded
first()Tells Angular "we're done now" — WITHOUT this, control stays PENDING forever!
🧠 How to Remember — "TSF Rule"
  • Timer → debounce the call
  • SwitchMap → cancel the old, start the new
  • First → complete the stream so Angular exits PENDING

If you forget F (first), your form will be stuck in 🔄 PENDING forever!

async-validator.ts
// Functional validator — gets HttpClient via inject()
export function uniqueEmailValidator(): AsyncValidatorFn {
  const http = inject(HttpClient);

  return (ctrl: AbstractControl) => {
    const email = ctrl.value as string;
    if (!email) return of(null);  // empty = no check needed

    return timer(400).pipe(     // T: wait 400ms
      switchMap(() =>            // S: cancel old, start new
        http.get<{ exists: boolean }>(
          '/api/check-email',
          { params: { email } }  // always use params object!
        )
      ),
      map(res =>
        res.exists
          ? { emailTaken: true }  // error → INVALID
          : null                    // no error → VALID
      ),
      catchError(() => of(null)),  // API down? allow submit
      first()                       // F: complete stream!
    );
  };
}

// Attach to a form control:
email: this.fb.control('', {
  validators:      [Validators.required, Validators.email],
  asyncValidators: [uniqueEmailValidator()],
  updateOn:        'blur'  // fires only on blur, not keystroke
})

Angular Forms + Signals (v22)

🌍 Real World Analogy — Think of Excel Cells

A Signal is like an Excel cell. When cell A1 changes, every formula that uses A1 automatically updates. formValue is cell A1 (the raw form data). isSearchActive is a formula like =IF(A1 > 2 chars, TRUE, FALSE) — it auto-recalculates. No subscriptions. No manual updates. Just declare the relationships.

The 4 Signal Primitives You Need

1

toSignal(observable) — converts RxJS Observable into a Signal. Now you can read it synchronously like a variable.

2

computed(() => ...) — derives a new Signal from other Signals. Auto-recalculates. Like an Excel formula.

3

linkedSignal(() => ...) — a writable Signal that resets itself when its source changes. E.g., page resets to 1 when search query changes.

4

resource() — like a Signal that does an async API call. Auto-re-fetches when its request Signals change. Auto-cancels old requests.

RxJS vs Signals — Quick Comparison

RxJS ObservableSignal (v22)
Read valueSubscribe (async)Call it: mySignal()
Update UINeeds Zone.jsAutomatic — no zone needed
Derived statepipe(map(...))computed()
CleanupManual unsubscribeAuto via DestroyRef
🧠 How to Remember — "TCLR" Pattern
  • ToSignal — bridge form's valueChanges into Signal world
  • Computed — derive stuff from that Signal
  • LinkedSignal — reset writable state when source changes
  • Resource — fetch API data reactively, auto-abort on change
search-form.component.ts — Forms + Signals
@Component({
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <input [formControl]="queryCtrl" />
    @if (isActive()) {
      <p>Searching for: {{ query() }}</p>
      <p>Page: {{ page() }}</p>
    }
    @if (results.isLoading()) { <span>Loading...</span> }
    @for (item of results.value() ?? []; track item.id) {
      <div>{{ item.name }}</div>
    }
  `
})
export class SearchComponent {
  private fb = inject(NonNullableFormBuilder);

  // Normal reactive form control
  queryCtrl = this.fb.control('');

  // STEP T: Bridge form → Signal via toSignal()
  readonly formValue = toSignal(
    this.queryCtrl.valueChanges.pipe(debounceTime(300)),
    { initialValue: '' }
  );

  // STEP C: Computed signals (like Excel formulas)
  readonly query    = computed(() => this.formValue().trim());
  readonly isActive = computed(() => this.query().length >= 2);

  // STEP L: linkedSignal resets page to 1 on new query
  readonly page = linkedSignal(() => (this.query(), 1));

  // STEP R: resource() — reactive API call
  readonly results = resource({
    request: () => ({
      q: this.query(),
      p: this.page()
    }),
    loader: async ({ request, abortSignal }) => {
      if (!request.q) return [];
      const r = await fetch(
        `/api/search?q=${request.q}&p=${request.p}`,
        { signal: abortSignal }  // auto-cancels stale requests
      );
      return r.json();
    }
  });
}

Custom Form Controls — ControlValueAccessor

🌍 Real World Analogy — Power Adapter

Angular's form system speaks one language. Your custom UI (like a tag-input or star-rating) speaks another. ControlValueAccessor (CVA) is the power adapter between them. You implement 4 methods that tell Angular: "here's how to read my value, write my value, disable me, and know when I'm touched."

The 4 Methods You Must Implement

1

writeValue(val) — Angular → Your Component.
Called when parent does form.setValue(). Update your internal UI.

2

registerOnChange(fn) — Store the callback.
Call fn(newValue) whenever the user changes something in your component.

3

registerOnTouched(fn) — Store the callback.
Call fn() when user leaves (blurs) your component.

4

setDisabledState(isDisabled) — Called when parent form is disabled.
Show/hide your disabled UI state.

📋 How to Register a CVA — Copy This Pattern

In providers array of your @Component:

{
  provide:     NG_VALUE_ACCESSOR,
  useExisting: forwardRef(() => YourComponent),
  multi:       true
}

Why forwardRef? The class isn't defined yet when the decorator runs. forwardRef says "look this up later at runtime."

🧠 How to Remember — "WRST"
  • WriteValue — Angular writes INTO your component
  • RegisterOnChange — you notify Angular when value changes
  • SetDisabledState — Angular disables your component
  • Touched (registerOnTouched) — you tell Angular "user left"

Direction: W & S = Angular → You  |  R & T = You → Angular

tag-input.component.ts — Custom Multi-Tag Input
@Component({
  selector: 'app-tag-input', standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  providers: [{            // Register as a form control
    provide:     NG_VALUE_ACCESSOR,
    useExisting: forwardRef(() => TagInputComponent),
    multi:       true
  }],
  template: `
    <div>
      @for (tag of tags(); track tag) {
        <span>{{ tag }}
          <button (click)="remove(tag)">×</button>
        </span>
      }
      <input [(ngModel)]="inputVal"
             (keydown.enter)="add($event)"
             (blur)="onTouched()"
             [disabled]="isDisabled()" />
    </div>
  `
})
export class TagInputComponent
  implements ControlValueAccessor {

  // Internal state as Signals
  tags       = signal<string[]>([]);
  isDisabled = signal(false);
  inputVal   = '';

  // CVA callbacks — Angular fills these in
  private _onChange = (_: string[]) => {};
  onTouched = () => {};

  // 1️⃣  W — writeValue: Angular → our component
  writeValue(v: string[] | null): void {
    this.tags.set([...(v ?? [])]);
  }

  // 2️⃣  R — registerOnChange: store parent's listener
  registerOnChange(fn: any): void { this._onChange = fn; }

  // 3️⃣  T — registerOnTouched: store touched listener
  registerOnTouched(fn: any): void { this.onTouched = fn; }

  // 4️⃣  S — setDisabledState: parent disables → update UI
  setDisabledState(d: boolean): void { this.isDisabled.set(d); }

  add(e: Event): void {
    e.preventDefault();
    const tag = this.inputVal.trim();
    if (!tag || this.isDisabled()) return;
    this.tags.update(t => [...t, tag]);
    this.inputVal = '';
    this._onChange(this.tags()); // tell parent new value
  }

  remove(tag: string): void {
    this.tags.update(t => t.filter(x => x !== tag));
    this._onChange(this.tags());
  }
}

Dynamic Form Generation

🌍 Real World Analogy — Restaurant Menu Builder

Imagine a restaurant app where the chef (backend) sends a recipe list (JSON schema). The waiter (FormFactory) reads the recipe and builds the form automatically. The customer (user) sees the built form and fills it in — without the developer manually creating each field in code. When the recipe changes, the form updates.

How Dynamic Forms Work — 3 Steps

1

Define a Schema — a simple JS object/array describing each field: its key, type, label, and validators.

2

Pass schema to FormFactory — the factory loops through the schema and creates the correct FormControl / FormGroup / FormArray.

3

Render dynamically — your template uses @for to loop over schema items and render the correct input for each field type.

🧠 Key Performance Tips to Remember
  • track field.key on @for — prevents Angular recreating DOM when schema re-renders
  • emitEvent: false on bulk patchValue() — avoids 50 events when filling 50 fields at once
  • @defer (on viewport) — don't load date-pickers until they scroll into view
  • OnPush on every dynamic field component — only re-renders when its input changes
form-factory.ts + dynamic-form.component.ts
// STEP 1: Schema — what fields exist
interface FieldSchema {
  key:         string;    // e.g. 'email'
  type:        string;    // 'text' | 'email' | 'group'
  label:       string;    // shown as <label>
  required?:   boolean;
  defaultValue?: unknown;
  children?:   FieldSchema[];  // for 'group' type
}

// STEP 2: FormFactory — builds FormGroup from schema
@Injectable({ providedIn: 'root' })
export class FormFactory {
  private fb = inject(NonNullableFormBuilder);

  buildGroup(schema: FieldSchema[]): FormGroup {
    const controls: Record<string, AbstractControl> = {};

    for (const field of schema) {
      const validators = field.required
        ? [Validators.required]
        : [];

      if (field.type === 'group') {
        // Nested group — recurse into children
        controls[field.key] =
          this.buildGroup(field.children ?? []);
      } else {
        controls[field.key] =
          new FormControl(field.defaultValue ?? null, validators);
      }
    }
    return this.fb.group(controls);
  }
}

// STEP 3: Component renders the generated form
@Component({
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <form [formGroup]="form" (ngSubmit)="submit()">
      @for (field of schema; track field.key) {
        <label>{{ field.label }}</label>
        <input [formControlName]="field.key"
               [type]="field.type" />
      }
      <button [disabled]="form.invalid">Submit</button>
    </form>
  `
})
export class DynamicFormComponent {
  private factory = inject(FormFactory);

  readonly schema: FieldSchema[] = [
    { key: 'name',  type: 'text',  label: 'Full Name', required: true  },
    { key: 'email', type: 'email', label: 'Email',    required: true  },
    { key: 'age',   type: 'number',label: 'Age',     required: false },
  ];

  readonly form = this.factory.buildGroup(this.schema);

  submit() {
    console.log(this.form.getRawValue());
  }
}

Quick Revision — All 5 Topics at a Glance

1. Typed Forms

  • Use NNFB for required fields
  • Use new FormControl<T|null>(null) for optional
  • Always submit with .getRawValue()
  • FormRecord = dynamic runtime keys
  • .value = partial, may miss disabled fields
form.getRawValue()
// ALL fields, correct types ✅

2. Async Validation

  • Use updateOn: 'blur' — no API on keystroke
  • TSF Rule: Timer → SwitchMap → First
  • Forget first() = stuck in PENDING forever ⚠️
  • Expose PENDING via toSignal(statusChanges)
timer(400) → switchMap
→ map → catchError → first()

3. Signals + Forms (v22)

  • TCLR: ToSignal → Computed → Linked → Resource
  • toSignal(valueChanges) bridges RxJS → Signal
  • computed() = automatic, lazy derived state
  • linkedSignal() = writable, auto-resets
  • resource() = reactive fetch + auto-abort

4. CVA — Custom Controls

  • WRST: WriteValue, Register, SetDisabled, Touched
  • W & S = Angular → Your Component
  • R & T = Your Component → Angular
  • Register via NG_VALUE_ACCESSOR + forwardRef
  • Add NG_VALIDATORS for form-level validation

5. Dynamic Forms

  • Schema → FormFactory → FormGroup
  • Use track field.key for DOM performance
  • Bulk update: patchValue + emitEvent:false
  • Call updateValueAndValidity() once after batch
  • @defer for heavy controls (date pickers)
