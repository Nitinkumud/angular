CSR vs SSR vs SSG — Core Concepts

🖥 CSR — Client-Side Rendering

Default Angular behaviour. Server sends a near-empty index.html. Browser downloads JS, executes it, and builds the DOM entirely on the client.


  • First paint: blank screen until JS loads
  • Bad for SEO (crawler sees empty HTML)
  • Good for apps behind login (no SEO needed)
  • Fast navigation after initial load

🌐 SSR — Server-Side Rendering

Angular SSR (v22). Server renders the full HTML for every request and sends it to the browser. Browser displays content immediately, then hydrates.


  • First paint: full HTML — instant visibility
  • Great for SEO & social media crawlers
  • Best for dynamic, frequently changing pages
  • Requires a running Node.js server

📄 SSG — Static Site Generation (Pre-rendering)

Build-time rendering. HTML is generated once at ng build and served as static files — no server needed at runtime.


  • Fastest possible delivery (CDN-cacheable)
  • Great for content that rarely changes
  • Best for blogs, docs, marketing pages
  • Cannot handle dynamic user-specific data

Using app.routes.server.ts you can configure each route individually: home page as Prerender, product page as Server (dynamic), admin page as Client (no SSR needed). This is the most powerful feature of Angular SSR v19+.

What Changed — Angular Universal → Angular SSR

Angular Universal was the old name for Angular's SSR solution. It was renamed and rebuilt as @angular/ssr starting Angular v17. Everything below marked ❌ is deprecated or removed.

❌ OLD — Angular Universal (deprecated / removed)
  • Package: @nguniversal/express-engine
  • Setup: ng add @nguniversal/express-engine
  • Class: AppServerModule — NgModule-based server module
  • ServerModule from @angular/platform-server
  • BrowserModule.withServerTransition({ appId })
  • renderModule() / renderModuleFactory()
  • Two separate main.ts files (browser + server)
  • Manual ngExpressEngine setup in server.ts
✅ NEW — @angular/ssr (Angular v17+ / v22)
  • Package: @angular/ssr
  • Setup: ng add @angular/ssr
  • Standalone API: provideServerRendering()
  • app.config.server.ts — server-specific providers
  • app.routes.server.ts — per-route render mode (v19+)
  • RenderMode.Server / RenderMode.Prerender / RenderMode.Client
  • Single unified server.ts with Express
  • Non-destructive hydration built-in: provideClientHydration()
// ── OLD: AppServerModule (Angular Universal — DEPRECATED ❌) ────────────────
@NgModule({
  imports: [AppModule, ServerModule],  // ❌ ServerModule is removed
})
export class AppServerModule {}         // ❌ Not needed anymore

// ── NEW: app.config.server.ts (Angular v22 ✅) ──────────────────────────────
import { mergeApplicationConfig, ApplicationConfig } from '@angular/core';
import { provideServerRendering } from '@angular/platform-server';
import { provideServerRoutesConfig } from '@angular/ssr';
import { serverRoutes } from './app.routes.server';
import { appConfig } from './app.config';

const serverConfig: ApplicationConfig = {
  providers: [
    provideServerRendering(),              // ✅ enables SSR
    provideServerRoutesConfig(serverRoutes) // ✅ per-route render modes
  ]
};

export const config = mergeApplicationConfig(appConfig, serverConfig);

// ── server.ts — Express server (generated by CLI) ───────────────────────────
import { APP_BASE_HREF } from '@angular/common';
import { CommonEngine }  from '@angular/ssr/node';
import express from 'express';
import { fileURLToPath } from 'node:url';
import { dirname, join, resolve } from 'node:path';
import bootstrap from './src/main.server';

const serverDistFolder = dirname(fileURLToPath(import.meta.url));
const browserDistFolder = resolve(serverDistFolder, '../browser');
const app = express();
const engine = new CommonEngine();

// Serve static browser assets
app.use(express.static(browserDistFolder));

// All other routes → render with Angular SSR
app.get('*', (req, res, next) => {
  engine.render({
    bootstrap,
    documentFilePath: join(browserDistFolder, 'index.html'),
    url: req.originalUrl,
    publicPath: browserDistFolder,
    providers: [{ provide: APP_BASE_HREF, useValue: req.baseUrl }]
  }).then(html => res.send(html))
    .catch(next);
});

RenderMode Per Route — Angular v19+ / v22

The biggest Angular SSR feature since v19. Each route can have its own rendering strategy — Server, Prerender, or Client — all in one app.

RenderMode.Prerender

  • HTML generated at build time
  • Served as static file — fastest delivery
  • Best for: home, about, blog listing, product catalogue
  • No Node.js needed for this route

RenderMode.Server

  • HTML generated on every request
  • Can access request headers, cookies, session
  • Best for: product detail (real-time price/stock), user profile
  • Requires running Node.js server

RenderMode.Client

  • Standard CSR — no SSR for this route
  • Browser renders entirely client-side
  • Best for: admin panels, dashboards behind auth
  • No SEO needed, heavy interactivity
// ── app.routes.server.ts — per-route render mode (Angular v19+ / v22) ───────
import { RenderMode, ServerRoute } from '@angular/ssr';

export const serverRoutes: ServerRoute[] = [
  {
    path: '',                              // home page
    renderMode: RenderMode.Prerender      // ✅ static HTML at build time
  },
  {
    path: 'products',
    renderMode: RenderMode.Prerender      // product listing — static
  },
  {
    path: 'products/:id',
    renderMode: RenderMode.Server,        // dynamic — real-time stock/price
    // fallback if route not prerendered:
    fallback: PrerenderFallback.Server
  },
  {
    path: 'admin',
    renderMode: RenderMode.Client         // CSR — behind auth, no SEO needed
  },
  {
    path: '**',                             // catch-all wildcard
    renderMode: RenderMode.Prerender,
    fallback: PrerenderFallback.Client    // unknown routes fall back to CSR
  }
];

// ── Prerendering dynamic routes — provide all IDs ────────────────────────────
{
  path: 'blog/:slug',
  renderMode: RenderMode.Prerender,
  async getPrerenderParams() {
    // fetch slugs from CMS/API at build time
    const posts = await fetch('https://api.myblog.com/posts').then(r => r.json());
    return posts.map((p: any) => ({ slug: p.slug }));
  }
}
✅ Senior Pattern — Hybrid Rendering Strategy

In a real e-commerce app: Home + Category = Prerender (static, CDN-cached), Product Detail = Server (real-time price/stock), Checkout + Dashboard = Client (auth-protected, no SSR needed). One app, three strategies, maximum performance.

❌ Old Hydration (Destructive)

After the Angular client bundle loaded, it would completely throw away the server-rendered DOM and rebuild everything from scratch. This caused a visible flicker, CLS (layout shift), and wasted the SSR effort.

✅ Non-Destructive Hydration (Angular 16+ / v22)

Angular reuses the server-rendered DOM. It only attaches event listeners and fills in any missing pieces. No flicker, no CLS, no wasted server work. Enabled by provideClientHydration().

// app.config.ts — enable non-destructive hydration ✅
import { provideClientHydration, withEventReplay } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideClientHydration(
      withEventReplay()   // ✅ captures events during hydration gap
    )
  ]
};

// withEventReplay() — what problem does it solve?
// Gap between: server HTML displayed → Angular client JS loads
// If user clicks during this gap, the event is LOST without withEventReplay()
// withEventReplay() queues events and replays them once Angular is ready ✅
✅ Incremental Hydration (@defer)

In Angular v17+ / v22, parts of the page wrapped in @defer blocks are hydrated only when needed (on viewport, on interaction, etc.) — not all at once. This reduces TBT dramatically for content-heavy pages.

How Pre-rendering Works

  • During ng build, Angular boots the SSR engine once per route
  • Renders each route to a static index.html file
  • No server needed at runtime — serve with any CDN/static host
  • For dynamic routes, getPrerenderParams() provides all parameter combinations
// Build output with pre-rendering:
dist/my-app/browser/
  index.html              // /  (home)
  products/
    index.html            // /products
  products/1/
    index.html            // /products/1
  products/2/
    index.html            // /products/2
  blog/
    my-first-post/
      index.html          // /blog/my-first-post
⚠️ Pre-rendering Limitation

Pre-rendered pages are static — they cannot show user-specific data (logged-in user name, cart count). Use RenderMode.Server for those routes, or use client-side data fetching after hydration for user-specific parts.

Application Shell

What is Application Shell?

A minimal version of your app (header, navigation, skeleton layout) that is pre-rendered and served instantly, while the rest of the content loads asynchronously. Defined as a separate Angular route rendered at build time.


It is the visible UI wrapper of your app that appears before the actual route content loads. Think of it as the skeleton that's always there.

Why use it?

  • User sees meaningful content (nav, header) instantly — reduces perceived loading time
  • Avoids blank white screen during JS download
  • Excellent for PWA (Progressive Web App) scenarios
  • Shell is cached in Service Worker — available offline
  • Lighthouse scores improve significantly
// ── Step 1: Generate App Shell via CLI ──────────────────────────────────────
ng generate app-shell

// ── Step 2: app-shell.component.ts — minimal shell UI ───────────────────────
@Component({
  standalone: true,
  imports: [RouterLink],
  template: `
    <header>
      <nav>
        <a routerLink="/">Home</a>
        <a routerLink="/products">Products</a>
      </nav>
    </header>
    <main>
      <!-- Skeleton loader shown during content load -->
      <div class="skeleton-hero"></div>
      <div class="skeleton-cards"></div>
    </main>
  `
})
export class AppShellComponent {}

// ── Step 3: app.routes.server.ts — shell route config ───────────────────────
export const serverRoutes: ServerRoute[] = [
  {
    path: 'shell',                       // special internal route
    renderMode: RenderMode.AppShell    // ✅ renders this as the shell
  },
  // ... other routes
];

// ── angular.json — configure app-shell builder ──────────────────────────────
"app-shell": {
  "builder": "@angular-devkit/build-angular:app-shell",
  "options": {
    "route": "shell"   // the route that renders the shell HTML
  }
}
✅ App Shell = Instant Perceived Performance

App Shell is pre-rendered once at build time. On every visit the browser gets the shell HTML immediately from Service Worker cache — even on slow networks. Route content then loads into the <router-outlet>.

⚠️ Keep Shell Minimal

Shell should contain only structural elements (nav, header, skeleton). Do NOT include any data-dependent components. If the shell is too heavy, it defeats the purpose.

TransferState API — Prevent Double HTTP Requests

The Problem — Double HTTP Request

Without TransferState: server renders a page, makes an API call to fetch products, embeds the data in HTML. Then the browser gets the HTML, Angular hydrates, makes the same API call again — wasting a network request and causing a re-render flash.

❌ Without TransferState

Server: GET /api/products → renders HTML
↓ HTML sent to browser
Browser: Angular hydrates
Browser: GET /api/products AGAIN ❌
Page re-renders with same data 😤

✅ With TransferState

Server: GET /api/products → renders HTML
Server: embeds data in <script> tag in HTML
↓ HTML + embedded data sent to browser
Browser: Angular hydrates
Browser: reads data from <script> tag — no HTTP ✅
// ── Angular v22 — HttpClient + TransferState is AUTOMATIC ✅ ────────────────
// Just add withHttpTransferCache() — no manual TransferState code needed!

// app.config.ts
import { provideHttpClient, withFetch } from '@angular/common/http';
import { provideClientHydration, withHttpTransferCache } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(withFetch()),   // use fetch() instead of XMLHttpRequest
    provideClientHydration(
      withHttpTransferCache()           // ✅ auto cache + transfer HTTP responses
    )
  ]
};

// ── Manual TransferState (when you need custom data transfer) ───────────────
import { TransferState, makeStateKey } from '@angular/core';

const PRODUCTS_KEY = makeStateKey<Product[]>('products'); // typed key

@Injectable({ providedIn: 'root' })
export class ProductService {
  private state  = inject(TransferState);
  private http   = inject(HttpClient);
  private isPlatformBrowser = isPlatformBrowser(inject(PLATFORM_ID));

  getProducts(): Observable<Product[]> {
    if (this.isPlatformBrowser && this.state.hasKey(PRODUCTS_KEY)) {
      const data = this.state.get(PRODUCTS_KEY, []);
      this.state.remove(PRODUCTS_KEY);   // clean up after use
      return of(data);                   // return cached — no HTTP ✅
    }
    return this.http.get<Product[]>('/api/products').pipe(
      tap(data => this.state.set(PRODUCTS_KEY, data))  // store on server
    );
  }
}

SEO & Social Media Optimization

Why SSR is needed for SEO

Search engine crawlers (Googlebot, Bingbot) and social media bots (Facebook, Twitter/X, LinkedIn) read the initial HTML. With CSR they see an empty <app-root></app-root>. With SSR they see full content + meta tags.

Angular Services for SEO

  • Title service — sets <title> tag
  • Meta service — sets <meta> tags
  • Both work on server AND client side
  • Set them in component ngOnInit or resolver
// ── product-detail.component.ts — complete SEO setup ────────────────────────
import { Component, inject, signal } from '@angular/core';
import { Meta, Title } from '@angular/platform-browser';

@Component({ standalone: true, ... })
export class ProductDetailComponent {
  private title = inject(Title);
  private meta  = inject(Meta);

  ngOnInit() {
    const product = this.product(); // signal or resolved data

    // 1. Page title (shown in browser tab + Google results)
    this.title.setTitle(`${product.name} | MyShop`);

    // 2. Basic SEO meta tags
    this.meta.updateTag({ name: 'description', content: product.description });
    this.meta.updateTag({ name: 'keywords',    content: product.tags.join(', ') });

    // 3. Open Graph — controls how page looks when shared on Facebook/LinkedIn
    this.meta.updateTag({ property: 'og:title',       content: product.name });
    this.meta.updateTag({ property: 'og:description', content: product.description });
    this.meta.updateTag({ property: 'og:image',       content: product.imageUrl });
    this.meta.updateTag({ property: 'og:url',         content: `https://myshop.com/products/${product.id}` });
    this.meta.updateTag({ property: 'og:type',        content: 'product' });

    // 4. Twitter Card — controls Twitter/X preview
    this.meta.updateTag({ name: 'twitter:card',  content: 'summary_large_image' });
    this.meta.updateTag({ name: 'twitter:title', content: product.name });
    this.meta.updateTag({ name: 'twitter:image', content: product.imageUrl });
  }
}

Pitfalls — Browser-Only APIs & Common Mistakes

⚠️ The Core Problem

Angular SSR runs your code in Node.js — there is no window, no document, no localStorage, no navigator. Any code that uses these browser-only globals will crash the server with ReferenceError: window is not defined.

// ── WRONG — crashes on server ❌ ──────────────────────────────────────────
export class MyComponent {
  ngOnInit() {
    window.scrollTo(0, 0);            // ❌ window doesn't exist in Node
    localStorage.setItem('x', '1');   // ❌ localStorage is undefined
    document.getElementById('app');   // ❌ document doesn't exist in Node
  }
}

// ── FIX 1: isPlatformBrowser() guard ✅ ─────────────────────────────────────
import { isPlatformBrowser } from '@angular/common';
import { PLATFORM_ID, inject } from '@angular/core';

export class MyComponent {
  private platformId = inject(PLATFORM_ID);

  ngOnInit() {
    if (isPlatformBrowser(this.platformId)) {
      window.scrollTo(0, 0);          // ✅ only runs in browser
      localStorage.setItem('x', '1'); // ✅
    }
  }
}

// ── FIX 2: Inject DOCUMENT token (safe DOM access) ✅ ──────────────────────
import { DOCUMENT } from '@angular/common';

export class MyComponent {
  private doc = inject(DOCUMENT);  // SSR-safe — Angular provides a mock doc

  scrollToTop() {
    this.doc.defaultView?.scrollTo(0, 0); // null-safe ✅
  }
}

// ── FIX 3: afterNextRender / afterRender (Angular 16+) ✅ ──────────────────
import { afterNextRender } from '@angular/core';

export class ChartComponent {
  constructor() {
    afterNextRender(() => {
      // This ONLY runs in the browser, NEVER on server ✅
      new Chart(document.getElementById('myChart'), { ... });
    });
  }
}
⚠️ Common Pitfall: setTimeout / setInterval

Timers left running on the server prevent the render from completing. Use isPlatformBrowser guard or afterNextRender for any timer logic.

✅ afterNextRender — Best Practice for DOM Access

Prefer afterNextRender() over isPlatformBrowser for DOM manipulation. It's cleaner, explicitly signals "browser-only", and works correctly with SSR + hydration.

Senior Interview Q&A — Part 1

Q1. What is the difference between SSR, SSG, and CSR in Angular? CONCEPT
CSR: browser downloads JS and builds DOM client-side — fast navigation but slow first paint & bad SEO. SSR: Node.js renders full HTML per request — instant first paint, dynamic data, SEO-friendly, requires a server. SSG: HTML pre-generated at build time, served as static files — fastest delivery but no dynamic data. Angular v22 lets you mix all three per route using RenderMode in app.routes.server.ts.
Q2. What replaced Angular Universal in Angular v17+? MIGRATION
The @nguniversal/express-engine package was replaced by @angular/ssr. The AppServerModule / ServerModule class-based approach was replaced by the standalone API: provideServerRendering() in app.config.server.ts. The CLI command changed from ng add @nguniversal/express-engine to ng add @angular/ssr.
Q3. What is non-destructive hydration and why does it matter? HYDRATION
Old hydration was destructive — Angular would discard the server-rendered DOM and rebuild it, causing a visible flicker and CLS. Non-destructive hydration (Angular 16+, enabled via provideClientHydration()) reuses the existing server DOM and only attaches event listeners. This eliminates the flicker, improves CLS, and avoids wasted server work. withEventReplay() additionally captures user events during the hydration gap so they aren't lost.
Q4. What is the TransferState API and when do you use it? PERFORMANCE
TransferState prevents double HTTP requests. When Angular SSR renders a page on the server it makes API calls. Without TransferState, the browser makes those same calls again during hydration. TransferState serialises the server API responses into a <script type="application/json"> tag in the HTML. The browser Angular app reads this data instead of re-fetching. In Angular v22 with withHttpTransferCache(), this is automatic for all HttpClient requests — no manual code needed.
Q5. How do you handle window is not defined errors in Angular SSR? PITFALL
Angular SSR runs in Node.js where window, document, localStorage don't exist. Three approaches: (1) isPlatformBrowser(inject(PLATFORM_ID)) guard — run browser-only code conditionally. (2) Inject the DOCUMENT token instead of using document directly — Angular provides a server-safe mock. (3) Use afterNextRender() for DOM manipulation — this hook only fires in the browser, never on the server.
Q6. What is the Application Shell and how does it improve performance? APP SHELL
App Shell is a minimal pre-rendered version of the app (navigation, header, skeleton UI) that is generated once at build time and cached in the Service Worker. On every visit, the user sees the shell instantly from cache — even offline. The actual route content then loads into <router-outlet>. It improves perceived loading time and Lighthouse scores by ensuring there's never a blank white screen.

Senior Interview Q&A — Part 2

Q7. What is RenderMode in Angular v22 and how do you configure it? ANGULAR V22
Introduced in Angular v19, RenderMode lets you assign a rendering strategy per route in app.routes.server.ts. RenderMode.Prerender generates static HTML at build time. RenderMode.Server renders on every request. RenderMode.Client skips SSR entirely for that route. This allows a hybrid strategy — e.g. home as Prerender, product detail as Server, admin as Client — in a single application.
Q8. How do you pre-render dynamic routes like /products/:id? PRERENDER
Use getPrerenderParams() on the server route config. This is an async function that returns an array of parameter objects. During ng build, Angular calls this function, fetches all product IDs (e.g. from your API), and generates a separate HTML file for each combination. Set a fallback: PrerenderFallback.Server or .Client for IDs not included at build time.
Q9. What does withEventReplay() solve? HYDRATION
There's a gap between when the server-rendered HTML is displayed and when Angular's client JS finishes loading and hydrating. If the user interacts (clicks, types) during this gap, those events are normally lost. withEventReplay() captures and queues all events during this gap using a tiny inline script, then replays them once Angular is fully hydrated — ensuring no user interaction is ever lost.
Q10. Why use provideHttpClient(withFetch()) in an SSR app? PERFORMANCE
By default, Angular's HttpClient uses XMLHttpRequest. In Node.js (SSR), XHR is not native and requires a polyfill. withFetch() switches HttpClient to use the native fetch() API which is natively available in both Node.js 18+ and browsers. This removes the XHR polyfill, reduces bundle size, and improves server-side HTTP performance.
Q11. How does SSR impact Angular's change detection? ADVANCED
On the server, Angular runs a single synchronous render cycle — there's no zone.js-driven change detection loop (since the server doesn't need to respond to user events). After the initial render stabilises (all async operations complete), Angular serialises the DOM to HTML. This is why you must ensure all server-side HTTP calls complete before render — use resolvers or toSignal with initialValue. Infinite observables or unresolved promises will cause the server to hang.
Q12. How do you deploy an Angular SSR application? DEPLOYMENT
ng build generates two folders: dist/browser/ (static assets) and dist/server/ (Node.js bundle). Deploy as: (1) Node.js server — run node dist/server/server.mjs on any Node hosting (Cloud Run, App Engine, EC2, Render). (2) Firebase Hosting + Cloud Functions — Angular CLI has @angular/fire integration. (3) Netlify / Vercel SSR — via their Edge functions adapters. Pre-rendered routes can be served from CDN without the Node server.

Senior Interview Q&A — Part 3

Q13. What is the difference between afterNextRender and ngAfterViewInit in SSR context? LIFECYCLE
ngAfterViewInit runs on both server and browser — so DOM access inside it will crash the server. afterNextRender() is guaranteed to run only in the browser, after the first render. It's the safe replacement for any DOM manipulation code you used to put in ngAfterViewInit. Use it for initialising third-party DOM libraries (charts, maps, editors).
Q14. How does SSR affect Zone.js and signals? ZONELESS
Zone.js on the server tracks all pending async tasks. Angular waits for the zone to become stable (no pending tasks) before serialising the HTML. This means all HttpClient calls, timers, and promises must resolve before render completes. With signals + zoneless architecture (Angular v18+), the stability check is based on signal reactivity graph instead, making SSR faster and more predictable. Infinite observables prevent render from ever stabilising — always complete them on the server.
Q15. A colleague says they get duplicate API calls even with withHttpTransferCache(). What could be wrong? DEBUGGING
Possible causes: (1) Using XMLHttpRequest directly instead of HttpClient — TransferCache only works with Angular's HttpClient. (2) withFetch() not added — some versions require it. (3) The HTTP call is inside a service with different injection context on server vs client, creating a cache-key mismatch. (4) The request uses POST — TransferCache only works for GET requests. (5) provideClientHydration(withHttpTransferCache()) missing from app.config.ts.
