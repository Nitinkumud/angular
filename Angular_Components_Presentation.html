Anatomy of a Component

Three Required Parts

  • TypeScript class — logic, state, methods
  • HTML template — what renders in the DOM
  • CSS selector — how it's used in other templates

Standalone (Default v19+)

Every component is standalone by default since Angular v19. No NgModule needed. Just add it to the imports array of the parent component.

Component Tree

Your app is a tree of components. Each component manages its own view. Parent-child relationships define data flow and DI scope.

// ── Minimal component (Angular v22) ─────────────────────────────────────────
import { Component, signal, computed } from '@angular/core';

@Component({
  selector: 'app-user-card',          // used as <app-user-card /> in templates

  // Inline template (or use templateUrl: './user-card.component.html')
  template: `
    <div class="card">
      <h2>{{ fullName() }}</h2>
      <p>Age: {{ age() }}</p>
      <button (click)="birthday()">🎂 Birthday</button>
    </div>
  `,

  // Inline styles (or use styleUrl: './user-card.component.css')
  styles: `
    .card { background: #1a1a2e; padding: 16px; border-radius: 8px; }
    h2    { color: #58a6ff; }
  `
})
export class UserCardComponent {
  // State as signals
  firstName = signal('Nitin');
  lastName  = signal('Kumar');
  age       = signal(30);

  // Derived state via computed
  fullName = computed(() => `${this.firstName()} ${this.lastName()}`);

  birthday() { this.age.update(a => a + 1); }
}

import { Component, OnInit, OnDestroy, OnChanges,
         AfterViewInit, SimpleChanges, input } from '@angular/core';

@Component({ selector: 'app-demo', template: '...' })
export class DemoComponent implements OnInit, OnChanges, AfterViewInit, OnDestroy {
  userId = input<number>();

  // 1. Called ONCE — before ngOnInit. Avoid heavy logic here.
  constructor() { console.log('1. constructor'); }

  // 2. Called when @Input / input() signal value changes (decorator-based only)
  ngOnChanges(changes: SimpleChanges) {
    console.log('2. ngOnChanges', changes);
  }

  // 3. Called ONCE after first ngOnChanges — best place for init logic
  ngOnInit() { console.log('3. ngOnInit — fetch data here'); }

  // 4. Called after Angular checks child views — DOM is ready
  ngAfterViewInit() { console.log('4. ngAfterViewInit — DOM ready'); }

  // 5. Cleanup — unsubscribe, cancel timers, detach listeners
  ngOnDestroy() { console.log('5. ngOnDestroy — cleanup'); }
}

⚠️ ngOnChanges + signal inputs

ngOnChanges only fires for decorator-based @Input() properties. Signal-based input() changes are tracked reactively via effect() or computed() — not via ngOnChanges.

✅ New: afterNextRender / afterRender

Angular v16+ adds afterNextRender() (runs once after first render) and afterRender() (runs after every render) as SSR-safe alternatives to ngAfterViewInit.

03

Signal-based Inputs — input() Function (v22)

🤔 What is a Signal?

A signal is a special wrapper around a value that Angular can track. When the value changes, Angular knows exactly which parts of the UI to update — without checking the entire component tree.


You read a signal by calling it as a function: mySignal()
You write a writable signal with .set() or .update()


An InputSignal is a read-only signal — the parent sets the value, the child only reads it.

🤔 Why input() instead of a plain property?

Before signals, inputs were plain properties. Angular had no way to know when they changed — it had to check every property on every change detection cycle (expensive).


With input(), Angular knows exactly which signal changed. Only the parts that depend on that signal re-render. This makes apps faster and enables zoneless apps.


Also works directly in computed() and effect() without any extra boilerplate.

📖 Step-by-step — What happens when parent passes a value to child

// 1️⃣  Child declares an input signal
export class GreetComponent {
  name = input('Guest');
  //   ↑ default value = 'Guest'
  //   ↑ TypeScript infers: InputSignal<string>

  // 2️⃣  Read it in template by calling ()
  //     template: <h1>Hello, {{ name() }}!</h1>
}

// 3️⃣  Parent binds a value to it
//  <app-greet [name]="'Nitin'" />
//             ↑ square brackets = property binding

// 4️⃣  Angular sets the signal's value to 'Nitin'
// 5️⃣  Template re-renders: Hello, Nitin!
// 6️⃣  If parent changes it → signal updates → UI updates ✅
▶ No binding — uses default
<app-greet />
→ Hello, Guest!
▶ With binding
<app-greet [name]="'Nitin'" />
→ Hello, Nitin!
▶ With dynamic value
<app-greet [name]="currentUser.name" />
→ Hello, {currentUser.name}!
  (auto-updates when parent changes)
✅ Key Rule — Why call it with ()?

name is not the value — it is the signal container. name() calls the signal to get the current value inside it. Think of it like a function that always returns the latest value. This is how Angular tracks which signals are being used in a template or computed expression.

⚠️ Read-only — child cannot write to it

An InputSignal is read-only. The child can only read name(). Only the parent can change it via [name]="newValue". If you need the child to also write back, use model() instead (Slide 7).

computed() with input()

computed() creates a derived value that automatically recalculates whenever the signals it reads change. It's like a formula in a spreadsheet.


When value() changes → stars() automatically recalculates → template updates.

// ── rating.component.ts — full working example ───────────────────────────────
import { Component, input, computed } from '@angular/core';

@Component({
  selector: 'app-rating',
  template: `
    <div>
      <p>{{ label() ?? 'Rating' }}: {{ stars() }}</p>
      <small>Score: {{ value() }} / 5</small>
    </div>
  `
})
export class RatingComponent {
  // ── Form 1: input with default ─────────────────────────────────────
  value = input(0);
  //       ↑ default = 0      → InputSignal<number>
  //  Parent:  <app-rating />           → value() is 0
  //  Parent:  <app-rating [value]="4" /> → value() is 4

  // ── Form 2: input with no default (optional) ───────────────────────
  label = input<string>();
  //       ↑ no default → InputSignal<string | undefined>
  //  Parent:  <app-rating />                   → label() is undefined
  //  Parent:  <app-rating [label]="'Quality'" /> → label() is 'Quality'

  // ── computed: auto-recalculates when value() changes ───────────────
  stars = computed(() => {
    const n = Math.min(this.value(), 5); // reads the signal
    return '⭐'.repeat(n) || 'No rating';
  });
  // value() = 0 → stars() = 'No rating'
  // value() = 3 → stars() = '⭐⭐⭐'
  // value() = 4 → stars() = '⭐⭐⭐⭐'   (auto, no manual update needed)
}
▶ Parent: <app-rating [value]="4" [label]="'Quality'" />
Quality: ⭐⭐⭐⭐
Score: 4 / 5

The Problem — forgetting required inputs

Without input.required(), if the parent forgets to pass a value, your component gets undefined at runtime — causing silent bugs or crashes that are hard to trace.

The Solution — compile-time error

input.required() tells Angular the compiler to throw a build error if the parent doesn't provide the value. You catch the bug before the app even runs.

// ── product-card.component.ts ────────────────────────────────────────────────
export class ProductCardComponent {
  // input.required() — MUST be provided by parent — no default, no undefined
  product = input.required<{ name: string; price: number }>();
  //                 ↑ explicit generic type since there's no default to infer from
  //   Returns: InputSignal<{name:string, price:number}>  ← NO undefined

  // Read it safely — TypeScript KNOWS it will never be undefined
  // template: <h3>{{ product().name }}</h3>  ← no null check needed ✅
}

// ── parent template ───────────────────────────────────────────────────────────

// ✅ Correct — product is provided
<app-product-card [product]="{ name: 'Laptop', price: 55000 }" />

// ❌ BUILD ERROR — "Required input 'product' from component ProductCardComponent must be specified."
<app-product-card />
// ↑ Angular stops the build — you CANNOT miss it

// ── Difference in TypeScript types ───────────────────────────────────────────
optionalInput = input<string>();          // InputSignal<string | undefined>
// Must null-check:  optionalInput()?.toUpperCase()

requiredInput = input.required<string>(); // InputSignal<string>
// Safe to use directly: requiredInput().toUpperCase()  ✅

Laptop

🤔 Why do we need transforms?

HTML attributes always come as strings. When you write <app-toggle disabled />, HTML passes the string "disabled" or an empty string — not a boolean. Transforms automatically convert the incoming value before it's stored in the signal, so your component always gets the right type.

// ── transforms — coerce incoming values before storing ───────────────────────
import { input, booleanAttribute, numberAttribute } from '@angular/core';

export class ButtonComponent {

  // ── booleanAttribute transform ────────────────────────────────────────
  // Problem without transform:
  //   <app-btn disabled />  → disabled receives ""  (empty string) ← falsy bug!
  // With booleanAttribute transform:
  //   <app-btn disabled />        → disabled() = true   ✅
  //   <app-btn disabled="false" /> → disabled() = false  ✅  (special case)
  //   <app-btn [disabled]="true" />→ disabled() = true   ✅
  disabled = input(false, { transform: booleanAttribute });

  // ── numberAttribute transform ─────────────────────────────────────────
  // Problem without transform:
  //   <app-btn size="3" /> → size receives "3" (string) → "3" * 2 = NaN!
  // With numberAttribute transform:
  //   <app-btn size="3" />    → size() = 3   (number) ✅
  //   <app-btn [size]="3" />  → size() = 3   (number) ✅
  //   <app-btn size="abc" />  → size() = NaN (parse failed)
  size = input(1, { transform: numberAttribute });

  // ── Custom transform — your own conversion function ───────────────────
  // Accepts a number from parent, stores as 'Xpx' string
  //   <app-btn [width]="200" /> → widthPx() = '200px'
  widthPx = input('0px', { transform: (v: number) => `${v}px` });

  // ── Alias — different name in template vs class ───────────────────────
  // Class uses 'label', template uses 'btnLabel'
  //   <app-btn [btnLabel]="'Save'" /> → label() = 'Save'
  label = input('', { alias: 'btnLabel' });
}
▶ All transform examples
<app-btn disabled />          → disabled() = true  ✅ (not "" or undefined)
<app-btn size="3" />          → size() = 3          ✅ (number, not string "3")
<app-btn [width]="200" />     → widthPx() = '200px' ✅ (auto-converted)
<app-btn [btnLabel]="'Save'"> → label() = 'Save'    ✅ (alias resolved)
// ════════════ OLD WAY — @Input() decorator ═══════════════════════════════════
export class OldSliderComponent {
  @Input() value = 0;           // plain class property — NOT a signal

  // ❌ Problem 1: To react to changes, you need ngOnChanges boilerplate
  ngOnChanges(changes: SimpleChanges) {
    if (changes['value']) {
      console.log('value changed to', changes['value'].currentValue);
    }
  }

  // ❌ Problem 2: Can't use directly in computed()
  // ❌ Problem 3: Angular must check this property every CD cycle (slow)
  // ❌ Problem 4: No type-safe required enforcement at build time
}

// ════════════ NEW WAY — input() signal ✅ ════════════════════════════════════
export class NewSliderComponent {
  value = input(0);             // InputSignal — reactive, tracked by Angular

  // ✅ React to changes with effect() — clean, no boilerplate method needed
  constructor() {
    effect(() => {
      console.log('value changed to', this.value()); // auto-tracks
    });
  }

  // ✅ Use directly in computed() — recalculates automatically
  doubled = computed(() => this.value() * 2);

  // ✅ Angular only updates when this signal changes (fast, zoneless-ready)
  // ✅ input.required() gives build-time safety
}

Custom Events with output() Function (v22)

output() returns an OutputEmitterRef

  • Call .emit(value) to fire the event
  • Parent listens with (eventName)="handler($event)"
  • Events do not bubble up the DOM (unlike native events)
  • Output names are case-sensitive

// ── counter.component.ts — emits events to parent ────────────────────────────
import { Component, signal, output } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `
    <div>
      <button (click)="decrement()">−</button>
      <span> {{ count() }} </span>
      <button (click)="increment()">+</button>
    </div>
  `
})
export class CounterComponent {
  count = signal(0);

  // Declare output events using output() function
  countChanged = output<number>();         // emits a number
  limitReached = output<void>();           // emits nothing (void)

  increment() {
    this.count.update(c => c + 1);
    this.countChanged.emit(this.count());   // emit new value
    if (this.count() >= 10) this.limitReached.emit();
  }

  decrement() {
    this.count.update(c => c - 1);
    this.countChanged.emit(this.count());
  }
}

// ── app.component.ts — parent listens to events ──────────────────────────────
@Component({
  imports: [CounterComponent],
  template: `
    <app-counter
      (countChanged)="onCountChange($event)"
      (limitReached)="onLimit()"
    />
    <p>Parent sees: {{ parentCount() }}</p>
  `
})
export class AppComponent {
  parentCount = signal(0);

  onCountChange(value: number) { this.parentCount.set(value); }
  onLimit() { alert('Limit of 10 reached!'); }
}

Counter shows:  − 3 +
Parent sees: 3

(At 10) → alert: "Limit of 10 reached!"

Data Sharing — Parent → Child (via Inputs)

Parent passes data down to child using property binding [property]="value". The child declares the input. This is the most common pattern.

// ── user-profile.component.ts (CHILD) ────────────────────────────────────────
@Component({
  selector: 'app-user-profile',
  template: `
    <div class="profile">
      <img [src]="avatarUrl()" [alt]="name()" />
      <h3>{{ name() }}</h3>
      <p>{{ role() | uppercase }}</p>
      @if (isAdmin()) {
        <span class="badge">⚡ Admin</span>
      }
    </div>
  `
})
export class UserProfileComponent {
  name      = input.required<string>();
  role      = input.required<string>();
  avatarUrl = input('/assets/default-avatar.png');
  isAdmin   = input(false, { transform: booleanAttribute });
}

// ── dashboard.component.ts (PARENT) ──────────────────────────────────────────
@Component({
  imports: [UserProfileComponent],
  template: `
    <h1>Dashboard</h1>

    <!-- Pass static values -->
    <app-user-profile name="Nitin" role="developer" isAdmin />

    <!-- Pass dynamic values from parent's signals -->
    <app-user-profile
      [name]="currentUser().name"
      [role]="currentUser().role"
      [avatarUrl]="currentUser().avatar"
      [isAdmin]="currentUser().isAdmin"
    />

    <!-- Loop through users -->
    @for (user of users(); track user.id) {
      <app-user-profile [name]="user.name" [role]="user.role" />
    }
  `
})
export class DashboardComponent {
  currentUser = signal({ name: 'Nitin', role: 'Senior Dev', avatar: '/img/nitin.jpg', isAdmin: true });
  users       = signal([...]);
}
▶ Rendered output
[Avatar] Nitin Kumar
         SENIOR DEV
         ⚡ Admin
🔑 Data Flow Rule

Parent → Child = Input. Data flows downward via [property]="value". Child never modifies the input directly. This enforces one-way data flow and makes debugging predictable.

Data Sharing — Child → Parent (via Outputs)

Child notifies parent using custom events via output(). Parent listens with (event)="handler($event)". Data flows upward.

// ── search-bar.component.ts (CHILD) ──────────────────────────────────────────
@Component({
  selector: 'app-search-bar',
  template: `
    <input
      type="text"
      [value]="query()"
      (input)="onType($event)"
      (keydown.enter)="onSearch()"
      placeholder="Search..."
    />
    <button (click)="onSearch()">🔍 Search</button>
    <button (click)="onClear()">✕</button>
  `
})
export class SearchBarComponent {
  query    = signal('');

  // ↑ emit typed search term to parent
  searched = output<string>();
  cleared  = output<void>();

  onType(e: Event) {
    this.query.set((e.target as HTMLInputElement).value);
  }

  onSearch() { this.searched.emit(this.query()); } // ↑ notify parent
  onClear()  { this.query.set(''); this.cleared.emit(); }
}

// ── products-page.component.ts (PARENT) ──────────────────────────────────────
@Component({
  imports: [SearchBarComponent],
  template: `
    <app-search-bar
      (searched)="loadResults($event)"
      (cleared)="resetResults()"
    />
    <p>Results for: "{{ activeQuery() }}"</p>
    <ul>
      @for (r of results(); track r.id) { <li>{{ r.name }}</li> }
    </ul>
  `
})
export class ProductsPageComponent {
  activeQuery = signal('');
  results     = signal<any[]>([]);

  loadResults(query: string) {
    this.activeQuery.set(query);
    // call API with query...
  }
  resetResults() { this.activeQuery.set(''); this.results.set([]); }
}

Two-way Binding — model() Function

What is model()?

A special signal input that allows the child to write back to the parent. It auto-creates a corresponding output named valueChange. Use the banana-in-a-box [(value)] syntax in the parent template.

When to use model()?

  • Custom form controls (date picker, slider, toggle)
  • Any component that modifies a value based on user interaction
  • Replaces the manual @Input() + @Output() valueChange pattern
// ── star-rating.component.ts — uses model() ───────────────────────────────────
import { Component, model } from '@angular/core';

@Component({
  selector: 'app-star-rating',
  template: `
    @for (star of stars; track star) {
      <span
        [class.filled]="star <= rating()"
        (click)="setRating(star)"
        style="cursor:pointer;font-size:24px"
      >{{ star <= rating() ? '⭐' : '☆' }}</span>
    }
  `
})
export class StarRatingComponent {
  // model() — writable signal input with auto-created 'ratingChange' output
  rating = model(0);   // ModelSignal<number>

  stars = [1, 2, 3, 4, 5];

  setRating(star: number) {
    this.rating.set(star);    // ← writes new value, propagates to parent
  }
}

// ── product-review.component.ts (PARENT) ─────────────────────────────────────
@Component({
  imports: [StarRatingComponent],
  template: `
    <h3>Rate this product</h3>

    <!-- banana-in-a-box [(rating)] = two-way binding ✅ -->
    <app-star-rating [(rating)]="productRating" />

    <p>You rated: {{ productRating() }} / 5</p>

    <!-- Also works with plain property (non-signal) -->
    <app-star-rating [(rating)]="plainRating" />
  `
})
export class ProductReviewComponent {
  productRating = signal(3);  // signal binding
  plainRating   = 3;           // plain property binding also works
}
▶ User clicks 4th star
☆ ☆ ☆ ☆ ☆  (initial)
⭐⭐⭐⭐☆  (after click on 4th star)

Parent: You rated: 4 / 5  ✅ (auto-synced via model())
✅ model() auto-creates the change output

When you declare rating = model(0), Angular automatically creates an output named ratingChange. The [(rating)] syntax in the parent is shorthand for [rating]="val" (ratingChange)="val = $event".

Sibling Communication — Shared Service + Signals

Sibling components have no direct connection. They communicate via a shared service injected into both. In Angular v22, signals in services are the cleanest approach.

// ── cart.service.ts — shared state service ────────────────────────────────────
import { Injectable, signal, computed } from '@angular/core';

@Injectable({ providedIn: 'root' })       // singleton — shared across all components
export class CartService {
  private items = signal<CartItem[]>([]);

  // Public read-only derived signals
  itemCount = computed(() => this.items().length);
  total     = computed(() => this.items().reduce((s, i) => s + i.price, 0));
  cartItems = this.items.asReadonly();   // expose as read-only

  addItem(item: CartItem) {
    this.items.update(list => [...list, item]);
  }
  removeItem(id: number) {
    this.items.update(list => list.filter(i => i.id !== id));
  }
}

// ── product-list.component.ts (SIBLING A) — adds items ───────────────────────
@Component({ template: `
  @for (p of products(); track p.id) {
    <div>{{ p.name }} <button (click)="addToCart(p)">Add to Cart</button></div>
  }
` })
export class ProductListComponent {
  private cart = inject(CartService);
  products     = signal([{ id:1, name:'Laptop', price:55000 }]);

  addToCart(p: any) { this.cart.addItem(p); }
}

// ── cart-icon.component.ts (SIBLING B) — shows count ─────────────────────────
@Component({ template: `<button>🛒 {{ count() }}</button>` })
export class CartIconComponent {
  private cart = inject(CartService);
  count        = this.cart.itemCount;   // reactive signal — auto-updates ✅
}

Content Projection — ng-content

What is ng-content?

Allows a component to accept and render HTML/components passed between its opening and closing tags by the parent. Think of it as a slot system — like Vue's <slot> or React's children prop.

Three Types

  • Default slot — single <ng-content />
  • Named slots — <ng-content select="[header]" />
  • Fallback content — shown when nothing is projected
// ── card-layout.component.ts — defines slots ─────────────────────────────────
@Component({
  selector: 'app-card',
  template: `
    <div class="card">

      <!-- Named slot: content with [header] attribute goes here -->
      <div class="card-header">
        <ng-content select="[header]" />
      </div>

      <!-- Default slot: everything else goes here -->
      <div class="card-body">
        <ng-content />
      </div>

      <!-- Named slot for footer -->
      <div class="card-footer">
        <ng-content select="[footer]">
          <span>Default footer</span>  <!-- fallback -->
        </ng-content>
      </div>

    </div>
  `
})
export class CardComponent {}

// ── parent template — projects content into slots ─────────────────────────────
<app-card>
  <h2 header>User Profile</h2>           <!-- → card-header slot -->

  <p>Name: Nitin Kumar</p>                 <!-- → default slot (card-body) -->
  <p>Role: Senior Developer</p>

  <button footer>Edit Profile</button>      <!-- → footer slot -->
</app-card>

✅ Use cases for ng-content

UI component libraries (cards, modals, accordions, tabs). Any reusable wrapper that needs to be flexible about what it displays inside. If every instance always shows the same content, use @Input() instead.

viewChild() & contentChild() Queries

viewChild() — query own template

Access a child component, directive, or DOM element declared in this component's own template. Result is available from ngAfterViewInit onwards.

contentChild() — query projected content

Access a child component or element projected via ng-content by the parent. Result is available from ngAfterContentInit onwards.

// ── parent.component.ts — query child components ─────────────────────────────
import { Component, viewChild, viewChildren,
         contentChild, afterNextRender } from '@angular/core';

@Component({
  template: `
    <app-chart #chart />
    <input #searchInput type="text" />
    @for (item of items(); track item) {
      <app-list-item />
    }
    <ng-content />
  `
})
export class ParentComponent {
  // viewChild — query single child by component type or template reference
  chart       = viewChild.required(ChartComponent);    // by type
  searchInput = viewChild<ElementRef>('searchInput');    // by template ref

  // viewChildren — query ALL matching children
  listItems   = viewChildren(ListItemComponent);        // Signal<readonly ListItemComponent[]>

  // contentChild — query projected child (from ng-content)
  projectedHeader = contentChild(HeaderComponent);

  constructor() {
    afterNextRender(() => {
      // viewChild is a Signal — read with ()
      this.chart().refresh();                         // call child method
      this.searchInput()?.nativeElement.focus();      // DOM access

      console.log('Items count:', this.listItems().length);
    });
  }
}

(search input auto-focused)
(chart.refresh() called on child)
✅ viewChild() is now a Signal (v17+)

The new viewChild() function returns a Signal — read it with this.chart(). The old @ViewChild() decorator returns a plain property. Both work, but the signal version integrates with effect() and computed() reactively.

Senior Interview Q&A — Part 1

Q1. What is the difference between input() and @Input() in Angular v22? INPUTS
input() is the new signal-based API introduced in Angular 17. It returns a read-only InputSignal — reactive, works with computed() and effect(), and is tree-shakeable. @Input() is the decorator-based approach — returns a plain property, requires ngOnChanges to react to changes. Angular recommends input() for all new code. Both have identical template binding syntax: [value]="x".
Q2. How does input.required() differ from @Input({ required: true })? REQUIRED
input.required() enforces the input at compile time — Angular throws a build error if the parent doesn't provide it. The returned InputSignal type does NOT include undefined (it's InputSignal<T> not InputSignal<T|undefined>). @Input({ required: true }) only enforces in some IDEs/tools — no strict build-time guarantee. input.required() is the safer, stricter choice.
Q3. What is ngOnChanges and when does it NOT fire? LIFECYCLE
ngOnChanges fires whenever a bound @Input() property changes. It does NOT fire for: (1) Signal-based input() — use effect() instead. (2) Changes to object/array properties (mutation) — only fires when the reference changes. (3) Properties not decorated with @Input(). Use effect(() => this.value()) to react to signal input changes.
Q4. What is the difference between output() and @Output() + EventEmitter? OUTPUTS
output() returns an OutputEmitterRef — no RxJS dependency, lighter weight, consistent with the signal API. @Output() EventEmitter extends Subject from RxJS — adds overhead. Both use identical template binding (event)="handler($event)". The output() function also auto-cleans subscriptions when the component is destroyed. For new code, always use output().
Q5. What is model() and how does it differ from combining @Input() + @Output()? TWO-WAY
model() is a writable signal that auto-creates a corresponding change output. It's equivalent to writing @Input() value = 0 + @Output() valueChange = new EventEmitter<number>(), but in a single declaration. The child writes new values with this.value.set(x), and they automatically propagate to the parent. The parent uses banana-in-a-box syntax: [(value)]="myVar".
Q6. How do sibling components communicate in Angular? COMMUNICATION
Siblings have no direct connection. Best approaches: (1) Shared service with signals — inject the same service in both siblings. Signal state auto-updates both. (2) Lift state up — move shared state to the common parent, pass via inputs/outputs. (3) NgRx Store — for app-wide state. Signal-based shared service is the recommended modern approach in Angular v22.

Senior Interview Q&A — Part 2

Q7. What is the difference between viewChild() and contentChild()? QUERIES
viewChild() queries elements defined in the component's own template. Available from ngAfterViewInit. contentChild() queries elements projected via ng-content from the parent. Available from ngAfterContentInit. Both return signals in Angular v17+ — read them with this.chart(). viewChild.required() throws if the element is not found.
Q8. What is ng-content and what are its limitations? PROJECTION
ng-content projects DOM content passed between a component's tags into named or default slots. Limitations: (1) Projected content is created in the parent's context — the child cannot access its data directly. (2) Cannot conditionally show/hide projected content from the child. (3) ng-content cannot be combined with *ngIf on itself. Use ngTemplateOutlet with @ContentChild when you need conditional content.
Q9. Why should you avoid direct DOM manipulation in ngAfterViewInit in an SSR app? SSR
ngAfterViewInit runs on both server and browser. In Node.js (SSR), there's no real DOM — accessing document or window inside it crashes the server. Use afterNextRender() instead — it is browser-only guaranteed. For accessing child component references, viewChild() signals are safe to read in both environments (they're just object references, not DOM queries).
Q10. What is "prop drilling" and how do you avoid it? ARCHITECTURE
Prop drilling is passing data through multiple intermediate components that don't need it, just to reach a deeply nested component. In Angular v22: (1) Inject a shared service with signal state — any component at any level can inject it directly. (2) DI with route-level providers — provide a scoped service at the route level. (3) NgRx for complex global state. Prop drilling makes refactoring painful and creates unnecessary coupling.
Q11. What is the component host element and how do you style it? HOST
The host element is the DOM element matching a component's selector. When you use <app-card>, the <app-card> element itself is the host. Style it using the :host CSS selector in the component's styles, or using the host property in @Component: host: { class: 'my-card', '[attr.role]': '"region"' }. In templates, use @HostBinding (decorator) or the host metadata for dynamic host properties.
Q12. What is the difference between OnPush and Default change detection for a component? PERFORMANCE
Default: Angular checks the component on every change detection cycle — any event, timer, or HTTP call. OnPush: Angular only checks when (1) an @Input() reference changes, (2) an event originates from within the component, (3) you manually call markForCheck() / detectChanges(). With signal-based components in v22, change detection is automatic and granular — signals tell Angular exactly what changed, making OnPush essentially the default behaviour without explicit configuration.

Senior Interview Q&A — Part 3

Q13. Can you call input() or output() inside a method? RESTRICTIONS
No. The input(), output(), model(), viewChild(), and contentChild() functions can only be called in class property initializers — not inside methods, constructors conditionally, or lifecycle hooks. This is a compiler restriction — Angular statically analyses these calls at build time to generate optimised code. Calling them anywhere else results in a runtime error.
Q14. When would you use input transforms? Give a real example. TRANSFORMS
Input transforms coerce or sanitise values before they're stored. Real examples: (1) booleanAttribute — a button component accepts <app-btn disabled /> (no value) as true. (2) numberAttribute — an input like <app-slider width="200" /> passes a string from the template, transform coerces to number. (3) Custom trim transform — a label input auto-trims whitespace. Transforms are statically analysable and must be pure functions.
Q15. How do you communicate between deeply nested components without prop drilling? DEEP NESTING
Three patterns: (1) Shared service + inject() — the cleanest. Declare state in a service, inject it in any component at any depth. Signal-based state auto-updates all consumers. (2) Route-level providers — provide a scoped service at the route level using the route's providers array; all child components in that route tree can inject it. (3) NgRx — for complex enterprise-level state that needs time-travel debugging, effects, and selectors.
Q16. What happens if you mutate an object passed as @Input() instead of replacing it? PITFALL
If the child mutates the object (e.g. this.user.name = 'New'), Angular's change detection for the parent may not detect the change because the object reference didn't change. With OnPush, the child won't re-render either because its input reference is the same. Always treat inputs as immutable — create new objects/arrays: this.user = { ...this.user, name: 'New' }. With signal inputs, you can use effect() to react to any value change regardless.

ViewEncapsulation — Style Scoping

Angular controls how component styles are applied to the DOM via encapsulation in @Component. Choosing the wrong mode is a common cause of style leaking bugs.

Emulated Default

Angular rewrites CSS selectors by appending a unique attribute (e.g. _ngcontent-abc-c1) to every rule. Styles are scoped to the component — they never leak out.


Works in all browsers. No native Shadow DOM required.

None

Styles are added to the <head> as global CSS. They affect every element in the entire app — including child and sibling components.


Use only when you intentionally want global styles (e.g. a theme component).

ShadowDom

Uses the browser's native Shadow DOM API. Styles are truly isolated inside a shadow root. External CSS (even global) cannot penetrate in.


Strictest isolation. Not compatible with some CSS-in-JS or global UI libraries.

import { Component, ViewEncapsulation } from '@angular/core';

// ── Emulated (DEFAULT) ───────────────────────────────────────────────────────
@Component({
  selector: 'app-card',
  template: `<p class="text">Hello</p>`,
  styles: `p { color: red; }`,
  // encapsulation: ViewEncapsulation.Emulated  ← this is the default, no need to write it
})
export class CardComponent {}
// Angular emits:  p[_ngcontent-abc-c1] { color: red; }
// Only <p> tags INSIDE this component are red ✅

// ── None ─────────────────────────────────────────────────────────────────────
@Component({
  encapsulation: ViewEncapsulation.None,
  styles: `p { color: red; }`  // ← applies to ALL <p> tags in the app ⚠️
})
export class ThemeComponent {}

// ── ShadowDom ────────────────────────────────────────────────────────────────
@Component({
  encapsulation: ViewEncapsulation.ShadowDom,
  styles: `p { color: red; }`  // only inside shadow root — true isolation
})
export class IsolatedComponent {}

// ── :host — style the component's own host element ──────────────────────────
// Works in Emulated and ShadowDom modes
styles: `
  :host { display: block; border: 1px solid #30363d; }  /* targets <app-card> itself */
  :host(.active) { border-color: #3fb950; }             /* when host has .active class */
  :host-context(.dark-theme) p { color: white; }        /* when ancestor has .dark-theme */
`
⚠️ ::ng-deep — use with caution

::ng-deep .child-class {} pierces encapsulation to style child component elements. It is deprecated but still works. A better approach is to expose a CSS custom property (CSS variable) from the child: color: var(--card-text-color, black) and let the parent set it: app-card { --card-text-color: red; }.

Dynamic Components — Runtime Rendering

Dynamic components are created and inserted into the DOM at runtime — not declared in any template. Essential for modals, toasts, dynamic dashboards, and plugin architectures.

How it works

  • Get a ViewContainerRef — a reference to a location in the DOM
  • Call .createComponent(MyComponent) — Angular instantiates it
  • Set inputs via instance.myInput.set(value) or setInput()
  • Subscribe to outputs via instance.myOutput.subscribe()
  • Call .destroy() to remove it from the DOM

Where to anchor — 3 options

  • viewChild(ViewContainerRef) on a <ng-container>
  • inject(ViewContainerRef) — uses the component itself as anchor
  • A dedicated <ng-container #host></ng-container> in template
// ── alert.component.ts — the dynamic component ──────────────────────────
@Component({
  selector: 'app-alert',
  template: `<div class="alert">{{ message() }} <button (click)="close()">✕</button></div>`
})
export class AlertComponent {
  message = input.required<string>();
  closed  = output<void>();
  close() { this.closed.emit(); }
}

// ── host.component.ts — creates the dynamic component ─────────────────────
@Component({
  template: `
    <button (click)="showAlert()">Show Alert</button>
    <ng-container #alertHost />
  `
})
export class HostComponent {
  // Get a reference to the ng-container location in the DOM
  alertHost = viewChild.required('alertHost', { read: ViewContainerRef });

  showAlert() {
    const vcr = this.alertHost();           // read the signal
    vcr.clear();                               // remove any previous

    // ① Create the component dynamically
    const ref = vcr.createComponent(AlertComponent);

    // ② Pass inputs via setInput() (works with both signal and decorator inputs)
    ref.setInput('message', 'File saved successfully! ✅');

    // ③ Listen to outputs
    ref.instance.closed.subscribe(() => ref.destroy());
  }
}

[ File saved successfully! ✅  ✕ ]
(clicking ✕ → component removed from DOM)

NgComponentOutlet — declarative dynamic components

Cleaner template-based approach. Angular handles creation/destruction automatically. Best when the component type is determined by data (e.g. rendering different widget types from a config array).

// ── Three widget components ─────────────────────────────────────────────────────────
export class ChartWidgetComponent  { title = input('Chart'); }
export class TableWidgetComponent  { title = input('Table'); }
export class MetricWidgetComponent { title = input('Metric'); }

// ── dashboard.component.ts — renders widgets dynamically from config ──────
@Component({
  imports: [NgComponentOutlet],
  template: `
    @for (widget of widgets(); track widget.id) {
      <!-- NgComponentOutlet creates the component from a class reference -->
      <ng-container
        [ngComponentOutlet]="widget.component"
        [ngComponentOutletInputs]="{ title: widget.title }"
      />
    }
  `
})
export class DashboardComponent {
  widgets = signal([
    { id: 1, component: ChartWidgetComponent,  title: 'Sales Chart'  },
    { id: 2, component: TableWidgetComponent,  title: 'User Table'   },
    { id: 3, component: MetricWidgetComponent, title: 'Revenue KPI'  },
  ]);

  addWidget() {
    // Add a new widget at runtime — NgComponentOutlet auto-creates it
    this.widgets.update(w => [...w, { id: 4, component: ChartWidgetComponent, title: 'New Chart' }]);
  }
}

// ── toast.service.ts — global dynamic component service ────────────────────
@Injectable({ providedIn: 'root' })
export class ToastService {
  private vcr!: ViewContainerRef;

  // Called once from AppComponent to register the root container
  setContainer(vcr: ViewContainerRef) { this.vcr = vcr; }

  show(message: string, type: 'success' | 'error' = 'success') {
    const ref = this.vcr.createComponent(ToastComponent);
    ref.setInput('message', message);
    ref.setInput('type', type);

    // Auto-destroy after 3 seconds
    setTimeout(() => ref.destroy(), 3000);
  }
}

// ── app.component.ts — register the container ─────────────────────────────
@Component({
  template: `<router-outlet /> <ng-container #toastHost />`
})
export class AppComponent {
  toastHost = viewChild.required('toastHost', { read: ViewContainerRef });
  private toast = inject(ToastService);

  constructor() {
    afterNextRender(() => this.toast.setContainer(this.toastHost()));
  }
}

// ── any.component.ts — use the service anywhere ─────────────────────────
export class SaveButtonComponent {
  private toast = inject(ToastService);

  save() {
    // ... save logic
    this.toast.show('Saved successfully ✅', 'success');
    // ToastComponent appears in DOM, auto-destroys after 3s
  }
}

  Component,
  input,           // InputSignal (read-only)

@Component({
  selector: 'app-example',
  standalone: true,     // default in v19+
  imports: [...],       // child components
  templateUrl: '...',
  styleUrl: '...'      // singular (v17+)
})
export class ExampleComponent {

  name    = input.required<string>();
  count   = input(0);

  changed = output<string>();

  value   = model(0);

  data    = signal<any[]>([]);

  // Queries
  child   = viewChild.required(ChildComponent);
}
