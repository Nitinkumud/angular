Cold vs Hot Observables

One of the most common RxJS interview topics — understand the producer model, subscription behaviour, memory leaks and real-world patterns.

What is Cold? What is Hot? Code Examples share() / shareReplay() Subject Real-World Use-Cases Memory Tips Interview Q&A
01

The Core Difference

🧊 Cold Observable

The producer is created inside the Observable. Every subscriber gets its own independent execution — the stream starts fresh for each one.


  • New producer per subscriber
  • Data replayed from the beginning
  • Unicast — 1 subscriber = 1 producer
  • Lazy — nothing runs until subscribed

🔥 Hot Observable

The producer exists outside the Observable. All subscribers share the same execution — late subscribers miss earlier emissions.


  • Single shared producer
  • Late subscribers miss past values
  • Multicast — N subscribers = 1 producer
  • Active regardless of subscribers

💡 The Key Mental Model — "Where is the producer?"

Cold → producer is created inside the new Observable() callback → each subscribe call creates a brand-new producer.
Hot → producer is created outside (before subscribe) → all subscribers tap into the same running stream.

🎬

Cold = Netflix / VOD

Every viewer starts the movie from Episode 1, independently, at their own time.

📺

Hot = Live TV

You tune in mid-broadcast and see only what's airing right now. You miss what aired earlier.

02

🧊 Cold Observable — Code Example

Each subscriber gets a completely independent execution

Notice: "Producer started!" logs twice — once per subscriber. Each gets a different random value.

import { Observable } from 'rxjs';

// Producer (Math.random) is INSIDE the Observable — Cold!
const cold$ = new Observable(observer => {
  const value = Math.random();        // new value created per subscription
  console.log('Producer started! value=', value);
  observer.next(value);
  observer.complete();
});

// Subscriber 1
cold$.subscribe(val => console.log('Sub 1:', val));
// Producer started! value= 0.4281   →  Sub 1: 0.4281

// Subscriber 2
cold$.subscribe(val => console.log('Sub 2:', val));
// Producer started! value= 0.8734   →  Sub 2: 0.8734

// ✅ Each subscriber gets a DIFFERENT value — completely independent

Common Cold Sources

  • new Observable(...)
  • of(1, 2, 3)
  • from([1, 2, 3])
  • interval(1000)
  • timer(500)
  • range(1, 10)
  • this.http.get(...) — Angular HTTP
  • ajax(...) — RxJS ajax
⚠️ Real-World Problem with Cold HTTP

If 3 components subscribe to the same http.get() Observable, 3 separate HTTP requests are fired! This is a cold observable pitfall. Solution: use shareReplay(1) to make it hot.

03

🔥 Hot Observable — Code Example

Single producer, shared by all subscribers

Notice: Subscriber 2 misses value 1 because it subscribed late. The Subject (producer) is outside the subscription logic.

import { Subject } from 'rxjs';

// Subject is created OUTSIDE — it's Hot!
const hot$ = new Subject<number>();

// Subscriber 1 joins immediately
hot$.subscribe(val => console.log('Sub 1:', val));

hot$.next(1);   // Sub 1: 1

// Subscriber 2 joins LATE
hot$.subscribe(val => console.log('Sub 2:', val));

hot$.next(2);   // Sub 1: 2,  Sub 2: 2
hot$.next(3);   // Sub 1: 3,  Sub 2: 3

// ❌ Sub 2 MISSED value 1 — that's hot behaviour!

Common Hot Sources

  • Subject
  • BehaviorSubject
  • ReplaySubject
  • AsyncSubject
  • fromEvent(button, 'click')
  • fromEvent(window, 'scroll')
  • WebSocket streams
  • cold$.pipe(share())

Hot Timeline Visualised

Subject$ ─── 1 ──── 2 ──── 3 ──▶
Sub 1 (t=0) ─── 1 ──── 2 ──── 3 ──▶
Sub 2 (late) ────────── 2 ──── 3 ──▶

Sub 2 missed value 1 — joined after it was emitted

Converting Cold → Hot with share() & shareReplay()

share()

Multicasts the source Observable. All subscribers share one subscription. Does NOT replay past values to late subscribers.

shareReplay(n)

Multicasts AND replays the last n values to any new subscriber. Perfect for HTTP calls you want cached.

import { interval } from 'rxjs';
import { share, shareReplay, tap } from 'rxjs/operators';

// ─── share() ───────────────────────────────────────────
const shared$ = interval(1000).pipe(
  tap(() => console.log('Tick!')),   // logs once per tick (not twice!)
  share()
);

shared$.subscribe(v => console.log('Sub A:', v));   // 0, 1, 2, 3...
setTimeout(() => {
  shared$.subscribe(v => console.log('Sub B:', v)); // joins at ~2s → 2, 3...
}, 2000);


// ─── shareReplay(1) for HTTP caching ───────────────────
const user$ = this.http.get<User>('/api/user').pipe(
  shareReplay(1)   // one HTTP call, result shared + replayed to late subscribers
);

user$.subscribe(u => console.log('Component A:', u));  // fires HTTP
user$.subscribe(u => console.log('Component B:', u));  // gets cached — no new HTTP!
✅ When to use share()
  • Event streams: mouse, WebSocket
  • Interval/timer shared among components
  • Side effects that should run only once
✅ When to use shareReplay(1)
  • HTTP calls in Angular services
  • Computed data that's expensive to recalculate
  • Late subscribers needing the last value

Side-by-Side Comparison

Feature 🧊 Cold Observable 🔥 Hot Observable
Producer location Inside the Observable callback Outside the Observable / pre-existing
Per-subscription execution Yes — independent No — shared
Multicast Unicast (1:1) Multicast (1:N)
Late subscriber gets past values? Yes — starts from beginning No — misses past values
Starts without subscribers? No (lazy) Yes (eager)
Examples of(), from(), http.get(), interval() Subject, fromEvent(), share()
Problem to watch Multiple HTTP calls if subscribed multiple times Late subscribers miss data
Solution Use share() or shareReplay(1) Use BehaviorSubject or ReplaySubject

Cold → when to use

  • Fetching fresh data every time (e.g., form submission)
  • Each user/component needs isolated data
  • Unit testing — predictable per-subscriber execution

Hot → when to use

  • Broadcasting events to many consumers (clicks, WebSocket)
  • Shared state across components (BehaviorSubject in a service)
  • Performance — avoid duplicate work / HTTP calls

Real-World Angular Examples

🧊 Cold — Angular HTTP Service

Every call to getUser() fires a fresh HTTP request — cold by default.

// ❌ Problem: 3 components → 3 HTTP calls
@Injectable({ providedIn: 'root' })
export class UserService {
  getUser() {
    return this.http.get('/api/user'); // Cold!
  }
}

// ✅ Fix: shareReplay makes it Hot + cached
@Injectable({ providedIn: 'root' })
export class UserService {
  private user$ = this.http.get('/api/user').pipe(
    shareReplay(1)  // one call, shared result
  );
  getUser() { return this.user$; }
}

🔥 Hot — BehaviorSubject State Store

Components share the same state stream via a BehaviorSubject — classic hot pattern.

@Injectable({ providedIn: 'root' })
export class CartService {
  // BehaviorSubject = Hot + replays last value
  private _cart$ = new BehaviorSubject<Item[]>([]);

  // Expose as read-only Observable
  cart$ = this._cart$.asObservable();

  addItem(item: Item) {
    const current = this._cart$.getValue();
    this._cart$.next([...current, item]);
  }
}
// All components subscribing to cart$ share
// the SAME subject — hot multicast!

🔥 Hot — DOM Events (always hot)

fromEvent() wraps an existing DOM event listener — the event source (button, window) exists independently.

import { fromEvent } from 'rxjs';
import { debounceTime, map } from 'rxjs/operators';

const search$ = fromEvent(inputEl, 'input').pipe(  // Hot!
  debounceTime(300),
  map(e => (e.target as HTMLInputElement).value)
);
// Multiple subscribers all hear the SAME keystrokes — hot!

🧠 Memory Tips & Mnemonics

🎬 Netflix vs 📺 Live TV

Cold = Netflix — You always watch from Episode 1. Each viewer gets their own private stream.


Hot = Live TV — You tune in and see only what's on right now. Miss anything that aired before you joined.

💧 Tap vs Pipe

Cold = Tap water — You open the tap and fresh water flows just for you. Turn off the tap, water stops.


Hot = A pipe in the wall — Water is always flowing inside the pipe. You just plug in your cup when ready.

Interview Q&A

Q1. What is the difference between a Cold and Hot Observable?
A Cold Observable creates a new producer for each subscriber — each gets an independent execution from the start. A Hot Observable has a shared producer — all subscribers tap into the same stream and late subscribers miss past values.
Q2. Is Angular's HttpClient.get() cold or hot? What problem can this cause?
It is Cold. If 3 components subscribe to the same Observable returned by a service method, 3 separate HTTP requests are fired. Fix: store the Observable in the service with shareReplay(1) so one HTTP call is made and the result is cached + replayed.
Q3. How do you convert a Cold Observable into a Hot one?
Use the share() operator (multicast without replay) or shareReplay(n) (multicast + replay last n values). Alternatively, pipe through a Subject manually, or use a BehaviorSubject / ReplaySubject as the source.
Q4. Is fromEvent(button, 'click') cold or hot? Why?
It is Hot, because the event source (the DOM button) exists outside the Observable. All subscribers share the same click events; a late subscriber will not receive clicks that happened before it subscribed.
Q5. What is the difference between share() and shareReplay(1)?
share() multicasts the source but does not replay past values to late subscribers. shareReplay(1) multicasts AND replays the last 1 emission to any new subscriber — ideal for caching HTTP responses so late subscribers immediately get the cached result.
Q6. Is a Subject cold or hot?
A Subject is always Hot. It is both an Observable and an Observer, and its values are emitted from outside. All subscribers share the same multicast stream.
